형식

jstatd [ options ]

파라미터

options
커멘드행 옵션. 옵션은 임의의 순서로 지정할 수 있습니다. 중복 또는 모순되는 옵션이 있는 경우, 마지막에 지정한 옵션이 우선됩니다.

설명

jstatd 툴은, 계측 된 HotSpot Java 가상 머신 (JVM)의 작성과 종료를 감시해, 로컬 시스템상에서 실행하고 있는 Java 가상 머신에, 원격 감시 툴이 접속할 수 있게 하기 위한 인터페이스를 제공하는 RMI 서버 어플리케이션입니다.

jstatd 서버에서는, 로컬 호스트에게 RMI 레지스트리가 존재하는 것이 필요하게 됩니다. jstatd 서버는, 디폴트 포트 또는 -p port 옵션으로 지정된 포트상의 RMI 레지스트리에 접속하려고 합니다. RMI 레지스트리가 발견되지 않는 경우,-p port 옵션으로 지정된 포트, 또는 -p port 가 생략 되고 있는 경우는, 디폴트 RMI 레지스트리에 바인드 된 jstatd 어플리케이션내에, 1 개의 RMI 레지스트리가 작성됩니다. 내부 RMI 레지스트리의 작성은,-nr 옵션을 지정하는 것에 의해 금지할 수가 있습니다.

주:이 유틸리티는 지원 대상외이며, 장래의 JDK 의 버젼에서는 이용할 수 없게 될 가능성이 있습니다. 현재, Windows 98 및 Windows Me 플랫폼에서는 사용할 수 없습니다.

옵션

jstatd 커멘드는 다음의 옵션을 지원하고 있습니다.

-nr
기존의 RMI 레지스트리가 발견되지 않는 경우,jstatd 프로세스내에 내부 RMI 레지스트리를 작성하지 않게 합니다.
-p  port
RMI 레지스트리가 있다고 예상되는 포트 번호입니다. 발견되지 않는 경우는,-nr 가 지정되어 있지 않으면 작성됩니다.
-n  rminame
RMI 레지스트리에 대해, 원격 RMI 객체가 바인드 되는 이름입니다. 디폴트명은 JStatRemoteHost 입니다. 복수의 jstatd 서버가 같은 호스트상에서 기동하고 있는 경우, 각 서버의 export 한 RMI 객체의 이름은, 이 옵션을 지정하는 것에 의해, 일의의 이름으로 할 수 있습니다. 다만, 이 옵션을 사용하는 경우, 감시 클라이언트의 hostid 및 vmid 캐릭터 라인에, 그 일의의 서버명을 포함할 필요가 있습니다.
-Joption
javac 가 호출하는 java 기동 툴에,option 를 건네줍니다. 예를 들어,-J-Xms48m 와 지정하면(자), 스타트 업 메모리는 48M 바이트로 설정됩니다. -J 를 사용해, Java 로 기술된 어플리케이션을 실행하는 배후의 VM 에 옵션 건네주는 것은, 잘 행해지고 있습니다.

시큐리티

jstatd 서버는, 유효한 네이티브 액세스권을 가지는 JVM 만을 감시할 수 있습니다. 따라서,jstatd 프로세스는, 타겟 JVM 와 같은 사용자 자격으로 실행하고 있을 필요가 있습니다. 일부의 사용자 자격은, Unix(TM) 베이스의 시스템에 있어서의 root 사용자와 같이, 시스템상의 임의의 JVM 에 의해 export 된 Instrumentation에의 액세스권을 가지고 있습니다. 이러한 자격으로 실행하고 있는 jstatd 프로세스는, 시스템상의 모든 JVM 를 감시할 수 있습니다만, 시큐리티상의 다른 문제가 일어납니다.

jstatd 서버는, 원격 클라이언트의 인증을 일절 주지 않습니다. 그 때문에,jstatd 서버 프로세스를 실행하면(자),jstatd 프로세스가 액세스권을 가지는 모든 JVM 에 의한 Instrumentation의 export를, 네트워크상의 모든 사용자에게 공개하게 됩니다. 이 무방비인 상태는, 환경적으로 바람직하지 않은 경우가 있습니다. 특히 열매 가동 환경 또는 안전하지 않은 네트워크에서는,jstatd 프로세스를 기동하기 전에, 로컬 시큐리티 정책를 검토할 필요가 있습니다.

jstatd 서버는, 다른 시큐리티 매니저가 인스톨되어 있지 않은 경우에는, RMISecurityPolicy 의 인스턴스를 인스톨 합니다. 그 때문에, 시큐리티 정책 파일을 지정할 필요가 있습니다. 정책 파일은, 디폴트 정책 구현의정책 파일의 구문에 준거할 필요가 있습니다.

다음의 정책 파일의 경우,jstatd 서버는 일절의 시큐리티 예외를 사용하지 않고 실행할 수 있습니다. 이 정책는, 모든 코드 베이스에의 모든 액세스권을 인정하는 정책보다 자유도는 낮습니다만,jstatd 서버를 실행하기 위해서 최저한 필요한 액세스권 밖에 인정하지 않는 정책보다 자유도는 높아지고 있습니다. 

grant codebase "file:${java.home}/../lib/tools.jar" {

   permission java.security.AllPermission;

};

이 정책를 사용하려면 , 이 텍스트를 jstatd.all.policy 라고 하는 파일에 카피해, 다음과 같이 jstatd 서버를 실행합니다. 

jstatd -J-Djava.security.policy=jstatd.all.policy

보다 어려운 시큐리티를 실시하는 사이트의 경우, 커스텀 정책 파일을 사용해, 특정의 신뢰할 수 있는 호스트 또는 네트워크에 액세스를 제한할 수가 있습니다. 다만, 이러한 방법은, IP 주소의 도청 공격을 받기 쉬워집니다. 시큐리티의 문제에 대해, 커스터마이즈 한 정책 파일에서도 대처할 수 없는 경우는,jstatd 서버를 실행하지 않고 ,jstat 와 jps 툴을 로컬로 사용하는 것이 가장 안전한 방법이 됩니다.

원격 인터페이스

jstatd 프로세스가 export 하는 인터페이스는, 독자적으로 개발한 것이어 변경될 예정입니다. 사용자 및 개발자는, 이 인터페이스에의 기입을 실시하지 말아 주세요.

례

다음에 jstatd 를 기동하는 예를 소개합니다. jstatd 스크립트에 의해, 서버는 백그라운드에서 자동적으로 기동합니다.

내부 RMI 레지스트리의 사용

이 예는, 내부 RMI 레지스트리를 사용한 jstatd 의 기동을 나타내고 있습니다. 이 예에서는, 디폴트의 RMI 레지스트리 포트 (포트 1099)에는, 다른 서버는 바인드되어 있지 않으면 상정하고 있습니다. 

jstatd -J-Djava.security.policy=all.policy

외부 RMI 레지스트리의 사용

이 예는, 외부 RMI 레지스트리를 사용한 jstatd 의 기동을 나타내고 있습니다. 

rmiregistry&
jstatd -J-Djava.security.policy=all.policy

이 예는, 포트 2020 의 외부 RMI 레지스트리를 사용한 jstatd 의 기동을 나타내고 있습니다. 

rmiregistry 2020&
jstatd -J-Djava.security.policy=all.policy -p 2020

이 예는, AlternateJstatdServerName 의 이름에 바인드 된, 포트 2020 의 외부 RMI 레지스트리를 사용한 jstatd 의 기동을 나타내고 있습니다. 

rmiregistry 2020&
jstatd -J-Djava.security.policy=all.policy -p 2020 -n AlternateJstatdServerName

인 프로세스 RMI 레지스트리의 작성의 금지

이 예는, RMI 레지스트리가 발견되지 않는 경우에 RMI 레지스트리를 작성하지 않는 jstatd 의 기동을 나타내고 있습니다. 이 예에서는, RMI 레지스트리가 벌써 실행하고 있으면(자) 상정하고 있습니다. 실행하고 있지 않는 경우는, 해당하는 에러 메세지가 표시됩니다. 

jstatd -J-Djava.security.policy=all.policy -nr

RMI 로그 기능의 유효화

이 예는, RMI 로그 기능을 유효하게 한 jstatd 의 기동을 나타내고 있습니다. 이 방법은, 트러블 슈팅 또는 서버 활동의 감시에 도움이 됩니다. 

jstatd -J-Djava.security.policy=all.policy -J-Djava.rmi.server.logCalls=true
