형식

JAR 파일의 작성
jar c[v0M]f jarfile [-C dir] inputfiles [-Joption]
jar c[v0]mf manifest jarfile [-C dir] inputfiles [-Joption] [-e entrypoint]
jar c[v0M] [-C dir] inputfiles [-Joption]
jar c[v0]m manifest [-C dir] inputfiles [-Joption]
JAR 파일의 갱신
jar u[v0M]f jarfile [-C dir] inputfiles [-Joption]
jar u[v0]mf manifest jarfile [-C dir] inputfiles [-Joption] [-e entrypoint]
jar u[v0M] [-C dir] inputfiles [-Joption]
jar u[v0]m manifest [-C dir] inputfiles [-Joption]
JAR 파일의 추출
jar x[v]f jarfile [inputfiles] [-Joption]
jar x[v] [inputfiles] [-Joption]
JAR 파일의 내용의 겉(표) 형식에 의한 일람표시
jar t[v]f jarfile [inputfiles] [-Joption]
jar t[v] [inputfiles] [-Joption]
JAR 파일에의 인덱스의 추가
jar i jarfile [-Joption]

설명

cuxtiv0Mmf
jar 커멘드를 제어하는 옵션
jarfile
작성 (c), 갱신 (u), 추출 (x), 또는 내용의 일람표시 (t)의 대상이 되는 JAR 파일의 파일명. f 옵션과 파일명 jarfile 가 페어가 됩니다. 즉, 다른 한쪽을 기술하면, 이제(벌써) 한편도 기술할 필요가 있습니다. f 와 jarfile 를 생략 하면(자), 표준 입력으로부터 「JAR 파일」이 취득되는지 (x 및 t 의 경우), 표준 출력에 「JAR 파일」이 송신됩니다(c 및 u 의 경우).
inputfiles
jarfile 에 결합되는지 (c 및 u 의 경우),jarfile 로부터 추출 (x 의 경우) 또는 일람표시 (t 의 경우) 되는, 공백에서 단락지어진 파일 또는 디렉토리. 모든 디렉토리는 재귀적으로 처리됩니다. 이 파일은, 옵션 -0 (제로)을 사용하지 않는 한 압축됩니다.
manifest
name: value 의 페어가, JAR 파일의 MANIFEST.MF 에 포함되어 있는 기존의 Manifest 파일. m 옵션과 파일명 manifest 가 페어가 됩니다. 즉, 다른 한쪽을 기술하면, 이제(벌써) 한편도 기술할 필요가 있습니다. m 와 f 의 문자는,manifest 와 jarfile 를 기술한 순서와 같은 순서로 기술할 필요가 있습니다.
entrypoint
실행 가능 JAR 파일내에 번들 된 스탠드얼론 어플리케이션의 어플리케이션 엔트리 포인트로서 설정하는 클래스의 이름. -e 옵션과 entrypoint 는 대가 되어 있습니다. 어느 쪽인지를 지정하는 경우는 양쪽 모두 지정할 필요가 있습니다. m,f, 및 e 의 출현 순서는,manifest,jarfile,entrypoint 의 출현 순서와 일치할 필요가 있습니다.
-C dir
이것에 계속되는 inputfiles 인수를 처리하는 동안, 디렉토리를 dir 에 일시적으로 변경합니다. -C dir inputfiles 세트는 복수 사용할 수 있습니다.
-Joption
Java 실행 환경에 건네받는 옵션. -J 와 option 의 사이에는 공백을 들어갈 수 있지 말아 주세요.

설명

jar 툴은, 복수의 파일을 1 개의 JAR archive파일에 결합합니다. jar 는, ZIP 및 ZLIB 압축 형식을 베이스로 한, 범용의 어카이브(archive) 및 압축 툴입니다. jar 는, 주로 Java 의 애플릿이나 어플리케이션을 1 개의 어카이브(archive)에 패키지화하는 것을 용이하게 하기 위해서 고안 되었습니다. 애플릿이나 어플리케이션의 컴퍼넌트 (. class 파일, 이미지, 및 사운드)가 1 개의 어카이브(archive)에 결합되고 있으면(자), Java 에이전트 (브라우저등)는, 그러한 컴퍼넌트를 1 회의 HTTP 트랜잭션(transaction)로 다운로드할 수가 있어 컴퍼넌트 마다 새로운 접속을 필요로 하는 일이 없습니다. 이것에 의해, 다운로드의 회수가 큰폭으로 줄어듭니다. 또,jar 는 파일을 압축하므로, 다운로드 시간도 짧아집니다. 또, 파일내의 개개의 엔트리에 애플릿 작성자에 의한 서명을 기입할 수 있으므로, 배포원의 인증이 가능하게 됩니다. jar 툴의 구문은,tar 커멘드의 구문과 거의 같습니다. JAR 어카이브(archive)는, 압축되고 있는지 어떤지에 관계없이,클래스 패스의 엔트리로서 사용할 수 있습니다.

복수의 파일을 JAR 파일에 결합하는 일반적인 사용법은 다음과 같습니다. 

C:\Java> jar cf myFile.jar *. class
이 예에서는, 현재의 디렉토리에 있는 클래스 (. class) 파일이 모두, myFile.jar 라는 이름의 파일에 포함됩니다. META-INF/MANIFEST.MF 라고 하는 Manifest 파일 엔트리가 jar 툴에 의해 자동적으로 생성되어 항상 JAR 파일내의 선두에 놓여집니다. Manifest 파일에는,name : value 의 페어로서 어카이브(archive)의 메타 정보도 포함됩니다. Manifest 파일에 메타 정보가 어떻게 해 포함될까에 대한 자세한 것은,JAR 파일의 스펙을 참조해 주세요.

JAR 어카이브(archive)를 작성하는 경우에 jar 툴로 사용하고 싶은 name:value 의 페어를 가진 기존의 Manifest 파일이 있는 경우는, 다음과 같이,m 옵션을 사용해, Manifest 파일을 지정할 수 있습니다. 

C:\Java> jar cmf myManifestFile myFile.jar *. class
기존의 Manifest 파일은 개행 문자로 끝나 있을 필요가 있습니다. Manifest 파일이 개행 문자로 끝나지 않은 경우,jar 는 그 마지막 행을 해석하지 않습니다.

주: 커멘드행에 cmf 는 아니고 cfm 가 지정된 jar 커멘드 (-m 와 -f 옵션의 차례를 반대로 한다)의 경우,jar 커멘드행에 우선 JAR 어카이브(archive)의 이름을 지정해, 계속되어 Manifest 파일의 이름을 지정할 필요가 있습니다. 다음에 예를 나타냅니다. 
C:\Java> jar cfm myFile.jar myManifestFile *. class
Manifest는, RFC822 ASCII 형식에서 규정된 텍스트 형식이기 (위해)때문에, Manifest 파일의 내용을 간단하게 표시 및 편집할 수 있습니다.

JAR 파일로부터 파일을 추출하려면 , 다음과 같이 -x 를 사용합니다. 

C:\Java> jar xf myFile.jar

JAR 파일로부터 특정의 파일만을 추출하려면 , 그 파일명을 지정합니다. 

C:\Java> jar xf myFile.jar foo bar

Java 2 SDK 의 버젼 1.3 이후부터,jar 유틸리티로 JarIndex 가 지원되고 있습니다. JarIndex 를 사용하면(자), 어플리케이션의 클래스 로더로 JAR 파일로부터 클래스가 로드 될 때의 효율이 향상합니다. 어플리케이션 또는 애플릿이 복수의 JAR 파일에 번들 되고 있는 경우는, 클래스가 로드 될 때, 필요한 JAR 파일 이외의 다운로드 및 오픈은 행해지지 않습니다. 이 퍼포먼스의 최적화는, 새롭다 -i 옵션을 지정해 jar 를 실행하면(자) 유효하게 됩니다. 이 옵션을 사용하면(자), 지정한 JAR 메인 파일과 그 메인 파일이 의존하고 있는 모든 JAR 파일에 대해, 패키지 위치 정보가 생성됩니다. 메인 파일이 의존하고 있는 JAR 파일은, JAR 메인 파일의 Manifest의 Class-Path 속성으로 지정해 둘 필요가 있습니다. 

C:\Java> jar i main.jar

위의 예에서는, INDEX.LIST 파일이, main.jar 의 Manifest의 META-INF 디렉토리에 삽입됩니다.
이 파일에 포함되고 있는 정보를 사용해, 어플리케이션의 클래스 로더는, 효율적으로 클래스를 로드합니다. 인덱스 파일에 위치 정보를 포함하는 방법의 자세한 것은, JarIndex 의 스펙을 참조해 주세요.

디렉토리를 카피하는 표준적인 방법은, 우선 dir1 내의 파일을 표준 출력에 압축해, 계속되어 표준 입력으로부터 dir2 에 추출합니다 (양쪽 모두의 jar 커멘드로부터 f 를 생략 한다). 

C:\Java> (cd dir1; jar c . ) | (cd dir2; jar x)

jar 툴을 사용해 JAR 파일과 그 Manifest를 조작하는 예는, 이후에의「예」라고,「Java 튜토리얼」의 「JAR Files」를 참조해 주세요.

옵션

c
jarfile 의 이름으로 새로운 archive파일을 작성하는지 (f 가 지정되고 있는 경우), 표준 출력으로 작성합니다 (f 와 jarfile 가 생략 되고 있는 경우). inputfiles 로 지정된 파일과 디렉토리를, 이 어카이브(archive)에 추가합니다.
u
inputfiles 로 지정된 파일과 디렉토리를 추가하는 것으로써, 기존의 파일 jarfile 를 갱신합니다 (f 가 지정되고 있는 경우). 예를 나타냅니다. 
jar uf foo.jar foo.class
위의 커멘드는, 파일 foo.class 를 기존의 JAR 파일 foo.jar 에 추가합니다. 다음의 예에 나타내도록(듯이),-u 옵션은, Manifest 엔트리도 갱신할 수 있습니다. 
jar umf manifest foo.jar
foo.jar Manifest를 Manifest내의이름:치페어로 갱신합니다.
x
jarfile 로부터 (f 가 지정되고 있는 경우), 또는 표준 입력으로부터 (f 와 jarfile 가 생략 되고 있는 경우), 파일과 디렉토리를 추출합니다. inputfiles 가 지정되고 있는 경우는, 그 지정한 파일과 디렉토리만이 추출됩니다. 지정되어 있지 않으면, 모든 파일과 디렉토리가 추출됩니다. 추출된 파일의 일시는, 어카이브(archive)내에서 설정된 것입니다.
t
jarfile 로부터 (f 가 지정되고 있는 경우), 또는 표준 입력으로부터 (f 와 jarfile 가 생략 되고 있는 경우), 내용을 겉(표) 형식에서 일람표 가리킵니다. inputfiles 가 지정되고 있는 경우는, 그 지정한 파일과 디렉토리만이 일람표 나타납니다. 지정되어 있지 않으면, 모든 파일과 디렉토리가 일람표 나타납니다.
i
지정된 jarfile 와 거기에 의존하는 JAR 파일에 대해, 인덱스 정보를 생성합니다. 예를 나타냅니다. 
jar i foo.jar

위의 커멘드는,foo.jar 내에 INDEX.LIST 파일을 생성합니다. 이 파일에는,foo.jar 의 foo.jarClass-Path 속성으로 지정된 모든 JAR 파일에 들어가 있는 각 패키지의 위치 정보가 기입해지고 있습니다. 인덱스의 예를 참조해 주세요.

f
작성 (c), 갱신 (u), 추출 (x), 인덱스 추가 (i), 또는 표시 (t) 대상의 파일 jarfile 를 지정합니다. f 옵션과 파일명 jarfile 가 페어가 됩니다. 즉, 다른 한쪽을 기술하면, 이제(벌써) 한편도 기술할 필요가 있습니다. f 와 jarfile 를 생략 하면(자), 표준 입력으로부터 「JAR 파일」이 취득되는지 (x 및 t 의 경우), 표준 출력에 「JAR 파일」이 송신됩니다(c 및 u 의 경우).
v
상세한 출력을 표준 출력에 생성합니다. 나중에예를 나타내겠습니다.
0
(제로) ZIP 에 의한 압축을 사용하지 않고 , 보존합니다.
M
Manifest 파일 엔트리를 작성하지 않습니다 (c 및 u 의 경우). 또는, Manifest 파일 엔트리가 존재하는 경우에 그것을 삭제합니다 (u 의 경우).
m
META-INF/MANIFEST.MF 의 파일로 지정한 Manifest 파일 manifest 의 name: value 의 속성 페어를 짜넣습니다. 같은 이름의 페어가 벌써 존재하고 있는 경우를 제외해,name: value 의 페어가 추가됩니다. 같은 이름의 페어가 존재하고 있는 경우는, 그 값이 갱신됩니다.

커멘드행으로,m 와 f 의 문자는,manifest 와 jarfile 의 입력 순서와 같은 순서로 기술할 필요가 있습니다. 예를 들어, 다음과 같이 해 사용합니다. 

jar cmf myManifestFile myFile.jar *. class
디폴트의 Manifest에는 포함되지 않는 Manifest에, 특별한 목적의 name : value 의 속성 페어를 추가할 수 있습니다. 예를 들어, 벤더 정보, 버젼 정보, 패키지 실링, 또는 JAR 에 번들 된 어플리케이션을 실행하기 위한 속성을 추가할 수 있습니다. -m 옵션의 사용예는, Java 튜토리얼의 「JAR Files」 (을)를 참조해 주세요.

-e
실행 가능 JAR 파일내에 번들 된 스탠드얼론 어플리케이션의 어플리케이션 엔트리 포인트로서entrypoint 를 설정합니다. 이 옵션을 사용하면(자), Manifest 파일내의 Main-Class 속성치가 작성 또는 덧쓰기됩니다. 이 옵션은, JAR 파일의 작성중 또는 갱신중에 사용할 수 있습니다. 이 옵션을 사용하면, Manifest 파일을 편집 또는 작성하는 것 없이 , 어플리케이션의 엔트리 포인트를 지정할 수 있습니다.

예를 들어, 다음의 커멘드는 Main.jar 를 작성합니다만, 그 때, Manifest 파일내의 Main-Class 속성치는 Main 로 설정됩니다. 
jar cfe Main.jar Main Main.class

다음의 커멘드를 실행하면(자) java 런타임으로부터 직접 이 어플리케이션을 기동할 수 있습니다. 

java -jar Main.jar
엔트리 포인트의 클래스명이 특정의 패키지내에 존재하고 있는 경우, 엔트리 포인트의 단락 문자로서 닷 ( 「.」) 또는 slash ( 「/」)를 사용할 수 있습니다. 예를 들어,Main.class 가 foo 라는 이름의 패키지내에 존재하고 있는 경우, 다음과 같이 해 엔트리 포인트를 지정할 수 있습니다. 
jar -cfe Main.jar foo/Main foo/Main.class
또는 
jar -cfe Main.jar foo.Main foo/Main.class
주:  -m 옵션과 -e 옵션의 양쪽 모두를 동시에 지정하는 경우, 지정된 Manifest에도 Main-Class 속성이 포함되어 있으면(자),Main.class 의 지정이 애매하게 되어 에러가 발생해, JAR 의 작성 또는 갱신 처리가 이상종료(ABEND) 합니다.

-C  dir
jar 커멘드의 실행중에 후속의 inputfiles 인수를 처리할 경우에, 일시적으로 디렉토리를 변경합니다 (cd dir). 이 처리는, UNIX 의 tar 유틸리티의 -C 옵션의 기능에 유사하고 있습니다. 예를 나타냅니다. 
jar uf foo.jar -C classes bar.class
위의 커멘드는,classes 디렉토리로 옮겨, 그 디렉토리로부터 bar.class 를 foo.jar 에 추가합니다. 다음에 다른 예를 나타냅니다. 
jar uf foo.jar -C classes .  -C bin xyz.class
classes 디렉토리로 변경해,classes 디렉토리내의 모든 파일을,foo.jar 에 추가합니다 (이 JAR 파일내에 클래스 디렉토리는 작성하지 않는다). 다음에, 원의 디렉토리에 돌아와서,bin 디렉토리로 변경해 xyz.class 를 foo.jar 에 추가합니다. classes 에 bar1 와 bar2 가 보관 유지되고 있는 경우,jar tf foo.jar 를 사용하면(자), JAR 파일에는 다음의 요소를 포함할 수 있습니다. 
META-INF/

META-INF/MANIFEST.MF

-Joption
Java 실행 환경에 option 를 건네줍니다. option 에는,「Java 어플리케이션 기동 툴」의 레퍼런스 페이지에 기재되어 있는 옵션을 1 개 지정합니다. 예를 들어,-J-Xmx48M 와 지정하면(자), 최대 메모리는 48M 바이트로 설정됩니다. -J 를 사용해 배후의 실행 환경에 옵션을 건네주는 것은 잘 행해지고 있습니다.

커멘드행 인수 파일

jar 의 커멘드행을 짧게 하거나 간결하게 하거나 하기 위해서,jar 커멘드에 대한 인수 (-J 옵션을 제외하다)를 포함한 1 개(살) 이상의 파일을 지정할 수가 있습니다. 이것에 의해, 임의의 길이의 jar 커멘드를 작성할 수 있어 operating system에 의한 커멘드행의 제한으로부터 해제됩니다.

인수 파일에는 옵션과 파일명을 포함할 수가 있습니다. 파일내의 각 인수는, 스페이스 또는 개행으로 단락짓습니다. 인수 파일내의 파일명은, 현재의 디렉토리로부터 본 상대 패스가 됩니다. 인수 파일의 위치로부터 본 상대 패스가 아닙니다. 와일드 카드 (*)는, 일반적으로이라면 operating system 쉘에 의해 모든 문자로 해석됩니다만, 이 경우는 그처럼은 해석되지 않습니다. 인수 파일내의 인수로 @ 문자를 사용해, 복수의 파일을 재귀적으로 해석하는 것은 지원되고 있지 않습니다. 또,-J 옵션도 지원되고 있지 않습니다. 이 옵션은 기동 툴에게 건네집니다만, 기동 툴에서는 인수 파일을 지원하고 있지 않기 때문입니다.

jar 를 실행할 경우에, 각 인수 파일의 패스와 파일명의 선두에 @ 문자를 붙여 건네줍니다. jar 는,@ 문자로 시작되는 인수를 찾아내면(자), 그 파일의 내용을 전개해 인수 리스트에 삽입합니다.

예를 들어, 다음과 같이 해, 「classes.list」라고 하는 단일의 인수 파일에, 복수의 파일의 이름을 보존합니다. 

C:\Java> dir /b *. class > classes.list

계속되어, 이 인수 파일내의 파일명을 건네주는 jar 커멘드를 실행합니다. 

C:\Java> jar cf my.jar @classes.list
인수 파일은, 패스를 붙여 건네줄 수가 있습니다만, 상대 패스를 붙인 인수 파일 내부의 모든 파일명에는, 건네받는 패스는 아니고, 현재의 작업용 디렉토리에 대한 상대 패스를 붙일 수 있습니다. 이 예를 다음에 나타냅니다. 
C:\Java> jar @path1/classes.list

클래스 로드를 고속으로 하기 위해서 인덱스 파일을 JAR 파일에 추가하려면 , 「i」옵션을 사용합니다.

주식 거래 어플리케이션용의 상호의존한 클래스를, 3 개의 JAR 파일,   main.jar,   buy.jar, 및   sell.jar 에 분할해 봅시다. main.jar 의 Manifest의 Class-path 속성에 다음과 같이 지정했을 경우, 
Class-Path: buy.jar sell.jar
계속되어 다음과 같이 i 옵션을 사용하면(자), 어플리케이션의 클래스 로드를 고속화할 수 있습니다. 
C:\Java> jar i main.jar
이 경우,INDEX.LIST 파일이 META-INF 디렉토리에 추가됩니다. 이것에 의해, 어플리케이션의 클래스 로더가 클래스 또는 자원을 검색할 때에, 지정된 JAR 파일을 다운로드할 수 있게 됩니다.
