컴퍼넌트의 버젼

이하의 버젼의 관련 테크놀러지가 레퍼런스 구현에 짜넣어지고 있습니다.
  • Xerces 버젼 2.6. 2 + (Xerces 버젼 2.6. 2 와 해결 지음 버그 수정)
  • XSLTC 버젼 2.6. 0 + (XSLTC 버젼 2.6. 0 과 해결 지음 버그 수정. Xalan 2.6. 0 릴리스 베이스)

JAR 파일의 특정

이 릴리스에는, 6 개의 JAR 파일이 포함되어 있습니다.

jaxp-api.jar
JAXP 의 javax.xml.parsers,javax.xml.transform,javax.xml.xpath,javax.xml.validation,javax.xml.datatype 및 javax.xml.namespace 컴퍼넌트. XML 처리 구현의 인스턴스를 어플리케이션이 일관한 방법으로 취득하기 위한 API 가 포함되어 있습니다.  javax.xml.xpath 패키지에서는, 표준 XPath API 를 지원하고 있습니다.
sax.jar
Simple API for XML (SAX)의 API 와 헬퍼 클래스. XML 데이터에의 직렬 액세스에 사용합니다.  API 에서는 SAX 버젼 2.0. 2 를 지원하고 있습니다.
dom.jar
Document Object Model (DOM)의 API 와 헬퍼 클래스. 메모리내의 트리 구조를 XML 데이터로부터 작성하기 위해서 사용합니다.  API 에서는 DOM Level 3 을 지원하고 있습니다.
xercesImpl.jar
SAX 퍼서와 DOM 퍼서, 및 JAXP API 의 Xerces 고유 구현의 구현 클래스.
 
xalan.jar
Xalan 프로젝트에 의한 콘파이링 XSLT 프로세서인 XSLTC 가 포함됩니다.

XML 의 구문 분석

이 섹션내의 정보는, Xerces 테크놀러지에 관계합니다. 최신의 업데이트는, 항상 http://jaxp.dev.java.net 로부터 입수할 수 있습니다.  최신의 Apache 버젼에 있어서의 기존의 버그와 최신의 버그 수정에 대해서는,http://xml.apache.org/xerces2-j/releases.html 를 참조해 주세요.  

schema 처리의 기존의 제한

이 섹션에서는, schema 처리의 기존의 버그, 제한, 및 구현 의존의 조작에 대해 설명합니다.

제한

다음의 제한에서는, 상한치를 지정합니다.
  • Length, minLength, 및 maxLength 의 각 파셋트는, 2147483647 에 제한된다.
    이 제한보다 큰 항목은 올바르게 검증되지 않는다
       
  • 일자/시각의 데이터형에 있어서의 해와 초의 값은, 2147483647 에 제한된다.
    이 제한보다 큰 값은 올바르게 검증되지 않는다

구현 의존의 조작

다음의 구현 의존의 조작은, W3C XML schema 스펙 (http://www.w3.org/2001/XMLSchema )에서도 완전하게 정의되고 있지 않습니다. 그 결과, 다양한 구현이 존재하고 있습니다.
  • JAXP 1.2 레퍼런스 구현 퍼서 (Xerces 2.0. 1) <keyref> 일의성 제한에서는, 「XML Schema Part 1: Structures」로 정의되고 있도록(듯이),<keyref> 가 접속된 요소의 범위내의 <key> 또는 <unique> 일의성 제한을 참조하고 있습니다. 한편, XML schema의 도입 섹션 ( 「XML Schema Part 0:Primer」)에는, 대응하는 <key> 의 요소내에서 사용되고 있는 요소로 선언된 <keyref> 의 예가 실려 있습니다. 혼란을 피하기 위해서(때문에), 이와 같이 일관성이 없는 점을 여기서 지적해 둡니다.

이행의 기존의 문제

이 섹션에서는, 이전의 버젼의 JAXP 로부터 이행 할 경우에 발생하는 기존의 문제를 취급합니다.

JAXP 1.2 로부터의 이행

JAXP 1.2 는 JWSDP 와 J2EE 1.4 에 짜넣어지고 있습니다. 「JAXP 호환성 가이드」의 다음의 섹션에서는, 관련하는 이행의 문제를 취급하고 있습니다.

  • IOException added to SAX EntityResolver API
    SAX 2.0. 1 API 에 더해진 변경점
  • Methods added to DOM interfaces
    DOM Level 3 API 에 더해진 변경점
  • Package Name Changes
    구현 고유의 기능에 액세스 하는 어플리케이션으로 영향이 있는 변경에 대해

JAXP 1.1 으로부터의 이행

JAXP 1.1 은 J2EE 1.3 으로 J2SE 1.4 에 짜넣어지고 있습니다. JAXP 1.1 과의 기능상의 차이에 대해서는,「JAXP 호환성 가이드」를 참조해 주세요.

DOM Level 3 구현에 대해

이 섹션에서는, DOM Level 3 Core, DOM Level 3 Load, 및 Save 에 관한 구현에 대해 설명합니다.

DOM Level 3 Core

다음은 구현되고 있지 않습니다.

  • DOMConfiguration 의 「infoset」파라미터 - 치 「true」는 지원하지 않는다

지원하는 파라미터는 다음과 같습니다.

  • error-handler [DOMErrorHandler ]
  • resource-resolver [DOMResourceResolver ]
  • schema-type [String ]
  • schema-location [String ]
  • canonical-form [false ]
  • cdata-sections [true/false ]
  • comments [true/false ]
  • namespaces [true/false ]
  • entities [true/false ]
  • infoset [ false ]
  • namespace-declarations [true ]
  • normalize-characters [false ]
  • split-cdata-sections [true/false ]
  • validate-if-schema [true/false ]
  • whitespace-in-element-content [true ]
  • datatype-normalization [true/false ]
  • validate [true/false ] -- XML schema 에 대해서만, 재검사를 지원한다. 자세한 것은,How to revalidate DOM in memory? (을)를 참조 .
  • validate-if-schema [true/false ]
  • check-character-normalization [false ]
  • well-formed [true/false ]
  • psvi [true/false ] -- (비 DOM) 이 파라미터의 값을 true 로 설정하면(자), DOM 트리내의 PSVI 정보를 포함한다. 정보에 액세스 하려면 , dom.Element 를 org.apache.xerces.xs.ElementPSVI 에, dom.Attr 를 org.apache.xerces.xs.AttributePSVI 에 캐스트 한다

DOM Level 3 Load 및 Save

다음은 지원하고 있습니다.

  • LSParser parserWithContext
  • DOMImplementationLS MODE_ASYNCHRONOUS
  • LSSerializer - 다음의 파라미터는 지원하지 않는다
    • ignore-unknown-character-denormalizations [true ]
    • normalize-characters [true ]
    • well-formed [true ]

콘파이링 XSLT 프로세서 (XSLTC)

XSLTC 트랜스 포머는, 변환 엔진인 「translet」를 XSL 스타일 시트로부터 생성합니다. 이렇게 하는 것으로, 실행시 어플리케이션으로부터 XML 데이터에의 스타일 시트 지시의 해석을 분리할 수 있습니다.

XSLTC 는, 스타일 시트를 Java 바이트 코드 (translet)에 컴파일 하는 것으로 기능합니다. translet 는, XSLT 변환을 실행하기 위해서 사용됩니다. 이 방법에서는, 지정된 스타일 시트가 1 회 컴파일 되어 여러 차례 사용되기 (위해)때문에, XSLT 변환의 퍼포먼스가 크게 향상합니다. 또, 실제로 스타일 시트로 사용되고 있는 XSLT 지시만이 포함되기 (위해)때문에, 매우 경량인 translet 도 생성됩니다.

주:
XSLT 는 JAXP transform 패키지로 지원하고 있습니다. 구현에 의존하지 않는 방법으로 XSLT 의 기본적인 기능에 액세스 하는 자세한 것은,javax.xml.transform 를 참조해 주세요.

커스텀 클래스 로더의 문제

커스텀 클래스 로더를 변환 팩토리와 함께 사용하면(자), 문제가 발생하는 경우가 있습니다.

JAXP 에 있어서의 변환 팩토리에서는, 시스템 클래스 로더보다 문맥 클래스 로더가 항상 우선해 사용됩니다. 그 때문에, 어플리케이션으로 커스텀 클래스 로더를 사용하고 있을 때는, 커스텀 클래스 로더를 문맥 클래스 로더로서 변환 팩토리가 사용하도록(듯이) 설정할 필요가 있는 경우가 있습니다. 커스텀 클래스 로더를 현재의 thread로 설정하려면 , 다음과 같이 합니다. 

try {
  Thread currentThread = Thread.currentThread();
  currentThread.setContextClassLoader(customClassLoader);
}
catch (SecurityException e) {
  // ...
}

어플리케이션이 multi-thread화 되고 있는 경우는, 새로운 thread가 작성될 때마다, 커스텀 클래스 로더를 모든 thread로 설정할 필요가 있는 경우가 있습니다. 어플리케이션에 문맥 클래스 로더를 설정하는 권한이 없는 경우는, 시큐리티 예외가 throw 됩니다.

JWSDP 시큐리티에 대해

어플리케이션을 Java Web Services Developer Pack (JWSDP)등의 Web 서버상에서 실행하고 있어, 시큐리티를 유효하게 하고 있는 경우는, 다음의 권한을 설정할 필요가 있습니다. 

permission java.io.FilePermission

    "/${webserver.home}/common/endorsed/xercesImpl.jar", "read";
permission java.io.FilePermission
    "/${webserver.home}/common/endorsed/xalan.jar", "read";

permission java.util.PropertyPermission
    "javax.xml.parser.SAXParserFactory", "read, write";
permission java.util.PropertyPermission
    "javax.xml.transform.TransformerFactory", "read, write";

permission java.util.PropertyPermission "user.dir",        "read";
permission java.util.PropertyPermission "file.separator",  "read";
permission java.util.PropertyPermission "line.separator",  "read";
permission java.util.PropertyPermission "JavaClass.debug", "read";

permission java.lang.RuntimePermission "createClassLoader";
permission java.lang.RuntimePermission "accessDeclaredMembers";

주:
읽기 권한을 xercesImpl.jar 로 설정하지 않으면 다음과 같은 FactoryConfigurationError 가 throw 됩니다.
      Provider org.apache.xerces.jaxp.SAXParserFactoryImpl not found

JAXP RI 버젼의 변경점

JAXP RI 버젼 1.3 의 변경점

  • Xerces 버젼이 2.3. 0 으로부터  2.6. 2 로 변경되었다 (+ 해결 지음 버그 수정)
  • JAXP 1.3 의 스탠드얼론 버젼으로 입수 가능한 XSL 변환 엔진은, Xalan 해석과 XSLTC 의 2 개 (Xalan 2.6. 0 베이스 (+ 해결 지음 버그 수정)).  J2SE 5 에 포함되는 JAXP 1.3 버젼에는, XSLTC 트란포마만이 포함되는   (Xalan 2.6. 0 베이스 (+ 해결 지음 버그 수정)).
  • 「호환성 가이드」의「 신기능」으로 설명한 신기능이나 특징이 추가되었다
  • 시스템 프로퍼티 elementAttributeLimit 를 추가하는 것으로, 서비스 방해 (DoS) 취약성을 수정했다. 이 프로퍼티은 요소내의 속성수를 제한하기 위해서 사용한다. 1 개의 요소의 속성의 디폴트의 최대치는 10,000

JAXP RI 버젼 1.2. 4 의 변경점

  • Xerces 에 있어서의 J2SE 1.4. 2 의 컴파일 버그를 수정

JAXP RI 버젼 1.2. 3 의 변경점

  • Xerces 버젼이 2.2. 1 에서 2.3. 0 으로 변경되었다 (+ 해결 지음 버그 수정). 이 버젼에는 많은 버그 수정이 포함되어 있다
  • schema 스펙에 의해 무효로 되어 있던 모든 10 진수값가, 퍼서로 인식할 수 있게 되었다 (1.2. 2 에서의 제한을 없앴다)
  • JDK 1.4. 1 에서의 Xerces 컴파일의 문제를 해결했다
  • XML schema의 로드 퍼포먼스가 큰폭으로 향상했다 (실질적으로 반이 되었다)

JAXP RI 버젼 1.2. 2 의 변경점

  • 시큐리티를 향상시키기 위한 새로운 프로퍼티을 구현했다. 「시큐리티의 문제」를 참조
  • 퍼서 구현이 Xerces 2.0. 1_04 (으)로부터 Xerces 2.2. 1 으로 변경되었다 (+ 해결 지음 버그 수정)
  • XSLT 프로세서 구현이 Xalan 2.3. 1_01 (으)로부터 Xalan 2.4. 1 으로 변경되었다 (+ 해결 지음 버그 수정)
  • XSLTC 가 이 릴리스에 포함되어 있다

JAXP RI 버젼 1.2. 1 의 변경점

Xerces 퍼서의 퍼포먼스가 큰폭으로 향상했습니다.

XSLTC 는 이 릴리스에는 포함되지 않습니다. 단독으로 J2EE 플랫폼에 준비되어 있었습니다.  

JAXP RI 버젼 1.2. 0-FCS 의 변경점

퍼서 구현이 Xerces 2.0. 0_01 (으)로부터 Xerces-J 2.0. 1_01 (으)로 변경되었습니다
(Xerces 2.0. 1 최종판 + 해결 지음 버그 수정). Xalan XSLT 프로세서
구현이 xalan-j 2.3. 1_01 (Xalan 버젼 2.3. 1 +
해결 지음 버그 수정)에 갱신되었습니다.

제안된 1.2 JAXP 스펙을 완전하게 지원합니다.
이 스펙에서는, W3C XML schema를 사용한 문서 검증을 구현하고 있습니다.

JAXP RI 버젼 1.2. 0-EA2 의 변경점

퍼서 구현이 Xerces 2.0. 0 beta3 로부터 Xerces-J 2.0. 0_01 (으)로 변경되었다 (Xerces 2.0. 0 최종판 + 해결 지음 버그 수정). Xalan XSLT 프로세서 구현이 xalan-j 2.3. 0_01 (Xalan 버젼 2.3. 0 + 해결 지음 버그 수정)에 갱신되었습니다.

Xalan XSLTC 프로세서도 본릴리스로 추가되었습니다. (실행 준비를 할 수 있던 변환 엔진 (translat)에 스타일 시트를 컴파일 하기 위해서 사용)

제안된 1.2 JAXP 스펙을 완전하게 지원합니다. 이 스펙에서는, W3C XML schema를 사용한 문서 검증을 구현하고 있습니다.

JAXP RI 버젼 1.2. 0-EA1 의 변경점

퍼서 구현이 Apache Crimson 로부터 Xerces 2 버젼 2.0. 0 beta3 로 변경되었습니다. XSLT 프로세서 구현이 Xalan classic 버젼 2.2.D14 에 갱신되었습니다.

퍼서에서는 W3C XML schema를 지원하고 있습니다만, 제안된 JAXP 1.2 스펙의 모든 것은 지원하고 있습니다. 특히, 인스턴스 문서가 특정의 schema에 준거하는 것을 강제하는 기능은 구현되고 있지 않습니다. 다만, 스펙중 검증의 부분은, 인스턴스 문서내에서 schema 힌트와 함께 사용할 수 있습니다.
