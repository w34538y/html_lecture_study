HelloServer 를 실행하려고 하면(자) 다음과 같은 메세지가 표시됩니다. 왜입니까. "Exception in thread "main" java.lang.NoClassDefFoundError:HelloServer/java"
많은 경우, Java IDL 튜토리얼의 HelloServer (HelloClient 의 경우도 마찬가지)를 실행하면(자), 이 에러가 발생하는 것은,HelloServer 가 클래스 패스에 포함되지 않기 때문입니다. 클래스 패스 변수의 설정 방법법의 자세한 것은,「클래스 패스의 설정」 을 참조해 주세요.

2 번째로 많은 원인은, 다음의 코드행의 인용부호와 인용부호의 사이에 공백이 추가되고 있는 것입니다. 

NameComponent nc = new NameComponent("Hello", "");

인용부호의 사이에는 공백을 들어갈 수 있지 말아 주세요. 공백이 없으면, 이 값은 Null 로서 건네받습니다. 인용부호의 사이에 공백이 있으면(자), 공백 문자가 건네받습니다.

이번 릴리스로 기존의 프로그램을 실행할 수 없습니다. 호환성에 문제가 있습니까.

J2SE 1.4.x 로 J2SE 5.0 의 사이의 호환성에 관한 정보는,호환성의 문서를 참조해 주세요.

IOR 에 127.0. 0.0 의 호스트 주소가 포함되어 있으면(자), Linux 서버로부터 IOR 를 사용해 접속할 수 없습니다. 왜입니까.

Red Hat Linux 를 인스톨 했을 경우,InetAddress.getLocalHost() 가, 루프백 주소 (127.0. 0.1)에 대응하는 InetAddress 를 돌려주는 일이 있습니다. 이 문제가 발생하는 것은, 디폴트의 인스톨에 의해, 머신의 호스트명과 루프백 주소와의 관련짓고가 /etc/hosts 내에 작성되기 (위해)때문에입니다. InetAddress.getLocalHost() 에 의해 확실히 실제의 호스트 주소가 돌려주어지도록(듯이) 하려면 ,/etc/hosts 파일 또는 네임 서비스의 구성 파일 (/etc/nsswitch.conf)을 갱신해, 호스트를 검색하기 전에, dns 또는 nis 에 문의하도록(듯이) 합니다.

마이너 코드의 의미는 어떻게 하면 압니까.

최신 버젼의 표준 (OMG) 마이너 코드 예외에 대해서는, OMG 의 Web 사이트 (http://www.omg.org/ )를 참조해 주세요.

자주(잘) 조우하는 Sun 의 마이너 코드 예외에는, 다음과 같은 것이 있습니다.

  • COMM_FAILURE/201. 「vmcid: SUN 마이너 코드: 201」은, 「CONNECT_FAILURE」를 의미합니다. 이것은,java.net.SocketException 가 원인으로 생기는 일이 있어, 일반적으로은 BindException,ConnectException, 또는 NoRouteToHostException 의 머지않아인가입니다.

    다음과 같은 점을 확인할 필요가 있습니다.

    • 네임 서비스는 실행되고 있을까. 실행되어 있지 않은 경우는,「네임 서비스의 기동과 정지」의 설명에 따라,ORDB 네임 서비스를 개시합니다.
    • 네임 서비스에 대해서 -ORBInitialHost 및 -ORBInitialPort 의 값이 올바르게 설정되어 있을까. 어떻게 설정하면(자) 좋은가 모르는 경우는,「네임 서비스의 기동과 정지」를 참조해 주세요.
    • 클라이언트 어플리케이션과 서버 어플리케이션은, 네임 서비스가 실행되고 있는 포트 번호 (필요한 경우는, 한층 더 머신명)를 인식하고 있을까. 이것을 실시하는 방법에 대해서는,「네임 서비스의 기동과 정지」를 참조해 주세요.

  • COMM_FAILURE/208. 「vmcid: SUN 마이너 코드: 208」은, 「CONNECTION ABORT」를 의미합니다. 이것은 일반적으로, 접속이 끊어진 것을 나타냅니다.

  • COMM_FAILURE/209. 「vmcid: SUN 마이너 코드: 209」는, 「CREATE_LISTENER_FAILED: Unable to create the listener thread on the specific port. Either the post is taken or there was an error creating the daemon thread」를 의미합니다. 이것은 일반적으로, 네임 서비스를 실행하려고 한 포트가 다른 프로세스에 의해 사용중인 것을 나타내고 있습니다. Solaris 상에서 실행하고 있는 경우에는, 이 포트로 다른 프로세스가 실행중인가 어떤가를 조사하기 위해서(때문에), 단말 prompt로 다음의 커멘드를 실행합니다. 

    netstat | grep port_number

  • COMM_FAILURE/204. 「vmcid: SUN 마이너 코드: 204」는, 「SERVANT_NOT_FOUND」를 의미합니다. 이것이 throw 되는 유일한 장소는,corba.INSSubcontract.getINSReference 입니다.

  • MARSHAL/217. vmcid: SUN 마이너 코드: 217」은, 클라이언트가 GIOP 1.0 으로 wchar 나 wstring 의 어느쪽이든을 송신하려고 한 것을 의미합니다. 이것은, 스펙으로 허가되고 있지 않습니다.

  • MARSHAL/202. 「vmcid: SUN 마이너 코드: 202」는, org.omg.CORBA.Object 로부터 파생한 객체를 정렬화하려고 했지만, 그 특정의 인스턴스가 ORB 에 접속되어 있지 않은 것을 의미합니다. POA 를 사용하는 경우는, 우선 객체를 POA 에 등록할 필요가 있습니다. 객체를 POA 에 등록하는 방법의 상세한 것에 대하여는,POA 의 문서 또는튜토리얼을 참조해 주세요.

  • BAD_PARAM/201. 「vmcid: SUN 마이너 코드: 201」은, 「NULL_PARAM」를 의미합니다. 이 예외는, 많은 경우,write_string 나 write_octet_array 등의 write 메소드에 대해서 Java 의 null 를 건네주었을 때에 발생합니다. Java 메소드의 결과로서 Java 의 null 를 돌려줄 수 없습니다.

  • org.omg.CORBA.INTERNAL. 「vmcid: SUN 마이너 코드: 208」은, 「Unable to determine local hostname using InetAddress.getLocalHost(). getHostName()」를 의미합니다.

    ORB 는,InetAddress.getLocalHost(). getHostName() 를 사용해, 참조의 검색이나 바인드를 위해서(때문에) 네임 서비스에의 참조를 작성합니다. 또, 서버측에서 InetAddress.getLocalHost(). getHostName() 를 사용해, 닷 단락 10 진수와 포트의 편성은 아니고, 서버의 이름과 포트를 포함한 원격 객체 참조 (즉 IOR)를 작성합니다.

    getHostName 의 호출을 회피하려면 , 다음과 같은 프로퍼티을 설정합니다 (설정 방법법에 대해서는,「네임 서비스의 기동과 정지」를 참조).

    • ORB 가 서버로서 기능하고 있는 경우는,com.sun.CORBA.ORBServerHost 에 서버의 DNS 명 또는 닷 단락 10 진수 주소를 설정
    • com.sun.CORBA.ORBInitialHost 에, 네임서버-의 DNS 명 또는 닷 단락 10 진수 주소를 설정

    주: 이러한 프로퍼티은 독자적인 것으로, 삭제 또는 변경되는 일이 있습니다.

servertool 를 사용해 서버를 기동할 때, 복수의 클래스 패스를 추가할 수 있습니까.

이 질문은, 원래 다음과 같은 내용이었습니다.

나의 서버 코드에서는, 클래스 패스에 복수의 JAR 파일이 필요합니다. 거기서,servertool 를 사용해 서버를 등록할 경우에 클래스 패스를 지정하려고 했습니다만, register 커멘드로 구문 에러가 일어납니다. 사용한 커멘드행의 길이가 제한을 넘고 있는 것처럼 생각됩니다.

servertool 로 서버를 등록할 때, 매우 긴 클래스 패스를 지정하기에는 어떻게 하면 좋은 것일까요.

servertool 그 자체는 복수의 JAR 파일을 받아들입니다만, 현재로서는,servertool 의 커멘드행 버퍼의 길이에 관해서 버그가 있습니다. servertool 래퍼의 버그 (버그 4482166)의 상세한 것에 대하여는,http://developer.java.sun.com/developer/bugParade/bugs/4482166.html 를 참조해 주세요.

회피책으로서 servertool 내로부터 서버를 기동하는 대신에,ServerTool 클래스를 호출해 서버를 기동할 수가 있습니다. 예를 들어, 다음과 같이 합니다. 

${JAVA_HOME}/bin/java com.sun.corba.se.internal.Activation.ServerTool 
-ORBInitialPort ${ORB_INITIAL_PORT} -cmd register -server 
sample.MyServer -classpath jar1:jar2:jar3 -applicationName sample

servertool 클래스의 이름은, 장래의 릴리스로 변경이 될 가능성이 있는일로 주의해 주세요.

Linux 플랫폼에서 네임 서비스 (tnameserv,orbd)를 실행할 수 없는 경우에는, 어떻게 하면 좋을까요.

Linux 상에서 네임 서비스의 기동이 실패하는 경우는,/etc/hosts 파일에 다음의 행을 짜넣습니다. 

    <Local Host IP Address>   localhost
<Local Host IP Address>   <HostName>

여기서 <Local Host IP Address> 는, 네임 서비스 (tnameserv,orbd)를 기동하는 호스트의 IP 주소입니다.


기능

이번 릴리스로, CORBA 의 구현은 어떠한 thread 모델을 지원하고 있습니까.

Java SE 에 동고 되고 있는 Java CORBA ORB 는 multi-thread화 되고 있습니다. 서버 측에 준비되어 있는 thread 풀에서는, 각 착신 요구가 독립한 thread에 의해 처리됩니다. 풀의 thread가 모두 사용중 때에 새로운 요청를 받으면(자), 새로운 thread가 작성되어 풀에 추가됩니다. 이 thread는, 요구가 완료하면(자), 풀에 되돌려집니다.
Java CORBA ORB 는, 스케이라비리티와 요청의 동시 처리를 실현하기 위해서(때문에), thread화 되고 있습니다. POA thread 정책용의 SINGLE_THREAD 옵션은 지원되고 있지 않습니다.
Java CORBA ORB 의 thread 모델은 다음과 같이 암묵적입니다. 즉, 사용자는, ORB 의 thread 정책를 설정하지 않습니다. 또, thread 모델이나 thread수를 제어하기 위한 , 공개된 외부의 사용자 레벨 API 는 없습니다.

Java IDL 에는 통지 서비스나 이벤트 서비스가 있습니까. 인터페이스 리포지터리(repository)는 어떻습니까.

아니오, 없습니다. 이러한 서비스가 필요한 경우는, 스스로 구현하는지, 기성품을 구입하는지, 자유롭게 이용할 수 있는 서비스를 찾을 수가 있습니다. 그러한 서드 파티의 서비스는, INS 테크놀러지를 사용해 ORB 에 플러그 인 할 수 있습니다.

CORBA 클라이언트와 EJB 서버 어플리케이션의 사용예는 제공되고 있습니까.

            -ORBInitRef NameService=corbaloc:iiop:1. 2@:/NameService

orb.resolve_initial_references( "NameService" ) 를 실행하면(자), 서드 파티제의 네임 서비스에 접속할 수 있게 됩니다. 접속할 수 없는 경우는, 이하의 트러블 슈팅에 관한 힌트를 시험해 주세요.

  • 서드 파티제의 네임 서비스가 INS 를 지원하고 있는 것을 확인한다
  • 호스트와 포트의 정보가 올바른 것을 확인한다
  • 서드 파티제의 네임 서비스가 올바르게 기동되고 있는 것을 확인한다
  • 서드 파티제의 네임 서비스가 GIOP 1.2 를 지원하고 있는 것을 확인하는 지원하고 있지 않는 경우는, 네임서버-의 문서를 참조해, 적절한 GIOP 의 버젼을 확인해, 그 버젼에 맞추어 corbaloc:URL 를 수정한다
  • 서드 파티제의 네임 서비스가 다른 객체 키를 사용해 NameService 와 통신하고 있을지 어떨지를 조사한다. 그 경우는, 네임서버-의 문서를 참조한다

C++ 의 CORBA 서버와 함께 Java IDL ORB 를 사용할 수 있습니까. (상호 운용성이 있습니까)

Java IDL ORB 는 완전하게 Java 테크놀러지를 사용해 기술된 ORB 입니다. idlj 컴파일러는, 「IIDL to Java 언어 매핑 스펙」으로 정의된 규약에 따르는 코드를 생성합니다. Java ORB 는, Java 플랫폼 이외의 언어로 코드를 생성하는 컴파일러를 제공하지 않습니다. Java ORB 와 그 이외의 언어 (C++ 등)로 기술한 ORB 와의 상호 운용성을 테스트하는 경우, 그 언어로 기술한 ORB 와 그 언어 매핑에 일치하는 컴파일러를 찾을 필요가 있습니다. 언어 매핑 스펙은, Object Management Group 의 Web 사이트 (http://www.omg.org/ )로부터 입수할 수 있습니다. 한편으로 Java 플랫폼을 사용해, 이제(벌써) 한편으로 C++ 를 사용하려면 , IDL 를 공유하는 것만으로 충분합니다. C++ ORB 의 툴을 사용해 C++ ORB 로 사용하는 Stub와 스켈리턴을 생성할 필요가 있습니다만, IDL 는 변경할 필요가 없습니다. CORBA ORB 와 사용하는 언어의 언어 매핑 컴파일러를 제공하는 벤더는, 「C++ CORBA ORB」등의 키워드로 Web 를 검색합니다.

C++ 클라이언트와 Enterprise JavaBeans 컴퍼넌트를 사용한 샘플 어플리케이션을 포함한 개발자 가이드가, Web 사이트 http://java.sun.com/j2se/1. 5.0/docs/technotes/guides/rmi-iiop/interop.html 에 준비되어 있습니다.

다른 언어로 쓰여진 ORB 간에서의 통신이 가능할 것입니다가, Java ORB 와 다른 벤더의 ORB 와의 상호 운용성은 아직 테스트되고 있지 않습니다.


전반

Java IDL 와 Java RIM-IIOP 테크놀러지의 어느 쪽을 사용해야할 것인가

이것은 기본적인 문제이며, Java 프로그램 언어와 CORBA 를 통합하는 2 개의 방식의 차이를 이해하는 것이 중요합니다.

Java IDL 테크놀러지는, CORBA 인터페이스 정의 언어 (Interface Definition Language, IDL)로 정의된 인터페이스에 근거해 Java 프로그램 언어로 프로그램을 기술하고 싶은 CORBA 프로그래머전용입니다. 이것은 「일반적으로 대로의」CORBA 프로그래밍으로, C++ 나 COBOL 와 같은 다른 언어와 완전히 같은 방법으로 Java 를 지원하고 있습니다.

Java RMI-IIOP (Remote Method Invocation over Internet Inter-ORB Protocol) 테크놀러지는, IIOP 를 배후의 트랜스폴트로서 사용해 Java Remote Method Invocation (Java RMI) 인터페이스로 프로그램을 작성하고 싶은, Java 프로그래머전용입니다. RMI-IIOP 는 다양한 언어로 구현되는 CORBA 객체와의 상호 운용성을 제공합니다만, 원격 인터페이스를 미리 Java RMI 인터페이스로서 정의해 둘 필요가 있습니다. EJB 의 원격 객체 모델은 Java RMI 테크놀러지 베이스이므로, Enterprise JavaBeansTM (EJBTM) 테크놀러지를 사용하는 프로그래머에게는 특히 중요합니다.

idltojava 컴파일러는 어디서 다운로드할 수 있습니까.

idltojava 컴파일러를 다운로드할 수 없게 되었습니다. IDL-to-Java 컴파일러 idlj 의 최신 버젼을 사용하는 것을 강하게 추천합니다.

최신 버젼의 IDL-to-Java 컴파일러를 입수하려면 , 최신 버젼의 Java TM 2 Platform, Standard Edition (J2SE TM) 를 다운로드해 주세요. Java SE 를 인스톨 하면(자),idlj 는 bin 디렉토리에 포함됩니다.

J2SE v1. 3 의 idltojava 에 관한 그 외의 트러블 슈팅 정보에 대해서는,「Java IDL FAQ」를 참조해 주세요.

Java 테크놀러지를 사용해 CORBA 어플리케이션을 개발하는 경우에 어떠한 옵션이 있습니까.

Java 2 플랫폼의 일부인 CORBA 기술은, Java 언어로 쓰여진 Object Request Broker (ORB) (일부는 native code로 쓰여져 있다),RMI 프로그래밍 모델, 및 IDL 프로그래밍 모델로 구성되어 있습니다.
다른 언어간, 다른 벤더간의 상호 운용성은 Internet InterORB Protocol (IIOP)라고 하는 「마법」을 사용해 실현되었습니다. IIOP 는 IDL 또는 Java RMI 의 어느 쪽인지로 쓰여진 만큼 산어플리케이션용의 전송 프로토콜로 할 수 있습니다. IIOP 를 사용하면, 분산 객체를 OMG 의 CORBA 스펙에 적합시킬 수가 있습니다.
IDL 프로그래밍을 사용하는 경우는, 인터페이스가 방법이라고입니다. IDL 에서는, 원격 접근으로부터 불러 출일이 생기는 엔트리 포인트 (불려 간 수속이 받아들이는 인수의 형태등)나, 반환된 정보의 값이나 출력 파라미터등을 정의합니다. IDL 를 사용하면(자), 프로그래머는 통신 프로세스간을 이동하는 엔트리 포인트나 데이터형을 표준 언어인것 같이 취급할 수가 있습니다.
CORBA 는 언어로부터 중립의 시스템으로, 인수의 값이나 반환값은 사용하는 구현 언어로 나타낼 수가 있는 것에 제한됩니다. CORBA 에서는, 객체의 방향은 참조용으로 건네줄 수가 있는 객체에 제한되는지, 또는 전체의 시스템로 미리 정의되고 있습니다. 객체 코드 자체를 머신간에 건네줄 수가 없습니다. 건네주는 형태도 되돌리는 형태도 인터페이스로 선언되고 있을 필요가 있습니다.
RMI 에서는, IDL 와 구현 언어는 동일한 것이며, 서로를 매핑하기 위한 걱정은 없습니다. 언어 레벨의 객체 (코드)를 있는 프로세스로부터 다음의 프로세스에 건네줄 수가 있습니다. 값은, 선언된 형태는 아니고 실제의 형태로 돌려주어집니다. 또는, 인터페이스를 컴파일 해 IIOP 대응의 Stub와 스켈리턴을 생성해, 다른 CORBA 대응 언어로 쓰여져 있는 다른 머신의 객체와 통신할 수가 있습니다.

Java IDL 와 Java RMI-IIOP 는 어떻게 다릅니까.

이것은 기본적인 문제이며, Java 프로그램 언어와 CORBA 를 통합하는 2 개의 방식의 차이를 이해하는 것이 중요합니다.
Java IDL 는, CORBA 인터페이스 정의 언어 (IDL)로 정의된 인터페이스에 근거해 Java 프로그램 언어로 프로그램을 기술하고 싶은 CORBA 프로그래머전용입니다. 이것은 「일반적으로 대로의」CORBA 프로그래밍으로, C++ 나 COBOL 와 같은 다른 언어와 완전히 같은 방법으로 Java 언어를 지원하고 있습니다.
Java RMI-IIOP (Remote Method Invocation over Internet Inter-ORB Protocol)는, IIOP 를 배후의 트랜스폴트로서 사용해, Java 프로그램 언어를 사용해 Java RMI 인터페이스를 프로그래밍 하고 싶은 개발자전용입니다. Java RMI-IIOP 는 다양한 언어로 구현되는 CORBA 객체와의 상호 운용성을 제공합니다만, 모든 원격 인터페이스를 미리 Java RMI 인터페이스로서 정의해 둘 필요가 있습니다. EJB 의 원격 객체 모델은 RMI 베이스이므로, Enterprise JavaBeans (EJB)를 사용하는 프로그래머에게는 특히 중요합니다.
분산형 CORBA 어플리케이션의 작성에는 몇개의 정의 방법이 있습니다. 다음에 일반적인 방법을 나타냅니다.
  • Java IDL. 당분간의 사이 IDL 를 사용해 CORBA 어플리케이션을 개발해 온 경험이 있는 경우는, 그 환경을 그대로 사용할 수가 있습니다. IDL 를 사용해 인터페이스를 작성해, Java 프로그램 언어를 사용해 클라이언트와 서버 어플리케이션을 정의합니다. 이것에 의해, Java 의 「Write Once, Run AnywhereTM」이식성 기능, 생산성의 높은 실행 환경, 및 매우 견뢰한 플랫폼등의 이점을 활용할 수가 있습니다.
  • RMI-JRMP. 어플리케이션이 모두 Java 프로그램 언어로 쓰여져 있는 경우는, Java RMI 테크놀러지를 사용해, 다른 가상 머신 및 다른 물리 머신상의 Java 객체간에서의 통신을 가능하게 할 수 있습니다. IIOP 옵션을 가지지 않는 Java RMI 를 사용하면(자), 코드 이식성의 강함이나, 시큐리티, garbage collection를 활용할 수가 있습니다.
  • Java RMI-IIOP. 신규 어플리케이션의 대부분은 Java 프로그램 언어로 쓰고 있지만, 다른 언어로 기술되고 있는 종래의 어플리케이션도 보수할 필요가 있다고 하는 경우에는, Java RMI 를 IIOP 컴파일러 옵션과 병용 할 수가 있습니다.

Java IDL 나 Java RMI-IIOP 를 사용할 때, 무엇인가 제한 사항은 있습니까.
