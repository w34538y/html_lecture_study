제 6 장  

Color

칼라 이미징은 모든 그래픽스 시스템의 기본 요소의 1 개로, 자주 이미징 모델을 복잡하게 하는 원인이 되어 있습니다. Java 2D™ API 는, 사용하기 쉬운 고품질의 칼라 출력을 지원해, 세련된 색채를 가능하게 합니다.

Java 2D API 의 주된 칼라 관리 클래스에는,ColorSpace,Color, 및 ColorModel 가 있습니다.

  • ColorSpace 는, 주로 3 개의 독립한 수치 (성분)를 사용한 색을 평가하기 위한 시스템을 나타냅니다. ColorSpace 클래스에는, 색공간(color space)와 2 개의 표준 색공간(color space)인 CIEXYZ 및 RGB 와의 사이의 변환을 행하기 위한 메소드가 포함되어 있습니다.
  • Color 는, 특정의 ColorSpace 내에서 성분에 의해 정의되고 있는 고정 칼라입니다. 빨강등의 색으로 Shape 를 draw 하려면 , 목적의 색을 나타내는 Color 객체를 Graphics2D 문맥에 건네줍니다. Color 는 java.awt 패키지로 정의되고 있습니다.
  • ColorModel 는, 픽셀치를 색에 매핑 하는 방법을 기술합니다. ColorModel 은, 일반적으로,Image 또는 BufferedImage 와 관련지을 수 있어 픽셀치를 올바르게 해석하는데 필요한 정보를 제공합니다. ColorModel 는 java.awt.image 패키지로 정의되고 있습니다.

6.2 색의 컨셉

ColorModel 는, 이미지내의 픽셀 데이터를 해석하는데 사용됩니다. 픽셀 데이터의 해석에는, 이미지의 각 밴드의 성분을 특정의 색공간(color space)의 성분에 매핑 하는 작업이 포함됩니다. 또, 팩 된 픽셀 데이터로부터의 픽셀 성분의 추출, 마스크를 사용한 1 개의 밴드로부터의 복수의 성분의 꺼내, 룩업테이블을 개입시킨 픽셀 데이터의 변환도 이 처리에 포함되는 일이 있습니다.

이미지내의 특정의 픽셀의 칼라치를 결정하려면 , 각 픽셀에 칼라 정보가 어떻게 캡슐화되고 있을까를 알 필요가 있습니다. 이미지에 관련지을 수 있었던 ColorModel 는, 픽셀치를 칼라 성분에, 또 칼라 성분으로부터 픽셀치로 변환하는데 필요한 데이터와 메소드를 캡슐화합니다.

Java 2D™ API 는, JDK 1.1 소프트웨어 릴리스로 정의되고 있는 DirectColorModel 와 IndexColorModel 에 가세해 다음의 2 개의 칼라 모델을 제공합니다.

  • ComponentColorModel 는, 임의의 ColorSpace 와 칼라 성분의 배열을 처리해,ColorSpace 에 일치하도록(듯이) 합니다. 이 모델을 사용하면(자), 대부분의 종류의 GraphicsDevices 상에서 대부분의 칼라 모델을 나타낼 수가 있습니다.
  • PackedColorModel 는, 칼라 성분이 int 픽셀의 비트에 직접 파묻힌 픽셀치를 나타내는 모델의 base class입니다. PackedColorModel 는, 칼라 성분과 알파 성분을 채널로부터 어떻게 추출하는지를 기술한 팩 정보를 포함합니다. JDK 1.1 소프트웨어 릴리스의 DirectColorModel 은,PackedColorModel 입니다.

6.2. 0.1 ColorSpace

ColorSpace 객체는, 주로 3 개의 독립한 수치를 사용한 색을 평가하기 위한 시스템을 나타냅니다. 예를 들어, RGB 와 CMYK 는 색공간(color space)입니다. ColorSpace 객체는,Color 객체의 칼라 영역을 특정하거나ColorModel 객체를 개입시켜 Image,BufferedImage, 또는 GraphicsConfiguration 의 칼라 영역을 특정하거나하기 위한 칼라 영역 태그로서 기능합니다. ColorSpace 는, 특정의 칼라 영역의 Color 를 sRGB 와의 사이로, 또 명확하게 정의된 CIEXYZ 칼라 영역과의 사이에 변환하기 위한 메소드를 제공합니다.

모든 ColorSpace 객체는, 그 객체가 나타내는 칼라 영역의 색을 sRGB 에 매핑함과 함께,sRGB 칼라를, 그 객체가 나타내는 칼라 영역으로 변환할 수 없으면 안됩니다. 어느 Color 에도, 명시적으로 또는 디폴트로 설정된 ColorSpace 객체가 있으므로, 어떠한 Color 도 sRGB 로 변환할 수 있습니다. 각 GraphicsConfiguration 는 ColorSpace 객체에 관련지을 수 있고 있어 이 ColorSpace 객체에는, 대응하는 ColorSpace 가 있습니다. 칼라 영역에서 지정된 색은,sRGB 를 개입시켜 중간 칼라 영역으로서 매핑 하는 것으로, 임의의 디바이스로 표시할 수 있습니다.

이 처리에 사용되는 메소드는,toRGB 와 fromRGB 입니다.

  • toRGB 는, 표현된 색공간(color space)의 Color 를 sRGB 의 Color 로 변환한다
  • fromRGB 는,sRGB 의 Color 를 받아, 표현된 색공간(color space)로 변환한다

sRGB 를 개입시킨 매핑은 항상 기능합니다만, 반드시 최선의 해결책이 아닙니다. 그 1 개의 이유는,sRGB 에서는,CIEXYZ 칼라의 전역에 건너 모든 색을 표현할 수 있는 것은 아닌 것입니다. sRGB 와는 다른 색역 (표현 가능한 색의 스펙트럼)의 칼라 영역에서 색이 지정되고 있는 경우, 중간 영역으로서 sRGB 를 사용하면(자) 정보가 없어집니다. 이 문제에 대처하기 위해(때문에),ColorSpace 클래스는, 이제(벌써) 1 개의 칼라 영역인 「변환 영역」 CIEXYZ 와의 사이에 색을 매핑 할 수 있게 되어 있습니다.

toCIEXYZ 메소드와 fromCIEXYZ 메소드는, 표현된 칼라 영역으로부터 변환 영역에 칼라치를 매핑 합니다. 이러한 메소드는, 임의의 2 개의 색공간(color space)간으로, 한 번에 1 개의 Color 를 충분히 높은 정밀도로 변환할 수 있습니다. 다만, Java 2D API 의 구현에서는, 기본이 되는 플랫폼의 칼라 관리 시스템을 이용해 이미지 전체를 대상으로 처리를 실시하는, 퍼포먼스가 뛰어난 변환이 지원될 예정입니다. 「이미징」의 ColorConvertOp 를 참조해 주세요.

그림 6-1 과그림 6-2 은, RGB 칼라 모니터상에서 표시하기 위해서, CMYK 칼라 영역에서 지정된 색을 변환하는 처리를 나타내고 있습니다. 그림 6-1 은,sRGB 를 개입시킨 매핑입니다. 도화 가리키도록(듯이), CMYK 칼라로부터 RGB 칼라에의 변환은, 색역이 일치하지 않기 위해(때문에) 부정확하게 되어 있습니다. 1

6.2. 0.2 ICC_Profile 와 ICC_ColorSpace

ColorSpace 는, 실제로는 abstract 클래스입니다. Java 2D API 는, 그 1 개의 구현인 ICC_ColorSpace 를 제공하고 있습니다. ICC_ColorSpace 는,ICC_Profile 클래스에 의해 나타내지는 것 같은 ICC 프로파일 데이터에 근거하고 있습니다. 벌써 설명한 2 개의 메소드를 구현하면, 독자적인 서브 클래스를 정의해 임의의 칼라 영역을 나타낼 수가 있습니다. 다만, 대부분의 경우는, 디폴트의 sRGB 의 ColorSpace, 혹은, 모니터나 프린터용의 일반적으로 입수 가능한 ICC 프로파일, 또는 이미지 데이터에 파묻힌 프로파일에 의해 표현된 칼라 영역을 사용할 수 있습니다.

「ColorSpace」에서는,ColorSpace 객체가 칼라 영역을 어떻게 표현하고 있는지, 및 표현된 색을 변환 영역과의 사이에 어떻게 매핑 할 수 있을까에 임해서 설명하고 있습니다. 색공간(color space)간의 매핑의 처리에서는, 자주 칼라 관리 시스템이 사용됩니다. 전형적인 칼라 관리 시스템 (CMS)에서는,ColorSpace 객체를 닮은 ICC 프로파일을 관리합니다. ICC 프로파일에서는, 입력 공간과 접속 공간에 도착해 기술되고 있어 이러한 공간간에서의 매핑의 방법이 정의되고 있습니다. 칼라 관리 시스템은, 어느 프로파일의 태그가 붙은 색을 다른 프로파일의 색공간(color space)에 어떻게 매핑 하면 좋은가를 이해하는데 있어서, 많이 도움이 됩니다.

Java 2D API 에서는,ICC_Profile 로 불리는 클래스가 정의되고 있어 이 클래스가 임의의 ICC 프로파일의 데이터를 보관 유지합니다. ICC_ColorSpace 는, abstract ColorSpace 클래스의 구현입니다. ICC_ColorSpace 객체는,ICC_Profiles 로부터 구축할 수 있습니다. 다만, 모든 ICC 프로파일이 ICC_ColorSpace 의 정의에 적절하고 있는 것은 아닌 것 등, 몇개의 제한이 있습니다.

ICC_Profile 에는,ICC_ProfileRGB,ICC_ProfileGray 등, 특정의 색공간(color space) 타입에 대응하는 몇개의 서브 클래스가 있습니다. ICC_Profile 의 각 서브 클래스는, 명확하게 정의된 입력 영역 (RGB 영역등)과 명확하게 정의된 (CIEXYZ 와 같은) 접속 영역을 가지고 있습니다. Java 2D API 에서는, 플랫폼의 CMS 를 사용해, 스캐너, 프린터, 모니터등이 다양한 칼라 프로파일에 액세스 할 수 있습니다. 또, 플랫폼의 CMS 를 사용해, 프로파일간에서의 최적인 매핑을 찾아낼 수도 있습니다.

6.2. 1 색의 기술

Color 클래스는, 특정의 칼라 영역의 색을 기술합니다. Color 의 인스턴스에는, 칼라 성분의 값과 ColorSpace 객체가 포함되어 있습니다. Color 의 새로운 인스턴스를 생성할 때는, 칼라 성분에 가세해 ColorSpace 객체를 지정할 수 있으므로,Color 클래스는 임의의 칼라 영역의 색을 처리할 수 있습니다.

Color 클래스에는, 현재 제안되고 있는 sRGB 로 불리는 표준 RGB 칼라 영역을 지원하는 몇개의 메소드가 있습니다 (http://www.w3.org/pub/WWW/Graphics/Color/sRGB.html 를 참조). sRGB 는, Java 2D API 의 디폴트 칼라 영역입니다. Color 클래스에서 정의되고 있는 몇개의 생성자 에서는,ColorSpace 파라미터가 생략 되고 있습니다. 이러한 생성자 은, 색의 RGB 치가 sRGB 로 정의되고 있는 것과 가정해,ColorSpace 의 디폴트 인스턴스를 사용해 색공간(color space)를 나타냅니다.

Java 2D API 에서는, 칼라 변환을 위한 레퍼런스 칼라 영역으로서가 아니고, 어플리케이션 프로그래머의 편의를 생각해 sRGB 를 사용하고 있습니다. 많은 어플리케이션은 주로 RGB 이미지와 모니터를 대상으로 하고 있으므로, 표준 RGB 칼라 영역을 정의하면, 이러한 어플리케이션을 보다 간단하게 작성할 수 있게 됩니다. ColorSpace 클래스에서는 toRGB 메소드와 fromRGB 메소드가 정의되므로, 개발자는 표준 공간의 색을 용이하게 꺼낼 수가 있습니다. 이러한 메소드는, 고정밀도의 칼라 보정이나 변환으로 사용하는 것을 의도한 것이 아닙니다. 자세한 것은,「ColorSpace」를 참조해 주세요.

sRGB 이외의 칼라 영역에서 색을 작성하려면 Color 의 생성자 을 사용합니다. Color 의 생성자 은,ColorSpace 객체와 이 칼라 영역에 적절한 칼라 성분을 나타내는 float 의 배열을 인수에 취합니다. ColorSpace 객체는 색공간(color space)를 특정합니다.

인쇄용 시안과 같은 어느 특정의 색으로 구형을 표시하려면 , 시스템에 대해서 이 색을 기술할 방법이 필요합니다. 색을 기술하려면 , 다양한 방법이 있습니다. 예를 들어, 적, 록, 청 (RGB)의 성분의 편성에 의해 색을 기술하거나 시안, 마젠타, 황색, 흑 (CMYK)의 성분의 편성에 의해 색을 기술하거나 할 수 있습니다. 이러한 색을 지정하기 위한 다양한 수법은, 「색공간(color space)」라고 불립니다.

잘 알려져 있도록(듯이), 컴퓨터의 화면상의 색은, 적, 록, 청의 빛을 분량을 바꾸어 혼합하는 것으로 생성됩니다. 따라서, RGB 색공간(color space)의 사용은, 컴퓨터 모니터상에서의 이미징에서는 표준입니다. 같이 4 색인쇄법에서는, 시안, 마젠타, 황색, 흑의 잉크를 사용해 인쇄 페이지상에 색을 작성합니다. 이 경우, 인쇄하는 색은, CMYK 칼라 영역의 퍼센티지로서 지정되고 있습니다.

RGB 칼라 영역과 CMYK 칼라 영역은, 컴퓨터 모니터와 칼라 인쇄의 보급에 의해, 어느쪽이나 색을 기술하는데 넓게 사용되고 있습니다. 다만, 어느 쪽의 타입의 칼라 영역에도, 디바이스에 의존한다고 하는 기본적인 결점이 있습니다. 어느 프린터로 사용되는 시안 잉크는, 다른 프린터로 사용되는 시안 잉크와 같지 않는 것이 있습니다. 같이 RGB 칼라로서 기술된 색이, 어느 모니터에서는 파랑으로 보여 다른 모니터에서는 보라색이 사 보이는 일도 있습니다.

6.2. 2 sRGB 및 CIEXYZ 를 개입시킨 색의 매핑

Java 2D API 에서는, RGB 와 CMYK 를 칼라 영역 타입이라고 부릅니다. 특정의 형광관을 가지는 특정 모델의 모니터에서는, 그 모니터 독자적인 RGB 색공간(color space)가 정의되고 있습니다. 같이 특정의 모델의 프린터에는, 그 프린터 독자적인 CMYK 색공간(color space)가 있습니다. 다른 RGB 칼라 영역이나 CMYK 칼라 영역은, 디바이스에 의존하지 않는 칼라 영역을 개입시켜 서로 관련지을 수가 있습니다.

디바이스에 의존하지 않는 칼라 지정의 표준은, International Commission on Illumination (CIE)에 달려 있고 구두나 정의되고 있습니다. 이 중, 가장 잘 사용되고 있는 색공간(color space)는, CIE 에 의해 개발된 3 성분 XYZ 색공간(color space)입니다. CIEXYZ 를 사용해 색을 지정했을 경우는, 디바이스에는 의존하지 않게 됩니다.

다만,CIEXYZ 칼라 영역에서의 색의 기술은, 반드시 실제적이지 않은 것이 있습니다. 다른 칼라 영역에서 색을 나타낸 (분)편이, 적절한 경우도 있습니다. 특정의 RGB 영역 등, 디바이스에 의존하는 칼라 영역을 사용해 색을 나타내는 경우에, 일관성이 있는 결과를 얻을 수 있도록(듯이) 하려면 , 그 RGB 영역이,CIEXYZ 와 같은 디바이스에 의존하지 않는 영역과 어떠한 관계에 있는지를 나타낼 필요가 있습니다.

칼라 영역간에서의 매핑을 실시하는 1 개의 방법으로서 디바이스에 의존하는 영역이 디바이스에 의존하지 않는 영역과 어떠한 관계에 있는지를 나타내는 정보를, 어느 쪽의 영역에도 첨부하는 방법이 있습니다. 이 추가 정보는, 「프로파일」이라고 불립니다. 칼라 프로파일의 타입 가운데, 일반적으로 사용되고 있는 것으로서 International Color Consortium 가 정의한 ICC 칼라 프로파일이 있습니다. 자세한 것은,http://www.color.org 로부터 입수 가능한 「ICC Profile Format Specification, version 3.4」를 참조해 주세요.

그림 6-3 은, 전부 칠한 색과 스캔 한 이미지가 Java 2D API 에 어떻게 건네받아 다양한 출력 디바이스로 어떻게 표시되는지를 나타내고 있습니다. 그림 6-3 으로부터 알도록(듯이), 입력 칼라와 이미지의 양쪽 모두에 프로파일이 첨부되고 있습니다.

그림 6-3 칼라 영역간을 매핑 하는 프로파일의 이용

6.2. 2.1 칼라 매칭

API 는, 정확하게 지정된 색을 취득하면(자), 지정된 색을 모니터나 프린터등의 출력 디바이스상에 재현 하지 않으면 안됩니다. 이러한 디바이스에는 각각 독자적인 이미징 특성이 있으므로, 적절한 결과를 얻으려면 , 이 이미징 특성을 고려하지 않으면 안됩니다. 이 때문에, 각 출력 디바이스에는 프로파일을 관련지어 정확한 결과를 얻으려면 색을 어떠한 방법으로 변환할 필요가 있는지를 기술합니다.

일관성이 있는 정확한 색을 출력하려면 , 표준 칼라 영역을 기준으로서 입력 칼라와 출력 디바이스의 양쪽 모두의 프로파일을 설정할 필요가 있습니다. 예를 들어, 입력 칼라를, 원의 칼라 영역으로부터 디바이스에 의존하지 않는 표준의 칼라 영역에 매핑 해, 다음에 표준의 칼라 영역으로부터 출력 디바이스의 칼라 영역에 매핑 합니다. 색의 변환은, 많은 점으로써, 그래피컬 객체의 (x,  y) 좌표 공간에의 변환을 닮아 있습니다. 어느 쪽의 경우도, 변환을 사용해 「표준」공간에서의 좌표를 지정해, 다음에 이 좌표를 디바이스에 고유의 공간에 매핑 해 출력합니다.

1 이러한 그림으로 사용되고 있는 색은, 알기 쉬움을 우선시키고 있기 (위해)때문에, 정확하지는 않습니다. 즉, 적절한 변환 영역이 사용되지 않는 이상 색은 칼라 영역간에서 정확하게 매핑 되지 않습니다.
