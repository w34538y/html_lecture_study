J2SE 1.3 으로 J2SE 5.0 의 사이에서의 CORBA API 의 변경

이 문서에서는, CORBA 의 기능에 관련해 JDK 1.3 으로 5.0 의 사이에 변경된 모든 API 에 대해 정보를 제공합니다. 이 변경은, JDK 5.0 에 짜넣어지는 CORBA 기술용의 API 를, 다음에 나타내는 OMG 문서에 규정되고 있는 CORBA 2.3 매핑에 준거시키기 위해서(때문에) 행해진 것입니다.
  • CORBA 2.3. 1 (formal/99-10-07 )
  • IDL 로부터 Java 언어에의 매핑 (ptc/00-01-08 )
  • Java 로부터 IDL 언어에의 매핑 (ptc/00-01-06 )
  • 상호 운용 가능한 네임 서비스 (/ptc/00-08-07 )
  • 포타브르인타세프타 (ptc/04. 03.01 )

org.omg.COSNaming 클래스에서의 API 의 변경

CORBA 버젼 2.3 스펙으로 규정되고 있는 IDL 로부터 Java 에의 매핑에 준거하기 위해(때문에), 다음에 나타내는 org.omg.COSNaming 클래스를, JDK 에 동고 되고 있는 idlj 컴파일러로 재생성했습니다. 이 변경 (CCC 4284015)에는, 다음과 같은 API 의 변경이 포함됩니다. 호환성에의 영향:없음

org.omg.CORBA 클래스에서의 API 의 변경

  • OMG 스펙 99-07-57 「IDL to Java mapping」에 준거하기 위해(때문에), 다음에 나타내는 메소드가 변경되었습니다.

    • 메소드 org.omg.CORBA.Any.insert_fixed(java.math.BigDecimal, org.omg.CORBA.TypeCode) 의 시그니챠가, 예외 org.omg.CORBA.Bad_INV_ORDER 를 throw 하도록(듯이) 변경되었습니다. 사용자의 코드로 시스템 예외를 선언할 필요는 없습니다. (CCC 4340915)
    • 메소드 TypeCode.equivalent 가 abstract 로서 정의되고 있습니다. 변경 전:예외 NO_IMPLEMENT 를 throw 하고 있었습니다. 호환성에의 영향:라이센시(licensee)에게는 최소한의 영향이 있습니다만, 최종 사용자에는 영향이 없습니다. 라이센시(licensee)는, 이 메소드의 더미 구현을 제공할 필요가 있습니다. (CCC 4340932)
    • 메소드 TypeCode.get_compact_typecode 가 abstract 로서 정의되고 있습니다. 변경 전:예외 NO_IMPLEMENT 를 throw 하고 있었습니다. 호환성에의 영향:라이센시(licensee)에게는 최소한의 영향이 있습니다만, 최종 사용자에는 영향이 없습니다. 라이센시(licensee)는, 이 메소드의 더미 구현을 제공할 필요가 있습니다. (CCC 4340932)
    • org.omg.CORBA.portable.Delegate.java 의 메소드 get_interface_def(org.omg.CORBA.Object) 의 매핑이 abstract 로서 정의되고 있습니다. 호환성에의 영향:있어. 비 abstract 메소드가 abstract 에 변환됩니다. (CCC 4366071)
    • org.omg.CORBA.portable.OutputStream.java 의 메소드 write_Principal() 의 매핑이 abstract 는 아니게 되었습니다. 호환성에의 영향:없음. abstract 메소드가 비 abstract 에 변환됩니다. (CCC 4366072)
    • org.omg.CORBA.portable.InputStream.java 의 메소드 read_Principal() 의 매핑이 abstract 는 아니게 되었습니다. 호환성에의 영향:없음. abstract 라고 선언되고 있던 메소드가 abstract 가 아니게 되어도, 기존의 바이너리와의 호환성은 영향을 받지 않습니다. (CCC 4366074)

  • CORBA 2.3 스펙에 준거하기 위해(때문에),org.omg.DomainManagerOperations 인터페이스는 org.omg.CORBA.Object 를 확장한 것은 아니게 되었습니다. 호환성에의 영향:없음. DomainManagerOperations 인터페이스를 직접 사용하는 CORBA 프로그램은 없습니다. 프로그램은 DomainManagerOperations 인터페이스를 DomainManager 인터페이스를 개입시켜 사용할 필요가 있어, 그 인터페이스는 벌써 org.omg.CORBA.Object 를 확장한 것이 되고 있습니다. (CCC 4340907)

    • Any 의 메소드 extract_value 및 insert_value 가, abstract 로 변경되었습니다. Any 는 ORB 의 구현자에 의해 완전하게 구현되어 ORB 사용자가 서브 클래스화할 것은 없기 때문에, 이 변경은 사용자에게 영향이 없습니다.

BAD_PARAM RuntimeException 가 from_int 에 의해 명시적으로 throw 되는 것이 없어졌습니다. 호환성에의 영향:할 수 있습니다. 새로운 CompletionStatus.java 클래스를 사용해 기술하는 신규 코드에서는,BAD_PARAM 실행시 예외를 명시적으로 캐치 할 필요가 없습니다. 그 코드를 이전의 버젼의 JDK 로 재컴파일 하면(자), 컴파일시에 에러가 일어납니다.  이러한 변경은, JCK 리그렛션 (4375061)을 위해서(때문에) 바탕으로 되돌려졌습니다.  대신에, OMG 문제가 등록되었습니다 (http://www.omg.org/issues/java-rtf.html#Issue3946).
    • INV_POLICY.java 가 final 로서 선언되게 되었습니다. 호환성에의 영향은 없습니다.

  • 포타브르인타세프타가 추가되었기 때문에, 기존의 CORBA API 에 새로운 메소드를 추가하는 것으로, IDL 의 typedef 로부터 4 개의 새로운 클래스를 생성하는 것이 필요하게 되었습니다. 호환성에의 영향:없음. 이러한 변경은 모두 추가입니다. (CCC 4379826)

    • register_initial_reference 가 public class org.omg.CORBA.ORB 에 추가되었습니다.
    • 새로운 클래스 org.omg.CORBA.StringSeqHelper 가 2 개의 IDL typedef 에 추가되었습니다.
    • 새로운 클래스 org.omg.CORBA.StringSeqHolder 가 2 개의 IDL typedef 에 추가되었습니다.
    • 새로운 클래스 org.omg.CORBA.WStringSeqHelper 가 2 개의 IDL typedef 에 추가되었습니다.
    • 새로운 클래스 org.omg.CORBA.WStringSeqHolder 가 2 개의 IDL typedef 에 추가되었습니다.

javax.rmi.CORBA 클래스에서의 API 의 변경

OMG 의 API 스펙에 준거하기 위해(때문에),javax.rmi.CORBA 에 다음과 같은 변경을 더했습니다.

  • javax.rmi.CORBA.Stub 의 protected 생성자 을 삭제했습니다. 호환성에의 영향:없음(CCC 4366722)
  • OMG 스펙 00-01-06. pdf 에 준거하기 위해(때문에),jaxax.rmi.CORBA.ClassDesc 생성자 을 삭제했습니다. (CCC 4366734)

API 이외의 변경

JDK 5.0 에는, 다음과 같은 API 이외의 변경도 구현되고 있습니다.
  • 리포지터리(repository) ID 의 계산이 수정되었습니다. 호환성에의 영향:할 수 있습니다. ORB 의 버젼 관리 지원가 추가되었기 때문에, JDK 1.3 ORB 와의 상호 운용이 가능해, 하위 호환성이 유지되고 있습니다. 또, 다른 벤더의 ORB 와의 상호 운용도 가능합니다. (CCC 4365188)

  • 최근 승인된 OMG 문제 3015 에 준거하기 위해(때문에), 인터페이스 리포지터리(repository) 관련의 바젼프라그마를 모두,org.omg.CORBA.ir.idl 의 밖에 꺼냈습니다. 게다가org.omg.CORBA 에 짜넣은 소수의 IR 인터페이스의 리포지터리(repository) ID 치도 변경했습니다. 예를 들어,IDLTypeHelper 에서는,

    private static String _id = "IDL:omg.org/CORBA/IDLType:2. 3";
    (을)를, 다음과 같이 변경했습니다.
    private static String _id = "IDL:omg.org/CORBA/IDLType:1. 0";

    호환성에의 영향:할 수 있습니다. 이 변경은, 기존의 구현과의 상호 운용성에 영향을 주는 일이 있습니다. COMM_FAILURE 예외 (또는 유사한 예외)를 받았을 경우는, 어플리케이션 및 IR 를 재기동해 주세요. (CCC 4372718)

  • 모든 org.omg. * 패키지 파일에 @serial exclude 태그가 추가되었습니다. 그 때문에, JAVADOC 는, 직렬화의 스펙에 대해 패키지 전체를 무시하게 됩니다. Object Management Group (OMG)로부터는, 다른 버젼간에서의 직렬화의 호환성에 대해서는 아무것도 요구가 나와 있지 않습니다. 실용상은,org.omg 클래스를 1 개의 버젼으로 직렬화한 후, 다른 버젼으로 직렬화 해제한다고 하는 처리는 행해지는 일이 없습니다. 호환성에의 영향:없음(CCC 4394004)

  • POA 의 지원를 위해,idlj 컴파일러의 인수가 변경되었습니다. POA 는, J2SE 1.4 로 도입된 CORBA 의 신기능입니다. idlj 툴의 인수에 대한 변경에는, 다음과 같은 것이 있습니다.

    • 문서화되어 있지 않다 -poa 플래그가 삭제되었습니다.
    • 표준의 POA 베이스 스켈리턴의 생성이 디폴트로 유효하게 되게 되었습니다.
    • 새로운 플래그로서 -oldImplBase 가 추가되었습니다. 이것에 의해, 하위 호환성이 있는 스켈리턴 (JDK 1.2 및 1.3 으로 호환성이 있다)을 생성할 수 있습니다.

    호환성에의 영향:있어. 이 컴파일러를 사용하는 기존의 프로그램에 대해서 idlj 를 호출할 때 호환성을 유지하려면 , -oldImplBase 플래그를 추가할 필요가 있습니다. (CCC 4407835)

신규 API

JDK 5.0 에는, 다음과 같은 새로운 API 가 추가되었습니다.
  • PortableInterceptor API 가 추가되었습니다. 테스트 플랜에서는, 잘못한 스펙 ptc/2000-04-05 가 참조되고 있었습니다. 기능 리스트에서는, 다른 잘못한 스펙 orbos/99-12-02 가 참조되고 있었습니다 (ptc/2000-04-05 는, 형식이 변경된 것 이외는 orbos/99-12-02 와 같다). 이 신규 API 의 올바른 스펙 참조는, http://www.omg.org/cgi-gin/doc? ptc/2001-03-04 입니다. CCC 4307834 에, ptc/2000-04-05 에 대한 변경과 추가의 리스트가 포함되어 있습니다. 호환성에의 영향:없음. 이것은 신기능이므로, 이 API 를 사용하고 있는 기존의 코드는 없습니다.

  • 포타브르인타세프타가 추가되었기 때문에, public 인터페이스 org.omg.PortableServer.POA 에 새로운 메소드 id 가 필요하게 되었습니다.

J2SE 5.0 에서의 CORBA 지원의 공식 스펙

스펙의 참조

Java 2 Platform Standard Edition (J2SE) 5.0 ORB 의 컴파일시에 적용되는 스펙은 다음과 같습니다. 아래와 같은 링크로부터 올바른 스펙을 참조할 수 없는 경우는, OMG 가 URL 를 변경한 가능성이 있습니다. www.omg.org 를 참조해, 올바른 스펙을 검색해 주세요.
  • CORBA 2.3. 1 (formal/99-10-07 )
  • IDL 로부터 Java 언어에의 매핑 (ptc/00-01-08 )
  • 개정된 IDL 로부터 Java 언어에의 매핑 (ptc/00-11-03 )
  • Java 로부터 IDL 언어에의 매핑 (ptc/00-01-06 )
  • Interoperable Naming Service (ptc/00-08-07 )
  • 포타브르인타세프타 (ptc/2001-08-31 )
상기에 기재한 것은, 이 문서로 참조되고 있는 스펙만입니다.

지원되고 있는 스펙의 상세

J2SE 5.0 의 ORB 에서는, 현재로서는, 상기의 스펙의 모든 것이 구현되고 있는 것은 아닙니다. 지원되고 있는 섹션은 다음과 같습니다.
  • CORBA 2.3. 1 의 제 1? 3 장 및 제 6? 9 장
  • 개정된 IDL 로부터 Java 언어에의 매핑의 섹션 1.21. 8.2 「orb.properties file」
  • CORBA 2.3. 1 의 제 4 장. 다만, 일부의 내용이, 포타브르인타세프타 스펙의 이하의 섹션으로 옮겨놓을 수 있고 있습니다.
    • 섹션 4.2. 3.5 「destroy」
    • 섹션 4.5. 2 「CodeFactory」 및 「PICurrent」가 필요
    • 섹션 4.5. 3.4. register_initial_reference 에 대한 갱신
  • CORBA 2.3. 1 의 제 5 장. 다만, 포타브르인타세프타 스펙의 이하의 섹션의 갱신이 우선됩니다.
    • 5.5. 2 StringSeq 및 WStringSeq 가 필요합니다. 이것에 의해, 다음의 클래스가 org.omg.CORBA 에 추가됩니다.
      • StringSeqHolder
      • StringSeqHelper
      • WStringSeqHolder
      • WStringSeqHelper
  • CORBA 2.3. 1 의 섹션 10.6. 1 및 10.6. 2 (리포지터리(repository) ID 에 대해서 지원된다)
  • CORBA 2.3. 1 의 섹션 10.7 (TypeCode API 에 대해서)
  • CORBA 2.3. 1 의 제 11 장 「Portable Object Adapter (POA)」. 다만, 포타브르인타세프타 스펙의 이하의 섹션의 갱신이 우선됩니다.
    • 섹션 11.3. 7. POA 는 모든 정책를 보관 유지해야 한다
    • 섹션 11.3. 8.2. POA 는 모든 정책를 보관 유지해야 한다
    • 섹션 11.3. 8.26. POA::id 가 필요
  • CORBA 2.3. 1 의 제 13 장 및 제 15 장 (GIOP 1.0, 1.1, 및 1.2 의 정의). J2SE 5.0 의 ORB 에서는, 모든 버젼의 GIOP 가 완전하게 지원됩니다만, 섹션 15.8 및 15.9 로 정의되고 있는 쌍방향 GIOP 기능은 예외적으로 지원되지 않습니다.
  • Interoperable Naming Service 의 스펙은 모두 지원되고 있습니다.
  • 포타브르인타세프타 스펙의 섹션 13.8 (새로운 Coder 및 Decoder 인터페이스)과 제 21 장 전체 (인타세프타의 스펙)
  • 개정된 IDL 로부터 Java 언어에의 매핑 스펙 (ptc/00-11-03)의 섹션 1.21. 8 은, IDL 로부터 Java 언어에의 매핑 스펙 (ptc/00-01-08)의 내용으로부터 변경되었습니다.

툴

  • IDL 로부터 Java 에의 컴파일러 (idlj)는, 다음의 스펙에 준거하고 있습니다.
    • CORBA 2.3. 1 의 제 3 장 (IDL 의 정의)
    • CORBA 2.3. 1 의 제 5 장 및 제 6 장 (Value 형의 시멘틱스)
    • CORBA 2.3. 1 의 섹션 10.6. 5 (프라그마)
    • IDL 로부터 Java 에의 매핑 스펙
    • 개정된 IDL 로부터 Java 에의 매핑 스펙의 섹션 1.12. 1 「local interfaces」
  • Java 로부터 IDL 에의 컴파일러 (rmic 의 IIOP 연구 최종 단계)는, 다음의 스펙에 준거하고 있습니다.
    • CORBA 2.3. 1 의 제 5 장 및 제 6 장 (Value 형)
    • Java 로부터 IDL 언어에의 매핑 스펙. IDL 로부터 Java 언어에의 매핑 스펙의 섹션 1.21 을 암묵적으로 참조하고 있습니다.
    • -idl 플래그를 지정해 생성되는 IDL 는, CORBA 2.3. 1 의 제 3 장에 준거하고 있습니다.


그 외의 호환성 정보

  • java.util.Calendar 클래스는, RMI-IIOP 를 사용하면(자), J2SE 1.3.x 로 J2SE 5.0 의 사이에 상호 운용성이 없습니다. 이 점은 OMG 문제 3151 으로 해결될 예정입니다만, 그것이 구현되는 것은 Java 플랫폼의 장래의 릴리스가 될 전망입니다.

    J2SE 5.0 버젼의 Calendar 의 writeObject 메소드에서는,ZoneInfo 객체를 써냅니다. 같이 그 객체를 읽어들이려면 readObject 메소드를 호출합니다. 낡은 버젼이라고 대화하고 있는 경우는, 객체가 존재하지 않기 때문에, 스트림은 Java 객체 직렬화 스펙에 근거해 EOFException 를 throw 합니다. 스트림의 위치는 바뀌지 않습니다.
