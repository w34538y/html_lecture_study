Java Plug-in 에서의 OBJECT,EMBED, 및 APPLET 태그의 사용

 

이 장에서는, 다음의 토픽에 대해 설명합니다.

  • 처음에
  • Windows 상의 IE 로 동작하는 JavaTM Plug-in
  • Windows 또는 SolarisTM 오퍼레이팅(operating) 환경의 NavigatorTM 로 동작하는 Java Plug-in
  • IE 및 Navigator 로 동작하는 Java Plug-in
  • 그 외의 환경에서의 Java Plug-in
  • 통계

처음에

이 문서에서는, Java Plug-in 에 필요한,OBJECT 및 EMBED 태그를 포함한 태그 구조에 대해 설명합니다. 대상 독자로서 HTML 페이지에 Java Plug-in 태그를 수동으로 삽입하는 Web 페이지 작성자를 상정하고 있습니다.

 

설명

  1. Sun Microsystems 로부터 무료로 입수 가능한 HTML 컨버터에서는, 태그 구조를 자동적으로 생성할 수 있습니다. Web 페이지 작성자에게는, 이 툴을 이용하는 것을 강하게 추천합니다.
  2. 이 컨버터를 사용하면(자), 페이지를 아래와 같은 어느 형식에도 변환할 수 있습니다.
  3. 후술과 같이,. cab 파일을 개입시켜 자동적으로 다운로드할 수 있는 JRE 릴리스의 일람에 대해서는,「자동 다운로드 파일 (Windows 마셔)」를 참조해 주세요.

일반적으로, 애플릿은, HTML 파일내에서 다음과 같이 지정됩니다. 

<APPLET code="XYZApp.class" codebase="html/" align="baseline"
    width="200" height="200">
<PARAM name="model" value="models/HyaluronicAcid.xyz">
    No Java 2 SDK, Standard Edition v 1.4. 2 support for APPLET!!
</APPLET>

또, 일반적으로,APPLET 태그에는, 애플릿에 관한 정보를 지정합니다. <APPLET> 와 </APPLET> 태그의 사이에 배치되는 <PARAM> 태그에는, 인스턴스마다의 애플릿 정보를 포함합니다.

APPLET 는 브라우저에 의해 draw 됩니다만, 브라우저에 인터럽트를 실시해, Sun 의 Java Runtime Environment (JRE)를 사용한 애플릿의 실행을 강제하는, 간단한 방법은 없습니다. 브라우저에 이 동작을 실행시키려면 , HTML 페이지로, 일반적으로의 APPLET 태그 대신에,OBJECT 와 EMBED 태그의 언젠가 또는 양쪽 모두를 포함한, Java Plug-in 의 특별한 태그 구조를 사용합니다. 이것에 의해, 브라우저가 Java Plug-in 를 기동시켜, Sun 의 JRE 를 사용해 애플릿이 실행됩니다.

다음의 섹션에서는, 특정의 브라우저와 플랫폼을 조합한 환경에서 실시하는 조작을 세세하게 설명합니다.

버젼 번호에 관한 주기

제품의 버젼 번호는, 다음의 형식이 됩니다.

n1.n2.n3_n4n5

여기서,n1.n2 는 「메이저 버젼 번호」,n3 는 「마이너 버젼 번호」( 「보수 버젼 번호」라고도 불린다),n4n5 는 「패치 버젼 번호」( 「갱신 버젼 번호」라고도 불린다)을 나타냅니다.

이후의 예에서는, 버젼 번호의 메이저 릴리스로서 1.4 를, 마이너 릴리스 번호로서 2 를,가상의 패치 번호로서 mn 를 사용합니다. 가상의 값을 실제의 Web 페이지의 HTML 에 기입하지 말아 주세요.

Windows 상의 IE 로 동작하는 Java Plug-in

Windows 상의 IE 로 Java Plug-in 를 사용하는 경우,OBJECT 태그를 이용합니다. APPLET 태그를 Java Plug-in 태그에 매핑 하는 예를, 다음에 나타냅니다.

원의 APPLET 태그:

<APPLET code="XYZApp.class" codebase="html/" align="baseline"
    width="200" height="200">
<PARAM name="model" value="models/HyaluronicAcid.xyz">
    No Java 2 SDK, Standard Edition v 1.4. 2 support for APPLET!!
</APPLET>

신규 OBJECT 태그: 

<OBJECT classid="clsid:CAFEEFAC-0014-0002-0000-ABCDEFFEDCBA"
    width="200" height="200" align="baseline" 
    codebase="http://java.sun.com/products/plugin/autodl/jinstall-1_4_2-windows-i586.cab#Version=1, 4,2,0">
    <PARAM name="code" value="XYZApp.class">
    <PARAM name="codebase" value="html/">
    <PARAM name="type" value="application/x-java-applet;jpi-version=1. 4.2">
    <PARAM name="model" value="models/HyaluronicAcid.xyz">
    <PARAM name="scriptable" value="true">
        No Java 2 SDK, Standard Edition v 1.4. 2 support for APPLET!!
</OBJECT>

주:

정적 버젼 관리의 경우는,codebase 를 다음과 같이 설정합니다.

http://java.sun.com/products/plugin/autodl/<.cab file for the version you want>

예를 들어, 1.4. 2 릴리스의 정적 버젼 관리의 경우는 (clsid:CAFEEFAC-0014-0002-0000-ABCDEFFEDCBA 라고 하는 classid 가 사용된다),codebase 를 다음과 같이 설정합니다.

http://java.sun.com/products/plugin/autodl/jinstall-1_4_2-win.cab

. cab 파일을 개입시켜 자동적으로 다운로드할 수 있는 JRE 릴리스의 일람에 대해서는,「자동 다운로드 파일 (Windows 마셔)」를 참조해 주세요.

동적 버젼 관리의 경우는 (clsid:8AD9C840-044E-11D1-B3E9-00805F499D93 라고 하는 classid 가 사용된다), codebase 를 다음과 같이 설정합니다.

http://java.sun.com/products/plugin/autodl/jinstall-<major version of family in underscore format>-windows-i586.cab#Version=<full version number for lowest version in family in comma-separated format>

어느 버젼의 Java 도 인스톨되어 있지 않은 경우나, 패밀리의 메이저 버젼 번호보다 작은 버젼이 인스톨 되고 있는 경우는, 패밀리의 최신 버젼에 대응하는 최신의 . cab 에 자동적으로 리다이렉트(redirect) 됩니다.

예를 들어, 1.4. 2 릴리스의 경우로,codebase 를 다음과 같이 설정했다고 합니다.

http://java.sun.com/products/plugin/autodl/jinstall-1_4-windows-i586.cab#Version=1, 4,0,0

사용자의 환경에 버젼 1.3. 1 이 인스톨 되고 있는 경우는, 1.4 패밀리의 최신 버젼이 다운로드되어 인스톨 됩니다. 다만, 1.4 패밀리의 몇개의 버젼이 인스톨 되고 있는 경우는 그것이 사용되므로, 다운로드는 행해지지 않습니다.

자동 다운로드용의 . cab 파일은, 패치의 릴리스에 수반해 변경될 가능성이 있습니다. codebase 의 해석에 관한 자세한 것은, 이하를 참조해 주세요.

OBJECT 태그에는,APPLET 태그와 같은 정보가 포함되는 것에 유의해 주세요. IE 로 Java Plug-in 를 기동하려면 , 이것으로 충분합니다. OBJECT 태그의 classid 는, Java Plug-in 자체의 클래스 식별자입니다. IE 가 OBJECT 태그내에서 이 클래스 식별자를 draw 할 때, 브라우저내에의 Java Plug-in 의 로드를 시도합니다. 버젼 관리에 관한 정보는, 후술의「classid 속성」을 참조해 주세요.

width,height, 및 align 등,OBJECT 태그내의 몇개의 속성은,APPLET 태그내의 대응하는 속성으로부터 직접 매핑 됩니다. IE 는, 이것들에 포함되는 포맷 정보를 이용해, Java Plug-in 의 위치를 설정합니다. 이 정보는 변경되는 일 없이 직접 매핑 되기 (위해)때문에, Java Plug-in 를 사용하는 애플릿의 위치 및 외관은,APPLET 태그를 사용하는 애플릿과 같게 됩니다.

APPLET 태그의 모든 속성을,OBJECT 태그 속성에 매핑 할 수 있는 것은 아닙니다. 예를 들어, APPLET 태그의 code 및 codebase 속성은,OBJECT 태그 속성에 매핑 되지 않습니다. 그 대신해,code 속성은 PARAM code 에 매핑 됩니다. 이것은, w3c.org 의 HTML 스펙에서는,OBJECT 태그내에 code 속성은 존재하지 않기 때문입니다. OBJECT 태그 속성에 대응하지 않는 다른 속성도 존재합니다. 이러한 속성은, 1 개의 예외를 제외해 PARAM 태그에 매핑 할 필요가 있습니다.

주:

Java Plug-in 의 OBJECT 태그에서는, 중복 하는 파라미터명의 사용은 금지되고 있습니다.

다만,codebase 속성은 예외입니다. APPLET 태그에서는,codebase 속성은, 추가의 클래스 파일 및 JAR 파일의 다운로드원래의 위치를 나타냅니다. 한편,OBJECT 태그에서는,codebase 속성은, 로컬 머신에 Java Plug-in 가 발견되지 않는 경우의 Java Plug-in 의 다운로드원래의 위치를 나타냅니다. codebase 속성은 APPLET 태그와 OBJECT 태그로 다른 의미를 가지고 있습니다. 따라서, 경합을 회피하기 위해서,APPLET 태그내의 codebase 속성은,OBJECT 태그내의 PARAM codebase 에 매핑 됩니다.

위의 예에서는, APPLET 태그의 code 및 codebase 속성은,OBJECT 태그의 파라미터에 매핑 됩니다. PARAM code 는 애플릿을 식별하기 위해(때문에), 그 값을 APPLET 태그의 code 속성과 같게 할 필요가 있습니다. PARAM codebase 는, 애플릿의 codebase 를 식별합니다. Java Plug-in 는, 파라미터로부터 이 정보를 읽어낼 수가 있기 (위해)때문에, 애플릿 또는 JavaBeans 컴퍼넌트의 다운로드처를 인식합니다. 파라미터 type 는,APPLET 태그로부터 매핑 될 것은 없습니다만,OBJECT 태그에서는 필요하게 됩니다. 이것은, Java 실행 가능 파일의 타입 (애플릿 또는 JavaBean 등)을 식별합니다. 이 때문에, Java Plug-in 는 Java 실행 가능 파일의 초기화 방법을 인식할 수 있습니다. 위의 예로, 이것들 3 개의 PARAM 태그 (code,codebase, 및 type)는, Java Plug-in 용으로 지정되고 있습니다. 이러한 태그는, 원의 APPLET 태그의 PARAM 에는 존재하지 않습니다. OBJECT 태그내의 model 파라미터는,APPLET 태그내의 model 파라미터와 동일한 것으로 유의해 주세요. Java Plug-in 로 지정하는 이것들 최초의 3 개의 파라미터 이외의 파라미터는,APPLET 태그내의 파라미터와 같습니다.

Java Plug-in 1.3 에서는,PARAM scriptable 가 새롭게 추가되었습니다 (1.4.x 에서도 유효). 이것이 추가된 이유는, JavaScript 나 VBScript 를 필요로 하지 않는 애플릿의 퍼포먼스 향상입니다. 애플릿이 스크립팅 지원를 필요로 하는 경우는 true 에, 필요로 하지 않는 경우는 false 에 값을 설정할 필요가 있습니다. 디폴트 값는 false 입니다.

PARAM scriptable 는 PARAM mayscript 와 같지 않는 것에 유의해 주세요. mayscript 는 Java 애플릿으로부터 JavaScript 에의 통신만을 지원하는데 대해,scriptable 는 Internet Explorer 에서의 JavaScript 로부터 Java 애플릿에의 통신만을 지원합니다.

APPLET 태그내의 텍스트 「No Java 2 SDK, Standard Edition v 1.4. 2 support for APPLET!!」(은)는,<OBJECT> 및 </OBJECT> 태그내에서 매핑 됩니다. 본래, 이 텍스트는, 브라우저가 Java 를 지원하지 않는 경우에게만 표시됩니다. 이 텍스트를 OBJECT 태그의 내부에 매핑 하면(자), 브라우저가 OBJECT 태그를 지원하지 않는 경우, 이 텍스트가 표시됩니다.

동적 버젼의 지원의 경우는, 같은 값 (clsid:8AD9C840-044E-11D1-B3E9-00805F499D93 등)을 보관 유지할 필요가 있습니다. 정적 버젼의 지원의 경우는, 버젼 고유의 값 (clsid:CAFEEFAC-0014-0002-0000-ABCDEFFEDCBA 등)이 보관 유지됩니다.

주: 상기의 예에서는, 정적 버젼 관리가 사용되고 있습니다.

codebase 속성

네트워크상의 . cab 파일을 지시하는 완전한 URL 를 지정할 필요가 있습니다. 이 속성의 목적은, 사용자의 컴퓨터에 Plug-in 가 전혀 인스톨되어 있지 않은 경우나, 인스톨 되고 있는 Plug-in 의 버젼이 적절하지 않은 경우에, Plug-in 의 다운로드와 인스톨을 가능하게 하는 것입니다. 디폴트에서는, URL 는 Java Software Web 사이트상의 페이지를 지시합니다.

례:

상기의 예로,codebase 는 다음과 같습니다.

codebase="http://java.sun.com/products/plugin/autodl/jinstall-1_4_2-windows-i586.cab#Version=1, 4,2,0"

다만, 이것은 정적 버젼 관리의 경우의 예입니다. 또,. cab 파일의 후에 부가된 버젼 번호는, 사용자의 컴퓨터에 버젼 1.4. 2 의 Plug-in 가 인스톨 되고 있을 필요가 있는 것을 나타내고 있습니다. 이 조건이 채워지지 않은 경우는, 다운로드와 인스톨이 시도됩니다.

동적 버젼 관리의 경우는,codebase 는 다음과 같이 됩니다.

codebase="http://java.sun.com/products/plugin/autodl/jinstall-1_4-windows-i586.cab#Version=1, 4,0,0"

말미에 부가된 버젼 번호는, 사용자의 컴퓨터에 버젼 1.4. 0 이후의 Plug-in 가 인스톨 되고 있는 경우, 다운로드는 행해지지 않는 것을 나타내고 있습니다. 즉, 1.4 패밀리 (메이저 버젼 1.4)의 Plug-in 가 인스톨 되고 있으면, 그것이 실행됩니다. 이것보다 전의 버젼이 인스톨 되고 있는 경우는,. cab 파일을 개입시켜 새로운 버젼의 다운로드와 인스톨이 시도됩니다.

PARAM 요소의 type

type 파라미터의 value 는 다음과 같이 됩니다.

"application/x-java-<object_type>;<version_type>=<implementation_version>"

이 예에서는,

object_type 는 applet 또는 bean 입니다.

version_type 는, 정적 버젼 관리의 경우는 jpi-version, 동적 버젼 관리의 경우는 version 입니다.

implementation_version 는 Implementation-Version 입니다. 이것은,「부록 6:Sun 의 지원하는 Specification-Version 및 Implemenation-Version 의 형식」에서 정의되고 있습니다.

정적 버젼 관리와 동적 버젼 관리의 대비에 대해서는,「Encountering OBJECT, EMBED, and APPLET Tags With Different Plug-in Versions and Browsers」를 참조해 주세요.

례:

Plug-in 의 버젼이 1.4. 1, 패치 릴리스가 02 의 정적 버젼 관리를 위해서(때문에) 배치되는 애플릿의 경우는,type 의 value 는 다음과 같이 됩니다.

"application/x-java-applet;jpi-version=1. 4.1_02"

Plug-in 의 버젼이 1.4. 1, 패치 릴리스가 03 의 동적 버젼 관리를 위해서(때문에) 배치되는 Bean 의 경우는,type 의 value 는 다음과 같이 됩니다.

"application/x-java-bean;version=1. 4"

이것은, 동적 버젼 관리의 경우는 메이저 버젼 번호만이 체크되기 (위해)때문에입니다. 즉, 그 메이저 버젼의 패밀리에 속하는 Plug-in 가 인스톨 되고 있는 경우, 다운로드는 불필요합니다.

PARAM 요소의 codebase 애플릿의 베이스 URL 를 지정합니다. 상대 URL 또는 절대 URL 를 지정할 수 있습니다만, 현재의 문서와 같은 도메인내에 있는 것이 필요합니다. 이 PARAM 요소가 필요하게 되는 것은, 문서와는 다른 디렉토리에 애플릿이 놓여져 있는 경우만입니다.
PARAM 요소의 code Java 애플릿 또는 JavaBeans 컴퍼넌트의 이름을 지정합니다. 같은 OBJECT 태그내에, 상자로 된 PARAM 요소의 object 가 있는 경우는, 병용 할 수 없습니다.
PARAM 요소의 scriptable JavaScript 또는 VBScript 를 사용해, HTML 페이지로부터 애플릿을 스크립트 기술 가능한가 어떤가를 지정합니다. 값에는,true 또는 false 를 지정할 수 있습니다. 이것은, Java Plug-in 1.4 로 새롭게 추가된 속성입니다.
PARAM 요소의 object 직렬화 된 Java 애플릿 또는 JavaBeans 컴퍼넌트의 이름을 지정합니다. 같은 OBJECT 태그내에, 상자로 된 PARAM 요소의 code 가 있는 경우는, 병용 할 수 없습니다. 이 속성은 생략 가능합니다.
PARAM 요소의 archive Java Archive 의 이름을 지정합니다. 이 속성은 생략 가능합니다.
PARAM 요소의 mayscript netscape.javascript.JSObject 에의 액세스를 애플릿에 허가할지 어떨지를 지정합니다. 값에는,true 또는 false 를 지정할 수 있습니다. 이 속성은 생략 가능합니다.

원의 APPLET 요소에 type,codebase,code,object, 또는 archive 속성이 사용되고 있는 경우로, 상자로 된 PARAM 요소에도 type,codebase,code,object, 또는 archive 속성이 사용되고 있을 때는,APPELET 로부터 PARAM 에의 직접적인 속성 매핑에 문제가 발생합니다. 이것은, 복수의 PARAM 요소로 name 속성이 중복 하기 (위해)때문에입니다.

다음에 예를 나타냅니다.

<APPLET codebase = "a/b/c ...>
<PARAM name="codebase" value="my.jar">
</APPLET>

이 경우, 직접 매핑은 다음과 같이 됩니다.

<OBJECT ...>
<PARAM name="codebase" value="a/b/c">
<PARAM name="codebase" value="my.jar">
</OBJECT>

이 문제를 회피하려면 , 원의 애플릿의 APPLET 에, 그 속성과 같은 PARAM 요소가 포함되어 있는 경우는, 다음과 같이 매핑 할 필요가 있습니다.

APPLET 요소내의 원의 속성 OBJECT 요소의 PARAM 의 새로운 속성
code java_code
codebase java_codebase
archive java_archive
object java_object
type java_type

이러한 새로운 PARAM 요소의 속성명은, 필요한 경우만 사용하도록 해 주세요. 1 개의 OBJECT 태그에 PARAM 의 새로운 속성명과 원의 속성명의 양쪽 모두가 존재하는 경우, Java Plug-in 는 새로운 PARAM 요소의 속성명에 관련지을 수 있었던 값을 사용해, 애플릿 또는 JavaBean 를 로드합니다.

례

원의 애플릿이 다음과 같은 것이라고 가정합니다.

<APPLET codebase = "a/b/c ...>
<PARAM name="codebase" value="my.jar">
</APPLET>

이 경우, 매핑은 다음과 같이 됩니다.

<OBJECT ...>
<PARAM name="java_codebase" value="a/b/c">
<PARAM name="codebase" value="my.jar">
</OBJECT>

EMBED 태그에는 APPLET 태그와 같은 정보가 포함됩니다. Navigator 로 Java Plug-in 를 기동하려면 , 이 정보로 충분합니다. EMBED 태그의 type 속성은, Java 실행 가능 파일의 타입 (애플릿이나 Bean 등)의 식별로 사용합니다. Navigator 가 EMBED 태그내에서 속성을 draw 할 때, 브라우저내에의 Java Plug-in 의 로드를 시도합니다. 버젼 번호도 부가됩니다. 사용자의 컴퓨터에 인스톨 되고 있는 버젼과 이 번호가 비교되어 다른 버젼의 인스톨이 필요한가 어떤가가 판단됩니다. 이 상세한 것에 대하여는, 아래와 같은 2 번째의 겉(표)로 설명합니다.

위의 예에서는,EMBED 태그내의 width,height, 및 align 등의 속성은,APPLET 태그내의 대응하는 속성으로부터 직접 매핑 됩니다. Navigator 는, 이것들에 포함되는 포맷 정보를 이용해, Java Plug-in 의 위치를 설정합니다. 이 정보는 변경되는 일 없이 직접 매핑 되기 (위해)때문에, Java Plug-in 를 사용하는 애플릿의 위치 및 외관은,APPLET 태그를 사용하는 애플릿과 같게 됩니다.

OBJECT 태그와는 달라, 모든 정보를 <EMBED> 태그의 내부에 포함할 필요가 있습니다. EMBED 태그에서는 PARAM 요소는 사용되지 않습니다. 따라서,APPLET 의 속성 및 관련하는 PARAM 요소는 모두,EMBED 태그의 내부에 속성으로서 name="value" 페어의 형식에서 매핑 할 필요가 있습니다.

위의 예에서는, APPLET 태그의 code 및 codebase 속성은,EMBED 태그 속성에 매핑 됩니다. code 속성은, 애플릿을 식별합니다. 이 속성의 값은,APPLET 태그의 code 속성과 같게 할 필요가 있습니다. codebase 속성은, 애플릿의 코드 베이스를 식별합니다. Java Plug-in 는, 이 정보를 속성으로부터 읽어낼 수가 있기 (위해)때문에, 애플릿 또는 JavaBeans 컴퍼넌트의 다운로드처를 인식합니다. EMBED 태그내의 model 속성은,APPLET 태그내에 상자로 된 PARAM 요소의 model 로부터 매핑 됩니다.

OBJECT 태그의 codebase 속성과 같이, 사용자의 컴퓨터에 Java Plug-in 가 인스톨되어 있지 않은 경우나, 인스톨 되고 있는 Java Plug-in 의 버젼이 적절하지 않은 경우에는,EMBED 태그의 pluginspage 속성이 Navigator 에 의해 사용됩니다. 여기에서는 Java Software Web 사이트상의 Java Plug-in 다운로드 페이지를 가리킵니다. 혹은, 인트라넷에서의 배치의 경우는, 인트라넷상의 Java Plug-in 의 다운로드 페이지를 가리키도록(듯이) 합니다.

IE 및 Navigator 로 동작하는 Java Plug-in

애플릿 가지는 Web 페이지에서는, 전술과 같이 새로운 OBJECT 요소의 방식에 의해, Java Plug-in 를 Internet Explorer 로 사용할 수 있게 됩니다. 또, 새로운 EMBED 요소의 방식에 의해, Java Plug-in 를 Netscape 로 사용할 수 있게 됩니다. 다만, Web 페이지가 양쪽 모두의 브라우저로 표시되는 경우에는 변경이 필요하게 됩니다. 즉, Internet Explore 로 표시되는 경우에는 OBJECT 방식을 사용해, Netscape 로 표시되는 경우에는 EMBED 방식을 사용하는 것 같은 방법이 필요합니다. 이러한 방식에 대해서는, 이후에 설명합니다.

원의 APPLET 요소:

<APPLET code="XYZApp.class" codebase="html/" align="baseline"
    width="200" height="200">
<PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
    No Java 2 SDK, Standard Editoin v 1.4 support for APPLET!!
</APPLET>

상자로 된 EMBED 요소를 가지는 새로운 OBJECT 요소:

<OBJECT classid="clsid:8AD9C840-044E-11D1-B3E9-00805F499D93"
    width="200" height="200" align="baseline"
    codebase="http://java.sun.com/products/plugin/1. 4/jinstall-14-win32.cab#Version=1, 4,0, mn">
    <PARAM NAME="code" VALUE="XYZApp.class">
    <PARAM NAME="codebase" VALUE="html/">
    <PARAM NAME="type" VALUE="application/x-java-applet;jpi-version=1. 4">
    <PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
    <PARAM NAME="scriptable" VALUE="true">
    <COMMENT>
        <EMBED type="application/x-java-applet;jpi-version=1. 4" width="200"
           height="200" align="baseline" code="XYZApp.class" 
           codebase="html/" model="models/HyaluronicAcid.xyz"
           pluginspage="http://java.sun.com/products/plugin/1. 4/plugin-install.html">
            <NOEMBED>
                No Java 2 SDK, Standard Edition v 1.4 support for APPLET!!
            </NOEMBED>
        </EMBED>
    </COMMENT>
</OBJECT>

IE 는 <OBJECT> 태그를 인식하기 위해(때문에), Java Plug-in 의 기동을 시도합니다. <COMMENT> 태그는, IE 만이 이해하는 특수한 HTML 태그인 것에 유의해 주세요. IE 는,<COMMENT> 와 </COMMENT> 태그의 사이의 텍스트를 무시합니다. 실제의 곳, 상기의 태그는 다음과 같이 됩니다.

<COMMENT> 는 IE 고유의 요소입니다. IE 는,<COMMENT> 와 </COMMENT> 요소의 사이의 텍스트를 무시합니다. 따라서, 상기의 구조는, IE 에서는 다음과 같이 읽어내집니다. 

<OBJECT classid="clsid:8AD9C840-044E-11D1-B3E9-00805F499D93"
    width="200" height="200" align="baseline"
    codebase="http://java.sun.com/products/plugin/1. 4/jinstall-14-win32.cab#Version=1, 4,0, mn">
    <PARAM NAME="code" VALUE="XYZApp.class">
    <PARAM NAME="codebase" VALUE="html/">
    <PARAM NAME="type" VALUE="application/x-java-applet;jpi-version=1. 4">
    <PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
    <PARAM NAME="scriptable" VALUE="true">
</OBJECT>

Java Plug-in 를 지원하지 않는 플랫폼의 HotJava 브라우저, IE, 및 Navigator, 또는 JavaScript 를 지원하지 않는 브라우저는, 상기의 태그를 다음과 같이 읽어냅니다. 

<APPLET code="XYZApp.class" codebase="html/" align="baseline"
    width="200" height="200"></XMP>
<PARAM NAME="java_code" VALUE="XYZApp.class">
<PARAM NAME="java_codebase" VALUE="html/">
<PARAM NAME="java_type" VALUE="application/x-java-applet;jpi-version=1. 4">
<PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
<PARAM NAME="scriptable" VALUE="true">
   No Java 2 SDK, Standard Edition v 1.4 support for APPLET!!
</APPLET></NOEMBED></EMBED></OBJECT> 

이러한 브라우저는, 대응하는 <XMP>,<OBJECT>,<EMBED>, 및 <NOEMBED> 태그가 존재하지 않기 때문에,</XMP>,</OBJECT>,</EMBED>, 및 </NOEMBED> 태그를 무시합니다. Java Plug-in 는 Java 2 SDK, Standard Edition v 1.4 또는 그 이후의 릴리스의 기능을 타겟으로 하고 있기 (위해)때문에, Java 2 SDK 1.4 를 풀 지원하지 않고, 또 Java Plug-in 를 지원하지 않고 않은 이러한 브라우저에서는, 「No Java 2 SDK, Standard Edition v 1.4 support for APPLET」라고 하는 메세지가 표시됩니다.

전의 예와는 달라, 매핑 된 PARAM 명에는,code,codebase,type 대신에,java_code,java_codebase,java_type 가 포함됩니다. 이것이 필요한 이유는,<APPLET> 와 </APPLET> 태그 내부의 <PARAM> 에 code 및 codebase 를 지정하면(자), 브라우저에 따라서는 문제가 발생하기 (위해)때문에입니다.

Windows 상의 IE 는, 태그를 다음과 같이 읽어냅니다. 

<OBJECT classid="clsid:8AD9C840-044E-11D1-B3E9-00805F499D93"
    width="200" height="200" align="baseline"
codebase="http://java.sun.com/products/plugin/1. 4/jinstall-14-win32.cab#Version=1, 4,0, mn">
    <NOEMBED><XMP>
    <APPLET code="XYZApp.class" codebase="html/" align="baseline"
        width="200" height="200"></XMP>
    <PARAM NAME="java_code" VALUE="XYZApp.class">
    <PARAM NAME="java_codebase" VALUE="html/">
    <PARAM NAME="java_type" VALUE="application/x-java-applet;jpi-version=1. 4">
    <PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
    <PARAM NAME="scriptable" VALUE="true">
        No Java 2 SDK, Standard Edition v 1.4 support for APPLET!!
    </APPLET></NOEMBED></EMBED>
</OBJECT> 

<XMP> 태그를 사용하는 경우는, 주의가 필요합니다. IE 는 <OBJECT> 태그를 draw 하기 위한(해),<APPLET> 태그를 무효로 할 필요가 있습니다. 무효로 하지 않는 경우, 2 개의 애플릿이 브라우저에 동시에 표시됩니다. 1 개의 애플릿은 Microsoft 의 JVM 로, 이제(벌써) 1 개의 애플릿은 Sun 의 JVM 로 Java Plug-in 를 사용해 가동합니다. <XMP> 태그를 사용하는 것으로, 이 문제를 해결할 수 있습니다. 기본적으로,<XMP> 및 </XMP> 태그는, 태그 내부의 HTML 태그를 정적 텍스트의 스트림로 변환합니다. 위의 예에서는,<XMP> 및 </XMP> 태그를 사용하는 것으로, 브라우저는 <APPLET> 태그를 HTML 태그로서가 아니고, 정적 텍스트로서 처리합니다. 브라우저는 <OBJECT> 태그 및 <PARAM> 태그간의 정적 텍스트를 무시하기 위해(때문에), 상기의 태그는 실질적으로 다음과 같이 됩니다. 

<OBJECT classid="clsid:8AD9C840-044E-11D1-B3E9-00805F499D93"
    width="200" height="200" align="baseline"
codebase="http://java.sun.com/products/plugin/1. 4/jinstall-14-win32.cab#Version=1, 4,0, mn">
    <PARAM NAME="java_code" VALUE="XYZApp.class">
    <PARAM NAME="java_codebase" VALUE="html/">
    <PARAM NAME="java_type" VALUE="application/x-java-applet;jpi-version=1. 4">
    <PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
    <PARAM NAME="scriptable" VALUE="true">
        No Java 2 SDK, Standard Edition v 1.4 support for APPLET!!
</OBJECT> 

이것은, 벌써 개요를 설명한 OBJECT 태그의 예와 동일합니다. <OBJECT> 태그는,<NOEMBED>,</NOEMBED>, 및 </EMBED> 태그를 무시하는 것에 유의해 주세요.

Windows 오퍼레이팅(operating) 환경에서 동작하는 Navigator 는, 태그를 다음과 같이 읽어냅니다. 

<EMBED type="application/x-java-applet;jpi-version=1. 4" width="200" height="200"
    align="baseline" code="XYZApp.class" codebase="html/"
    model="models/HyaluronicAcid.xyz"
    pluginspage="http://java.sun.com/products/plugin/1. 4/plugin-install.html">
<NOEMBED><XMP>
    <APPLET code="XYZApp.class" codebase="html/" align="baseline"
        width="200" height="200"></XMP>
    <PARAM NAME="java_code" VALUE="XYZApp.class">
    <PARAM NAME="java_codebase" VALUE="html/">
    <PARAM NAME="java_type" VALUE="application/x-java-applet;jpi-version=1. 4">
    <PARAM NAME="model" VALUE="models/HyaluronicAcid.xyz">
    <PARAM NAME="scriptable" VALUE="true">
        No Java 2 SDK, Standard Edition v 1.4 support for APPLET!!
    </APPLET></NOEMBED></EMBED></OBJECT>

<APPLET> 태그도 무효로 하기 (위해)때문에,<XMP> 태그가 <EMBED> 태그내에서 재차 사용되고 있는 점에 주목해 주세요. <EMBED> 태그는,<PARAM> 및 </OBJECT> 태그도 무시합니다. 실제의 곳, 상기의 태그는 다음과 같이 됩니다. 

<EMBED type="application/x-java-applet;jpi-version=1. 4" width="200" height="200"
    align="baseline" code="XYZApp.class" codebase="html/"
    model="models/HyaluronicAcid.xyz"
    pluginspage="http://java.sun.com/products/plugin/1. 4/plugin-install.html">
<NOEMBED>
        No Java 2 SDK, Standard Edition v 1.4 support for APPLET!!
</NOEMBED>
</EMBED> 

이것은, 벌써 개요를 설명한 EMBED 태그의 예와 동일합니다.

결합된 OBJECT-EMBED-JavaScript 태그를 사용해, 적절한 플랫폼의 적절한 브라우저로 Java Plug-in 를 기동할 수 있습니다. 이 결합된 태그는 복잡하기 때문에, 이종 혼재 환경에서 사용자가 HTML 페이지를 브라우즈 하는 경우에게만 사용해 주세요.

통계

이 문서에서는, Java Plug-in 의 사용하는 OBJECT 태그 및 EMBED 태그의 스타일에 대해 설명했습니다. 특히,APPLET 태그로부터 OBJECT 및 EMBED 태그에의 변환에 초점을 맞혀 해설했습니다. 현재로서는, HTML 4.0 에서는, HTML 페이지에 Java 애플릿 및 JavaBeans 컴퍼넌트를 삽입하는 방법으로서는,OBJECT 태그가 최선의 선택사항입니다. 이 문서가 가까운 장래 갱신되려면 ,OBJECT 태그를 Java Plug-in 의 태그 스타일로 변환할 필요가 생겨 오겠지요. 이 문서로 공개한 정보의 목적은, HTML 이행 툴을 기술하는 ISV, 및 Java Plug-in 로 이행하려 하고 있는 Web 마스터를 지원하는 것입니다. 이 문서로 설명한 태그 스타일은, 장래 변경될 가능성이 있습니다.

Java Plug-in 의 사용은, 이 문서로 설명한 태그 스타일로 한정되고 있지 않습니다. 실제, Web 마스터에는, 필요에 따라서 태그 스타일을 변경하는 것이나, 태그를 JavaScript 와 조합하는 것이 권유받고 있습니다. 이 문서로 해설한 OBJECT 태그를 IE 로, 또 EMBED 태그를 Navigator 로 사용하는 한, Java Plug-in 의 실행에는 무슨 문제도 없습니다. 현재로서는, Java Plug-in HTML 컨버터에는 몇개의 변환 템플릿이 동고 되고 있습니다. Web 마스터는, 그 중에서 스스로의 요구에 맞은 템플릿을 찾아낼 수가 있습니다. 또, 필요에 따라서 이러한 템플릿을 수정하는 일도 권유받고 있습니다.
