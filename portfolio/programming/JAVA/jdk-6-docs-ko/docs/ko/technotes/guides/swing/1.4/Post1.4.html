JFileChooser 퍼포먼스의 확장

이 변경에 관련하는 버그 추적 리포트: 4679673 및 4712307

Windows XP 의 Look & Feel

릴리스 1.4. 2 에서는, Windows XP 플랫폼에서 실행하고 있는 경우, 디폴트로서 표준 Microsoft Windows XP 의 외관이 지원되고 있습니다. 이 Look & Feel 는, Windows XP operating system를 실행하고 있는 머신으로, 어플리케이션이 Swing 의 WindowsLookAndFeel 클래스 (UIManager.getSystemLookAndFeelClassName() 를 경유하는지,com.sun.java.swing.plaf.windows.WindowsLook & Feel 를 명시적으로 사용할까의 어느쪽이든)를 사용하고 있는 경우에 자동적으로 표시됩니다. 다음에, 네이티브 플랫폼의 것과 일치하는 Look & Feel 의 우선되는 설정의 예를 나타냅니다. 

  UIManager.setLookAndFeel(UIManager.getSystemLookAndFeelClassName());

Swing 의 어플리케이션에 구래의 Windows 어플리케이션의 외관을 지정하는 경우는, 시스템 프로퍼티 swing.noxp=true 를 사용해 XP 의 외관을 변경할 수가 있습니다. 이것은 일반적으로, 다음과 같이 커멘드행 파라미터를 사용해 실시합니다. 

  java -Dswing.noxp=true -jar SwingSet2.jar

swing.noxp 프로퍼티은, 장래의 릴리스로 지원되지 않게 될 가능성이 있을테니 주의해 주세요.

GTK+ 의 Look & Feel

릴리스 1.4. 2 에서는,GTK+ 2.0 에 근거한 Look & Feel 가 도입되고 있습니다. 이 Look & Feel 를 가져오려면 ,UIManager.setLookAndFeel (com.sun.java.swing.plaf.gtk.GTKLookAndFeel) 를 경유해, 또는,swing.defaultlaf 시스템 프로퍼티을 com.sun.java.swing.plaf.gtk.GTKLookAndFeel 에 정의해 명시적으로 요구할 필요가 있습니다. 다음번의 릴리스에서는, 적절한 때에,UIManager.getSystemLookAndFeelClassName() 가 이 Look & Feel 를 돌려주도록(듯이) 할 예정입니다.

GTK+ 의 Look & Feel 는,resource file에 의해 커스터마이즈 할 수 있습니다. Swing 의 GTK+ Look & Feel 에서는, 이하의 알고리즘을 사용해 resource file를 검색합니다.

  1. 시스템 프로퍼티의 swing.gtkthemefile 가 존재하는 경우는, 그 시스템 프로퍼티을 해석해 정지합니다. 예를 들어, 다음과 같이 됩니다. java -Dswing.gtkthemefile=/tmp/customTheme -jar SwingSet2.jar.
  2. user.home/.gtkrc-2. 0 파일이 존재하는 경우는, 그 파일을 해석해, 먼저 진행됩니다.
  3. XSETTINGS 를 사용해 결정되는 데스크탑 프로퍼티 gnome.net/ThemeName 에 의해, 사용자가 선택하는 테마명 (THEMENAME)을 결정합니다. 이 프로퍼티이 null 의 경우,Default 를 THEMENAME 로서 사용합니다.
    1. user.home/.themes/THEMENAME/gtk-2. 0/gtkrc 파일이 존재하는 경우는, 그 파일을 해석해, 정지합니다.
    2. 시스템 프로퍼티의 swing.gtkthemedir 가 있어,swing.gtkthemedir/THEMENAME/gtk-2. 0/gtkrc 파일이 존재하는 경우에는, 그 파일을 해석해, 정지합니다.
    3. 시스템 프로퍼티의 swing.gtkthemedir 가 없고,/usr/share/themes/THEMENAME/gtk-2. 0/gtkrc 가 존재하는 경우에는, 그 파일을 해석해, 정지합니다.
    4. 마지막으로,swing.gtkthemedir 가 정의되고 있는 경우는,swing.gtkthemedir/THEMENAME/gtk/gtkrc 를 해석합니다. 정의되어 있지 않은 경우는,/usr/share/themes/THEMENAME/gtk/gtkrc 를 해석합니다.

GTK+ 를 커스터마이즈 하는 1 개의 방법은,테마 엔진을 사용하는 것입니다. 몇개의 엔진이 존재합니다. 1.4. 2 에서는, Swing 는 Default,pixmap, 및 bluecurve 엔진의 테마 파일을 지원하고 있습니다. 현재는, API 를 오픈으로 해 추가의 GTK 엔진을 작성할 수 있도록(듯이) 하는 방법을 조사중입니다. 예에 대해서는,http://www.themes.org 를 참조해 주세요.

GTK+ 에서는,XLFD 를 사용하는지,pango 캐릭터 라인을 사용해 폰트를 지정할 수 있습니다. Pango 에서는, Java 플랫폼의 합성 폰트 와 유사한 기능을 제공하고 있습니다. 즉, 다양한 코드 포인트의 다양한 폰트로 구성되는 폰트를 합성하는 기능입니다. Pango 의 합성 폰트는, 정확하게는 Java 의 합성 폰트와 일치하지 않기 때문에,GTKLookAndFeel 를 사용하는 Swing 의 어플리케이션에서는, 네이티브의 GTK+ 어플리케이션과 엄밀하게는 일치하지 않습니다. 현재, 이 문제에 대처하는 방법을 조사중입니다. 1 개의 옵션으로서 Pango 의 폰트 파일을 Java 의 font.properties 파일과 동등의 버젼으로 옮겨놓는 방법이 있습니다. 또한 현재는, XLFD 를 무시해, sans 를 sansserif 에, monospace 를 monospaced 에 매핑 한 뒤에, Pango 캐릭터 라인을 직접 java.awt.Font 에 건네주고 있습니다.

GTK+ 에서는, MDI (JInternalFrame) 기능은 제공하고 있습니다. 또, 최상정도의 윈도우의 테마 구성을 직접 행하지 않습니다. 이 지원는, WindowManager 가 실시합니다. Swing 의 MDI 클래스에 대해서는,Metacity 의 형식에서 테마 구성이 지원되고 있습니다. 현재 지원되고 있는 것은,Crux 및 Bluecurve 의 테마 뿐입니다. 이러한 테마의 모두 검색할 수 없는 경우, 독자적인 커스텀 테마가 실행됩니다. 시스템 프로퍼티의 swing.metacitythemename 를 사용해, 사용하는 특정의 테마로 이름을 붙일 수가 있습니다. 그렇게 하지 않는 경우는,user.dir/.gconf/apps/metacity/general/%gconf.xml 내를 검색해 테마의 이름을 판정해,/usr/share/themes/THEMENAME/metacity-1/metacity-theme-1.xml 또는 /usr/gnome/share/themes/THEMENAME/metacity-1/metacity-theme-1.xml 내를 검색해 실제의 resource file를 찾습니다. 이하의 예에서는, Swing 를 트리거해 특정의 metacity 테마를 사용하는 방법을 나타내고 있습니다. 

  java -Dswing.metacitythemename=Bluecurve -jar SwingSet2.jar

GTK+ resource file에서는, 외관 이상으로 많은 Look & Feel 옵션을 커스터마이즈 할 수가 있습니다. 1.4. 2 에서는, 주요한 스타일 구성 (fg, bg, base ..), gtk-font-name, gtk-icon-sizes, 스톡 아이콘 (optionpane 용), focus-line-width, focus-padding, focus-line-pattern, internal-padding, shadow-type (도구모음용), 및 trough-border 가 지원되고 있습니다.

GTK 에서는, 실행중의 어플리케이션이 테마의 변경을 감지해 자동적으로 UI 를 갱신할 수 있는 기구를 제공하고 있습니다. 이것은, 향후의 Swing 의 GTK Look & Feel 의 버젼으로 지원될 예정입니다.

GTK 의 비호환성

Swing 의 현행의 아키텍쳐(architecture)로 GTK+ Look & Feel 를 적절히 지원하는 경우, 많은 과제가 있습니다. Swing 에서는, 불투명한 컴퍼넌트가 「항상」 ComponentUI 의 업데이트 메소드를 통해서 그 배경을 그립니다. GTK+ 에서는, 엔진에 따라서는, 불투명하지 않은 컴퍼넌트가 백 라운드를 페인트 하는 경우가 있습니다. GTK+ 를 적절히 반영하기 위해서, 이전에는 불투명함 많은 컴퍼넌트가 불투명하지 않게 되었습니다 (ComponentUI 서브 클래스의 installUI 메소드에 의한다). 게다가 컴퍼넌트의 외관은 포함 관계의 계층에 근거해 변경할 수가 있으므로, 생성자 의 완료 후에 불투명도가 바뀔 가능성이 있습니다. 이 결과, 컴퍼넌트로 setOpaque()를 호출하면(자), 예기치 않은 변경이 생길 가능성이 있습니다. 이것은 다음의 릴리스로 수정됩니다.

GTK+ 는 엔진에 불려 가, 위젯의 실제의 내용이 페인트 되기 전에 경계 장식을 그립니다. 엔진에 따라서는 (특히 pixmap 엔진은) 이 순서에 의해 내용의 배경의 장식을 올바르게 draw 합니다. Swing 에서는, 경계는, 위젯의 내용이 페인트 된 후에 draw 됩니다. GTKLookAndFeel 에서는, 공백을 제공하는 목적에서만 Border 가 인스톨 되어 실제의 경계 장식의 draw는 paint 메소드의 일부로서 UI 가 가고 있습니다. 이 때문에, GTKLookAndFeel 로 위젯에 Border 를 인스톨 하면(자), 바라고 있는 효과를 얻을 수 없는 것이 있습니다. 현재는 이것에 대신하는 방법을 조사하고 있어, 다음의 릴리스로 보다 좋은 해결책이 제공될 예정입니다.
