javax.swing.FocusManager 의 변경

이 클래스는 폐지되었습니다. 새로운 어플리케이션에서는,java.awt.KeyboardFocusManager 마셔 사용해 주세요. 다만, 기존의 어플리케이션에 대한 (애플릿은 제외하다) 하위 호환성은 남아 있어 독자적인 FocusManager 를 정의할 수 있습니다.

Swing FocusManager 는, 새로운 AWT KeyboardFocusManager 의 서브 클래스로 변경되었습니다. FocusManager.setCurrentManager(FocusManager) 및 FocusManager.getCurrentManager() 는, 각각 KeyboardFocusManager.setCurrentKeyboardFocusManager(KeyboardFocusManager) 및 KeyboardFocusManager.getCurrentKeyboardFocusManager() 의 래퍼로 변경되었습니다.

이것에 의해, 호출측 thread에 액세스권 「replaceKeyboardFocusManager」 가 주어지지 않는 경우 KeyboardFocusManager.setCurrentKeyboardFocusManager(KeyboardFocusManager) 가 SecurityException 를 throw 하게 되기 (위해)때문에, 이 변경에 관해서 많은 논의가 일어났습니다. 지금까지의 버젼에서는,FocusManager.setCurrentManager(FocusManager) 는 SecurityException 를 throw 할 것은 없었습니다. 복잡한 클래스를 설계해 이 문제를 피할 수 있었습니다만, 제안되고 있던 설계의 명해 마을 간결함은, 하위 호환성에 관한 몇 안 되는 문제보다 중요하다라고 생각했습니다.

disableSwingFocusManager() 및 isFocusManagerEnabled() 는 사용해야 하는 것이 아닙니다. Swing FocusManager 를 무효로 하는 것으로, 디폴트의 FocusTraversalPolicy 를 javax.swing.LayoutFocusTraversalPolicy 의 Swing 디폴트로부터 java.awt.DefaultFocusTraversalPolicy 의 AWT 디폴트에 되돌리는 시멘틱스를 제공할 수 있게 되었습니다.

javax.swing.DefaultFocusManager 의 변경

이 클래스는 폐지되었습니다. 새로운 어플리케이션에서는,java.awt.KeyboardFocusManager 마셔 사용해 주세요. 다만, 기존의 어플리케이션에 대한 (애플릿은 제외하다) 하위 호환성은 남아 있어 독자적인 DefaultFocusManager 를 정의할 수 있습니다.

클라이언트 코드가 종래의 DefaultFocusManager 를 인스톨 하는 경우, public 이외의 LegacyGlueFocusTraversalPolicy 를 인스톨 하면(자) 하위 호환성이 제공됩니다. 이것에 의해 FocusTraversalPolicy 메소드 호출이 DefaultFocusManager 의 대응하는 메소드에게 건네집니다.

javax.swing.JComponent 의 변경

isManagingFocus:이 메소드는 추천 되지 않게 되었습니다. isManagingFocus 는 JComponent 의 포커스 traversal 키를 TAB 및 SHIFT-TAB 로부터 CTRL-TAB 및 CTRL-SHIFT-TAB 로 변경했습니다. 또, 포커스 traversal가 자동적으로 JComponent 가 되지 않게,JComponent 를 마크 했습니다. 이것은 JComponent 를 포커스 사이클 루트로서 지정하는 것으로 닮습니다만, 다른 것입니다. 그 기능은 FocusTraversalPolicy 클래스, 컴퍼넌트마다의 포커스 traversal 키, 및 컴퍼넌트의 포커스 traversal 키를 정리해 무효로 하는 기능으로 옮겨졌습니다.

setNextFocusableComponent/getNextFocusableComponents:이러한 메소드는 폐지되었습니다. FocusTraversalPolicies 에서는 포커스 traversal의 알고리즘으로 표현할 수 있게 되어, 신뢰성이 높아지고 있습니다. 이러한 메소드는, 어플리케이션이 컴퍼넌트로부터 다른 컴퍼넌트에의 포커스 traversal를 하드 코드 하도록(듯이) 강제합니다.

setRequestFocusEnabled /isRequestFocusEnabled 의 스펙이 변경되어requestFocusEnabled 프로퍼티 스펙은 현재의 Look＆Feel 에 대한 힌트가 됩니다.

requestDefaultFocus:이 메소드는 추천 되지 않게 되었습니다. 이 메소드의 기능은 FocusTraversalPolicy.getDefaultComponent(Container). requestFocus() 로 옮겨놓을 수 있고 있습니다.

javax.swing.JWindow 의 변경

setFocusableWindow /isFocusableWindow :이러한 메소드는 JWindow 로 final 선언되어 「null」소유자에게 구축된 모든 JWindows 는 non-focusable 이다고 하는 제한이 적용됩니다. Swing 로 JWindows 를 구축해,JWindows 가 사유로 비표시의 Frame 에 소유되기 (위해)때문에, 또, 새로운 포커스 구현에,Window 나 그 자손을 대상으로 한 KeyEvents 가 그 Frame 로 프록시 될 필요가 있기 (위해)때문에, 이러한 제한이 필요하게 됩니다. X11 에서는 포커스를 설정하기 위해서 window 를 맵 할 필요가 있으므로, 「null」소유자를 가지는 JWindow 는 X11 로 KeyEvents 를 받을 수가 없습니다.

javax.swing.JInternalFrame 의 변경

getMostRecentFocusOwner :java.awt.Window 와 같이,JInternalFrame 는 java.awt.KeyboardFocusManager 로 포커스 관리를 집중화하기 이전으로부터 getFocusOwner() 메소드를 남기고 있었습니다. API 를 완성해, Window 에 대해서 일관성을 유지하기 위해서(때문에),getMostRecentFocusOwner() 가 JInternalFrame 에 추가되었습니다. 스펙은 Window 로 대응하는 메소드와 같습니다.

setFocusCycleRoot /isFocusCycleRoot :이러한 메소드는,JInternalFrames 에 AWT 의 최상정도 Windows 와 같은 시멘틱스를 줍니다. JInternalFrame 는 항상 포커스 사이클 루트일 필요가 있으므로, 이러한 메소드가 오버라이드(override) 되어 final 선언되어 제한이 적용됩니다. 다만 AWT Window 과는 달리,JInternalFrame 에는 JDesktopPane 와 같이 바탕으로 되는 포커스 사이클 루트가 있는 경우가 있습니다. 이 때문에,Component.getFocusCycleRootAncestor() 는 Window 의 경우와 같이 JInternalFrame 로 final 선언되는 일이 없습니다.

javax.swing.KeyStroke 의 변경

기존의 KeyStroke 클래스를 사용해 각 Component 에 포커스 traversal 키 API 를 설계할 수가 있었으므로, 이 클래스는 그다지 변경되지 않고 java.awt.AWTKeyStrokejava.awt.AWTKeyStroke 로서 AWT 로 변경되었습니다. Swing 클래스가 AWT 클래스의 서브 클래스가 되었습니다.

KeyStroke,getKeyStroke(char, boolean) 로 추천 되지 않는 메소드는 AWT 로 변경되지 않았습니다. 이 때문에, 이 메소드를 표준의 AWTKeyStroke 캐싱 스킴에 간단하게 관여시키는 방법은 없습니다. 이 메소드의 스펙은 변경되어 구현에는 이 메소드로부터 반환된 인스턴스를 캐쉬할 필요가 없는 것이 주의 사항에 추가되었습니다.

javax.swing.SwingUtilities 의 변경

findFocusOwner:이 메소드는 추천 되지 않게 되었습니다. 이 메소드의 기능은 KeyboardFocusManager.getFocusOwner() 로 옮겨놓을 수 있고 있습니다.
