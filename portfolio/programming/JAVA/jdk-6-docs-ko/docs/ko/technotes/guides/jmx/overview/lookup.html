제 6 장

검출·검색 서비스

Java Management Extensions (JMX) Remote API Specification」에서는, 기존의 검출·검색 인프라스트럭쳐(infrastructure)를 사용해 JMX API 에이전트를 통지하거나 검색하거나 할 방법이 설명되고 있습니다. 이 스펙에서는, JMX 테크놀러지 고유의 검출·검색 API 에 대해서는 정의되고 있지 않습니다.

이 장에서는, JMX 테크놀러지로 사용할 수 있는 기존의 검출·검색 인프라스트럭쳐(infrastructure)의 개요를, 이하의 섹션으로 간단하게 설명합니다.

  • "검색 서비스에 대해 "
  • "Service Location Protocol (SLP)의 사용 "
  • "Jini 네트워크 테크놀러지의 사용 "
  • "LDAP 연구 최종 단계에서의 Java Naming and Directory Interface (JNDI) API 의 사용 "

이러한 인프라스트럭쳐(infrastructure)에 대한 자세한 것은, 「Java Management Extensions Remote API 1.0 Specification」의 「References」섹션을 참조해 주세요.

검색 서비스에 대해

검색 서비스는 JMX 에이전트와 JMX 클라이언트가 사용합니다. 1 개의 Java VM 에는, 다수의 JMX 에이전트나 JMX 클라이언트가 포함되어 있는 경우가 있습니다.

  • JMX 에이전트는, 다음의 기능으로부터 되는 논리 서버 어플리케이션입니다.
    • MBean 서버가 1 개
    • JMX 연결기 서버가 1 개 이상. JMX 연결기 서버에 의해, 그 MBean 서버에 포함되는 MBean 에 대해서 원격 클라이언트를 액세스 할 수 있게 된다
  • JMX 클라이언트는, JMX 에이전트와의 클라이언트 접속을 여는 논리 클라이언트 어플리케이션입니다.

「Java Management Extensions (JMX) 테크놀러지의 튜토리얼」과 부속의 예에서는, 검색 서비스를 사용해 JMX 에이전트를 통지하거나 검색하거나 할 방법이 나타나고 있습니다. 3 개의 인프라스트럭쳐(infrastructure)의 순서는 닮았습니다. 크게 다른 점은, SLP 와 JNDI 의 경우는, 검색 서비스로 에이전트가 등록하는 것은 주소입니다만, Jini 네트워크 테크놀러지를 사용했을 경우에 등록하는 것은 JMX 연결기 Stub입니다.

주 – 이 스펙에서는, 독자적인 검출·검색 서비스는 정의되고 있지 않습니다. 기존의 검출·검색 서비스를 사용할 수도 있습니다만, JMX API 에이전트의 주소를 URL 의 형식에서 encode 해, 그 URL 를 매니저에게 통지할 수도 있습니다.

Service Location Protocol (SLP)의 사용

Service Location Protocol (SLP)에서는, 기업 네트워크내에서 네트워크화된 서비스의 존재, 위치, 및 구성을 네트워크 기능 어플리케이션을 검출할 수 있는 시스템를 제공합니다.

다음의 순서는, SLP 검색 서비스를 사용해 JMX 에이전트를 통지 및 검색하기 위한 , JMX Remote API 스펙으로 정의된 순서를 집계한 것입니다.

  1. 에이전트는 1 개 이상의 JMX 연결기 서버를 작성합니다.
  2. 공개하는 각 연결기에 대해, 에이전트는 SLP 검색 서비스를 사용해, 그 주소를 등록합니다. 이 때 경우에 따라서는, 에이전트나 연결기를 한정해, 필터로서 사용할 수 있는 것 같은 추가의 속성을 지정합니다.
  3. 클라이언트는 SLP 검색 서비스를 쿼리해, 쿼리에 들어맞는 1 개(살) 이상의 주소를 가져옵니다.
  4. 마지막으로, 클라이언트는, 취득한 주소로 특정되는 서버에 대해서 접속하고 있는 연결기를 가져옵니다.

JMX Remote API 스펙에서는, SLP 프로토콜에 준거한 URL 스킴을 정의하고 있습니다. Java Management Extensions (JMX) Remote API Specification」를 참조해 주세요. 스펙에서는, 등록시로 지정되는, 필수 및 생략 가능한 SLP 검색 속성도 정의하고 있습니다.

Jini 네트워크 테크놀러지의 사용

JiniTM 네트워크 테크놀러지는, 개방적인 소프트웨어 아키텍쳐(architecture)로, 개발자는 네트워크의 변화에 대응 가능한 서비스를 작성할 수 있습니다. Jini 의 스펙에서는, 표준의 검색 서비스가 있습니다. 실행중의 Jini 검색 서비스는, 간단한 API 호출로 검출할 수 있습니다.

다음의 순서는, Jini 검색 서비스를 사용해 JMX 에이전트를 통지 및 검색하기 위한 , JMX Remote API 스펙으로 정의된 순서를 집계한 것입니다.

  1. 에이전트는 1 개 이상의 JMX 연결기 서버를 작성합니다.
  2. 공개하는 각 연결기에 대해, 에이전트는 Jini 검색 서비스를 사용해, JMX 연결기 Stub를 등록합니다. 이 때 경우에 따라서는, 에이전트나 연결기를 한정해, 필터로서 사용할 수 있는 것 같은 추가의 속성을 지정합니다.
  3. 클라이언트는 Jini 검색 서비스를 쿼리해, 쿼리에 들어맞는 1 개(살) 이상의 연결기 Stub를 가져옵니다.
  4. 마지막으로, 클라이언트는, 취득한 연결기 Stub를 사용해, 서버에 직접 접속합니다.

JMX Remote API 스펙에서는, Jini 테크놀러지 베이스의 엔트리에 의한 바인드를 정의하고 있습니다. Java Management Extensions (JMX) Remote API Specification」를 참조해 주세요. 스펙에서는, 에이전트 연결기의 등록시로 지정하는, 필수 및 생략 가능한 엔트리도 정의하고 있습니다.

LDAP 연구 최종 단계에서의 Java Naming and Directory Interface (JNDI) API 의 사용

Java Naming and Directory InterfaceTM (JNDI) API 는, Java 플랫폼의 표준의 기능확장입니다. Java 테크놀러지 대응의 어플리케이션에, 다양한 네임 및 디렉토리 서비스에 대한 통일적인 인터페이스를 제공합니다.

JMX Remote API 스펙에서는, LDAP 서버를 사용해, JMX 에이전트에 공개된 JMX 연결기의 정보를 포함 및 취득하는 방법을 상세하게 정하고 있습니다.

다음의 순서는, Jini 검색 서비스를 사용하기 위한 , JMX Remote API 스펙으로 정의된 순서를 집계한 것입니다.

  1. 에이전트는 1 개 이상의 JMX 연결기 서버를 작성합니다.
  2. 공개하는 각 연결기에 대해, 에이전트는 JNDI 검색 서비스를 사용해, 그 주소를 등록합니다. 이 때 경우에 따라서는, 에이전트나 연결기를 한정해, 필터로서 사용할 수 있는 것 같은 추가의 속성을 지정합니다.
  3. 클라이언트는 JNDI 검색 서비스를 쿼리해, 쿼리에 들어맞는 1 개(살) 이상의 주소를 가져옵니다.
  4. 마지막으로, 클라이언트는, 취득한 주소로 특정되는 서버에 대해서 접속하고 있는 연결기를 가져옵니다.

JMX Remote API 에서는, 주소를 등록하기 위한 LDAP schema를 정의해, 또 등록한 에이전트를 클라이언트가 검출하는 방법에 대해 설명하고 있습니다. 자세한 것은, 「JVM 1.4 스펙」을 참조해 주세요. 스펙에서는, 리스 기구도 정의하고 있습니다.
