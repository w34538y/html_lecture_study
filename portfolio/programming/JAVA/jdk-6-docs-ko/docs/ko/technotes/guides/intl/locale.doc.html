지원되고 있는 로케일

지원되는 로케일세트는, Java SE (Java Platform Standard Edition)의 구현이나 기능 범위의 차이에 의해 다릅니다.

여기에서는, 다음의 분야에 있어 Sun 의 JDK (Java SE Development Kit) 6 및 JRE (Java SE Runtime Environment) 6 으로 지원되는 로케일에 대해 설명합니다.

Sun 의 Java SE Runtime Environment 6 의 Windows 판은, 완전한 국제판과 유럽 언어판의 어느 쪽의 버젼에서도 인스톨 할 수가 있습니다. JRE 의 인스톨 프로그램은, 호스트 operating system가 유럽 언어만을 지원하고 있는 것을 식별하면(자), 디폴트로 유럽 언어판을 인스톨 합니다. 인스톨 프로그램이 다른 언어의 필요성을 식별했을 경우, 또는 커스터마이즈 된 인스톨로 사용자가 유럽 언어 이외의 지원를 요구했을 경우는, 완전한 국제화판이 인스톨 됩니다. Solaris 및 Linux 판의 JRE 와 모든 플랫폼판의 JDK 에서는, 완전한 국제화판이 항상 인스톨 됩니다.

1. java.util 및 java.text 기능으로 유효한 로케일

java.util 및 java.text 패키지에서의 로케일에 의존하는 동작의 지원는, 거의 완전하게 플랫폼으로부터 독립하고 있습니다. 그 때문에, 모든 로케일은, 호스트의 operating system와 그 지역에는 의존하지 않고, 같은 방법으로 동시에 지원됩니다. 플랫폼에 의존하고 있는 유일한 기능은, 호스트 operating system의 로케일과 타임 존에 근거한, 최초의 디폴트의 로케일 및 최초의 디폴트의 타임 존의 설정입니다.

Sun 의 Java SE Development Kit 6 으로 국제화판의 Java SE Runtime Environment 6 은, 다음에 나타내는 로케일을 모두 지원하고 있습니다. 유럽 언어판의 Java SE Runtime Environment 6 은, 다음의 언어에 대응하는 로케일을 모두 지원하고 있습니다. 알바니아어, 베로르시아어, 불가리아어, 카타로니아어, 크로아티아어, 체코어, 덴마크어, 네델란드어, 영어, 에스토니아어, 핀란드어, 프랑스어, 독일어, 희랍어, 헝가리어, 아이슬랜드어, 인도네시아어, 이탈리아어, 라트비아어, 리투아니아어, 마케도니아어, 말레이어, 몰타어, 노르웨이어, 폴란드어, 포르투갈어, 루마니아어, 러시아어, 세르비아어 (키릴), 슬로바키아어, 슬로베니아어, 스페인어, 스웨덴어, 터키어, 우크라이나어.

2. Java Foundation Class (JFC)로 유효한 서기법

2.1 개요

Java Foundation Classes (AWT, Swing, 2 D, Input Method Framework, 드래그＆드롭)의 경우는, 일반적으로, 로케일은 서기 체계에 의해서만 특징지울 수 있어 나라나 언어에 의한 특별한 차이는 없습니다. JFC 내의 서기 체계의 지원는, 호스트 operating system에 있는 정도 의존하고 있어, 복수 언어의 동시 사용은, 항상 완전하게 지원된다고는 할 수 없습니다.

여기에서는, 어느 서기 체계에 대해서, JFC 에 의한 모든 기능이 다음의 상황으로 적절히 동작하는 경우에, 그 서기 체계가 JFC 로 지원되고 있으면(자) 간주합니다.

  • Windows 98 의 경우, 그 서기법을 사용하는 언어에 의해 로컬라이즈 된 Windows 시스템으로 실행될 때.
  • Windows 2000, XP, 및 2003 의 경우, 사용자 로케일 및 시스템 로케일의 양쪽 모두가 그 서기법을 사용하는 언어로 설정된 Windows 시스템으로 실행될 때
  • Solaris 및 Linux 의 경우, 그 서기 체계를 사용하는 언어, 및 아래의 겉(표)에 나타나고 있는, 그 서기 체계의 인코딩의 1 개로 설정된 로케일을 사용해, 호스트 operating system로 실행될 때

Sun 의 Java SE Development Kit 6 으로 국제화판의 Java SE Runtime Environment 6 은, 다음에 나타내는 서기법을 모두 지원하고 있습니다. 유럽 언어판의 Java SE Runtime Environment 6 은, 키릴, 희랍어, 및 라틴의 서기법만을 지원하고 있습니다. 피어 AWT 컴퍼넌트는, 서기 체계의 부분집합 에 대해서만 지원됩니다. 겉(표)의 우단의 열을 참조해 주세요.

기능의 각각의 범위에 대해서는, 후의 각 항에서 자세하게 설명합니다.

2.2 텍스트 입력

텍스트 입력에 대한 지원는, 키보드 배열의 해석과 입력 메소드를 사용한 텍스트 변환의, 2 개의 부분으로부터 구성됩니다. Java SE 에서는, 키보드 배열의 해석은, 호스트 operating system에 전면적으로 의존하고 있습니다. 입력 메소드를 사용한 텍스트 변환에 대해서는, Java SE 에서는, Java 프로그램 언어로 개발된 입력 메소드 외에, 호스트 operating system의 입력 메소드 매니저를 사용한, 네이티브 입력 메소드도 지원됩니다.

Java 프로그램 언어에 의해 구현된 입력 메소드에 의한 로케일의 지원는, 인스톨 되고 있는 입력 메소드세트에게만 의존해, 호스트 operating system와 그 로컬라이제이션에는 의존하지 않습니다. 다만, Java 프로그램 언어에 의해 구현된 입력 메소드를 피어 컴퍼넌트로 사용하는 경우의 지원는, 이하와 같이, 그 구현 방법으로 의존합니다.

키보드 배열과 네이티브 입력 메소드의 지원는, 플랫폼에 따라서 다릅니다.

Windows

Windows 98 에서는, JRE 는, 모든 키보드 배열과 특정 로컬라이즈판의 호스트 operating system에 인스톨 가능한, IMM 베이스의 입력 메소드의 사용을 지원합니다.

Windows 2000, XP, 및 2003 에서는, JRE 는, 모든 키보드 배열과 IMM 베이스의 입력 메소드의 사용을 지원합니다.

글로벌 IME 는 지원되지 않습니다.

Java 프로그램 언어에 구현된 입력 메소드는, Windows 의 모든 버젼의 모든 컴퍼넌트로 지원됩니다.

Solaris 및 Linux

JRE 는, 모든 키보드 배열과 특정의 Solaris 또는 Linux 로케일로 사용 가능한 입력 메소드의 사용을 지원합니다.

Java 프로그램 언어에 구현된 입력 메소드는, Swing 텍스트 컴퍼넌트등의 경량 컴퍼넌트로 지원됩니다만, AWT 텍스트 컴퍼넌트등의 피어 컴퍼넌트에서는 지원되지 않습니다.

2.3 텍스트의 렌더링

어플리케이션에는, 폰트를 선택하는 2 개의 옵션이 있습니다.

  • 논리 폰트명의 Serif, SansSerif, Dialog, DialogInput, 및 Monospaced 의 사용
  • 물리 폰트의 사용 (이름을 지정해 요구하는지,Font.createFont 메소드로 인스턴스를 생성한다)

경량 컴퍼넌트내에서의 텍스트의 렌더링

논리 폰트명을 사용하는 경우는, 적어도, 호스트의 로케일의 서기 체계라고 라틴 서기 체계의 서구 어파 부분집합에 의한 텍스트가 지원됩니다.

물리 폰트를 사용하는 경우는, 단순 서기 체계와 복잡 서기 체계를 구별할 필요가 있습니다. 단순 서기 체계에서는 문자와 Glyph가 1 대 1 으로 대응해, 각 Glyph는 baseline상에 왼쪽에서 오른쪽으로 연속해 배치되고 있습니다. 복잡 서기법에서는, 같은 문자에서도 문맥에 따라서 다른 Glyph가 사용되거나 합자가 사용되거나 금방 다른 곳으로라고 쓰여지는 경우가 있습니다. 또, 행의 레이아웃중에 재배치되거나 결합 기호등에 의해, Glyph의 배치에 다른 규칙이 적용되는 일도 있습니다.

2D 의 텍스트 렌더링 시스템에서는,위의 겉(표)에 일람표 나타나고 있는 단순 서기 체계와 복잡 서기 체계의 어떠한 편성에서도 지원됩니다. 이러한 제한내로는, 지원되는 서기 체계의 범위는 폰트에 의해 정해집니다. 1 개의 TrueType 폰트로, Unicode 전체에 대응하는 Glyph와 Unicode 베이스의 문자로부터 Glyph에의 1 개의 매핑 정보를 제공할 수 있습니다. 이러한 폰트에 의해, 2D 에서는 모든 단순 서기 체계외, 복잡 서기 체계도 지원할 수 있습니다. 그 외의 복잡 서기 체계는 지원되지 않습니다.

피어 컴퍼넌트내에서의 텍스트의 렌더링

논리 폰트명을 사용하는 경우는, 적어도, 호스트 operating system의 로케일의 서기법에 따르는 텍스트가 지원됩니다.

피어 컴퍼넌트에서는, 물리 폰트는 지원되지 않습니다.

인쇄시의 텍스트의 렌더링

인쇄 API 에는, 다음의 3 개가 있습니다.

  • java.awt.print.PrinterJob.getPrinterJob 메소드를 사용하는 2D 인쇄 API
  • java.awt.Toolkit.getPrintJob 메소드를 사용하는 AWT 인쇄 API
  • javax.print 패키지를 사용하는, 플러그 가능 서비스의 인쇄 API

AWT 및 2D 의 인쇄 API 를 사용한 텍스트 렌더링은, 화면상의 텍스트 렌더링과 동등의 기능입니다. 플러그 인 기능 서비스의 인쇄 API 를 사용한 텍스트 렌더링은, 사용하는 인쇄 서비스에 따라서 다릅니다. JRE 에 의해 제공되는 서비스는, 화면상의 텍스트 렌더링과 동등의 기능입니다.

2.4 드래그＆드롭

Windows 98 의 경우, 호스트 operating system의 로컬라이즈 된 서기법에 따르는 텍스트는, 어플리케이션간에 전송 할 수 있습니다.

Windows 2000, XP, 및 2003 의 경우, 모든 Unicode 캐릭터 세트를 사용한 텍스트를 어플리케이션간에 전송 할 수 있습니다.

Solaris 및 Linux 의 경우, 호스트 operating system의 로케일의 문자 인코딩에 의한 텍스트를 어플리케이션간에 전송 할 수 있습니다.

호스트 operating system로부터 독립한 임의의 텍스트를 전송 할 필요가 있는 어플리케이션에서는, 직렬화를 사용해 텍스트를 전송 합니다. 거기에는, DataFlavor.stringFlavor 후레이바만을 지원하는 Transferable 를 작성합니다. 이 후레이바는, String 의 직렬화 표현을 나타냅니다. 전송처에서도 stringFlavor 가 지원되고 있는 것을 확인해 주세요. 전송시에, AWT 는 전송원으로 String 를 직렬화해, 전송처에서 직렬화를 해제합니다. 이 동작은 네이티브 플랫폼의 텍스트 전송보다 꽤 저속입니다만, 네이티브의 전송을 이용할 수 없는 경우에 유효합니다.
