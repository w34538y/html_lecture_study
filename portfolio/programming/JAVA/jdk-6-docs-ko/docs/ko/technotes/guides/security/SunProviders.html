처음에

Java 플랫폼에서는, 암호화, 공개키 인프라스트럭쳐(infrastructure), 인증, 안전한 통신, 액세스 제어 등, 주요한 시큐리티 분야에 건너는 일련의 API 가 정의되고 있습니다. 이러한 API 에 의해, 개발자는 어플리케이션 코드에 시큐리티 기구를 간단하게 통합할 수 있습니다. Java 암호화 아키텍쳐(architecture) (JCA) 및 그프로바이더 아키텍쳐(architecture)는,Java Development Kit (JDK)의 코어 개념입니다. 이 아키텍쳐(architecture)에 대해 충분히 이해하고 있는 것이 전제가 됩니다.

이 문서에서는, Sun 의 Java 환경의 일부로서 동고 되고 있는 프로바이더의 기술적인 상세한 것에 대하여 설명합니다.

주: Sun JDK 에서의 암호화 구현은, 역사적인 이유 및 제공하는 서비스의 종류에 의해, 다양한 프로바이더에 의해 배포되고 있습니다 ( 「Sun」, 「SunJSSE」, 「SunJCE」, 「SunRsaSign」). 범용 어플리케이션은, 특정의 프로바이더로부터 암호화 서비스를 요구하지 않게 할 필요가 있습니다. 예를 들어, 다음과 같이 지정할 수 있습니다. 

    getInstance("...", "SunJCE");  // not recommended
	vs.
    getInstance("...");            // recommended

후자와 같이 하지 않으면 어플리케이션은 특정의 프로바이더에 연결시킬 수 있게 되어, 그 프로바이더는 다른 Java 구현으로 사용할 수 없을 가능성이 있습니다. 특정의 요구된 프로바이더보다 우선 순위가 높은, 사용 가능한 최적화된 프로바이더 (예를 들어, PKCS11 에 의한 하드웨어 가속기 또는 Microsoft 의 MSCAPI 등의 네이티브 OS 구현)를 이용할 수 없는 경우도 있습니다.

SunPKCS11 프로바이더

암호화 토큰 인터페이스 표준 (PKCS#11 )은, 하드웨어 암호화 가속기나 스마트 카드등의 암호화 기구에 대한 네이티브 프로그래밍 인터페이스를 제공하고 있습니다. 적절히 구성하면(자),SunPKCS11 프로바이더에 의해, 어플리케이션은 표준의 JCA/JCE API 를 사용해 네이티브 PKCS#11 라이브러리에 액세스 할 수 있게 됩니다. SunPKCS11 프로바이더 자체에는 암호화 기능은 포함되어 있지 않습니다. 이 프로바이더는, Java 환경과 네이티브 PKCS11 프로바이더간의 안녕 제트로서 기능합니다. 이 프로바이더의 자세한 것은,「Java PKCS#11 레퍼런스 가이드」를 참조해 주세요.

SUN 프로바이더

JDK 1.1 에서는 Provider 아키텍쳐(architecture)가 도입되었습니다. 최초의 JDK 프로바이더에는 SUN 라는 이름이 붙일 수 있어 2 개의 암호화 서비스 (MessageDigest 및 Signature)가 포함되어 있었습니다. 후속의 릴리스에서는, 다른 기구도 추가되었습니다 (SecureRandom 난수 제네레이터,KeyPairGenerator, KeyFactory 등).

당시 유효했던 미국의 수출 제한 사항에 의해, JDK 의 내부에서 사용 가능한 암호화 기능에 중대한 제한이 부과되었습니다. 이 때문에,SUN 프로바이더에는, 데이터를 직접 암호화 또는 복호화 하지 않는 암호화 엔진이 포함되어 있습니다.

SunJSSE 프로바이더

Java Secure Socket Extension (JSSE)는, 개별의 「옵션 패키지」( 「표준 확장 기능」이라고도 불리고 있습니다)로서 릴리스 되어 JDK 1.2.x 및 1.3.x 로 사용할 수 있었습니다. SunJSSE 프로바이더는 이 릴리스의 일부로서 도입되었습니다.

이전의 JDK 릴리스에서는, JDK 로 RSA 서명 프로바이더는 사용할 수 없었습니다. 이 때문에,SunJSSE 에서는, 공통으로 사용 가능한 RSA 베이스의 증명서를 사용하기 위해서 독자적인 RSA 구현을 제공할 필요가 있었습니다. JDK 5 에서는 SunRsaSign 프로바이더가 도입되었습니다. 이 프로바이더는,SunJSSE 프로바이더의 모든 기능 ( 및 그 이상의 기능)을 제공합니다. JDK 5.0 이후를 타겟으로 하고 있는 어플리케이션은, 대신에 SunRsaSign 프로바이더의 인스턴스를 요구할 필요가 있습니다. 하위 호환성을 위해서(때문에), 이 프로바이더로 RSA 알고리즘을 아직 사용할 수 있습니다만, 이 알고리즘은 실제로는 SunRsaSign 프로바이더에 구현되고 있습니다.

AES_256 를 사용하는 암호군에서는, 강도가 무제한의 JCE 관할 정책 파일이 인스톨 되고 있을 필요가 있습니다. 「Java SE 다운로드 페이지」를 참조해 주세요.

Elliptic Curve Cryptography (ECDSA, ECDH, ECDHE, ECDH_anon)를 사용하는 암호군에서는, 다음의 프로퍼티을 가지는 JCE 암호화 프로바이더가 인스톨 되고 있을 필요가 있습니다.

  • 패키지 java.security.spec 및 java.security.interfaces 내의 클래스와 인터페이스로 정의되고 있도록(듯이) ECC 를 구현합니다. Elliptic Curve 열쇠 객체의 getAlgorithm() 메소드는 캐릭터 라인 「EC」를 돌려줄 필요가 있습니다.

  • Signature 알고리즘의 SHA1withECDSA 와 NONEwithECDSA, KeyAgreement 알고리즘의 ECDH, 및 알고리즘 EC 의 KeyPairGenerator 와 KeyFactory 를 지원합니다. 이러한 알고리즘중 한쪽이 결핍 하고 있는 경우, SunJSSE 에서는 EC 암호군의 사용이 허가되지 않습니다.

  • 암호화 프로바이더는,RFC 4492 스펙의 섹션 5.1. 1 ( 「부록 A」도 참조)으로 설명 떠날 수 있어 모든 SECG 곡선을 지원할 필요가 있습니다. 증명서에서는, 압축되어 있지 않은 형식을 사용해 포인트를 encode 해,namedCurve 선택을 사용해 (즉 객체 식별자를 사용해) 곡선을 encode 할 필요가 있습니다. 이러한 요건이 채워지지 않은 경우, EC 암호군이 올바르고 협상 되지 않는 것이 있습니다.

SunJCE 프로바이더

「SUN 프로바이더」로 간단하게 설명되고 있도록(듯이), 당시의 미국의 수출 제한 사항에 의해,JDK 내에서 사용 가능한 암호화 기능이 제한되고 있었습니다. 이 때문에, 어플리케이션으로 데이터의 암호화 또는 복호화를 가능하게 하기 위해서, 개별의 API 및 레퍼런스 구현이 개발되었습니다. Java 암호화 확장 기능 (JCE)은, 개별의 「옵션 패키지」( 「표준 확장 기능」이라고도 불리고 있습니다)로서 릴리스 되어 JDK 1.2.x 및 1.3.x 로 사용할 수 있었습니다. JDK 1.4 의 개발시에 제한이 완화되었기 때문에, JCE ( 및 SunJSSE)는 JDK 의 일부로서 번들 할 수 있게 되었습니다.

알고리즘의 이름 디폴트 키 사이즈 제한/코멘트
AES 128 키 사이즈는 128, 192, 또는 256 으로 할 필요가 있습니다.
ARCFOUR (RC4) 128 키 사이즈는, 40 에서 1024 의 범위내일 필요가 있습니다.
Blowfish 128 키 사이즈는, 32 에서 448 의 범위에서, 8 의 배수일 필요가 있습니다.

키 사이즈가 112 의 경우에는 2 개의 중간열쇠를 보관 유지하는 트리플 DES 열쇠가, 168 의 경우에는 3 개의 중간열쇠를 보관 유지하는 트리플 DES 열쇠가, 각각 생성됩니다.

「Meet-In-The-Middle」문제이기 때문에, 112 비트 또는 168 비트의 열쇠 데이터가 사용됩니다만,유효한키 사이즈는 각각 80 비트 또는 112 비트입니다.

SunPCSC 프로바이더

SunPCSC 프로바이더를 사용하면(자), 어플리케이션은 Java 스마트 카드 I/O API 를 사용해, 기초가 되는 operating system의 PC/SC 스마트 카드 스택과 서로 작용할 수가 있습니다. 일부의 operating system에서는, PC/SC 스택을 사용하기 전에 이것을 유효하게 해 구성할 필요가 있는 일이 있습니다. 자세한 것은, operating system의 문서를 참조해 주세요.

Solaris 및 Linux 플랫폼에서는, SunPCSC 는 libpcsclite.so 라이브러리를 사용해 PC/SC 스택에 액세스 합니다. SunPCSC 는, 디렉토리 /usr/$LIBISA 및 /usr/local/$LIBISA 내에서 이 라이브러리를 검색합니다. $LIBISA 는, 32 비트의 플랫폼에서는 lib, 64 비트의 Solaris 플랫폼에서는 lib/64, 64 비트의 Linux 플랫폼에서는 lib64 에 각각 전개됩니다. 시스템 프로퍼티 sun.security.smartcardio.library 가, 대체의 libpcsclite.so 구현의 완전한 파일명으로 설정되는 경우도 있습니다. Windows 플랫폼에서는, SunPCSC 는 항상 winscard.dll 를 호출하기 (위해)때문에, Java 레벨의 구성은 불필요하고, 실시할 수가 없습니다.

호스트 플랫폼에서 PC/SC 가 사용 가능한 경우, SunPCSC 구현은 TerminalFactory.getDefault() 및 TerminalFactory.getInstance("PC/SC") 를 개입시켜 취득할 수 있습니다. PC/SC 가 사용 가능하지 않은지, 또는 올바르게 구성되어 있지 않은 경우는,NoSuchAlgorithmException 와 함께 getInstance() 호출이 실패해,getDefault() 는 단말을 지원하지 않는 JRE 편입 구현을 돌려줍니다.
