JNLP 파일은 XML 문서입니다. 다음에, JNLP 파일의 완전한 기술예를 나타냅니다.

례

<? xml version="1. 0" encoding="utf-8"? >
<! -- JNLP File for SwingSet2 Demo Application -->
<jnlp
  spec="6. 0+"
  codebase="http://my_company.com/jaws/apps"
  href="swingset2.jnlp">
  <information>
    <title>SwingSet2 Demo Application</title>
    <vendor>Sun Microsystems, Inc. </vendor>
    <homepage href="docs/help.html"/>
    <description>SwingSet2 Demo Application</description>
    <description kind="short">A demo of the capabilities 
    of the Swing Graphical User Interface. </description>
    <icon href="images/swingset2.jpg"/>
    <icon kind="splash" href="images/splash.gif"/>
    <offline-allowed/> 
    <association>
      <mime-type="application-x/swingset2-file"/>
      <extensions="swingset2"/> 
    </association>
    <shortcut online="false">
      <desktop/>
      <menu submenu="My Corporation Apps"/>
    </shortcut>
  </information>
  <information os="linux">   
    <title> SwingSet2 Demo on Linux </title>
    <homepage href="docs/linuxhelp.html">
  </information>
  <security>
      <all-permissions/>
  </security>
  <resources>
    <j2se version="1. 6+" java-vm-args="-esa -Xnoclassgc"/>
    <jar href="lib/SwingSet2.jar"/>
  </resources>
  <application-desc main-class="SwingSet2"/>
</jnlp>

이 예는, 문서의 기본 구조를 나타낸 것입니다. 루트 요소는 jnlp 입니다만, 이 요소에는, 4 개의 서브 요소 information,security,resources,application-desc 를 정의할 수 있습니다. 게다가 Java Web Start 는 applet-desc 요소를 사용하는 것으로, 애플릿의 기동도 지원합니다. 요소의 자세한 것은 다음과 같습니다.

JNLP 의 요소

jnlp 요소

spec 속성:이 속성이 1.0 이상이 아니면 이 릴리스로 올바르게 동작하지 않습니다. 디폴트 값는 「1.0+」입니다. 따라서, 일반적으로은 이 속성을 생략 해도 괜찮습니다. 이 버젼은, 버젼 1.0, 1.5, 및 6.0 의 스펙을 지원하고 있습니다만, 이전의 버젼에서는 1.0 으로 1.5 밖에 지원되고 있지 않습니다.  spec="6.0+" 를 지정하는 jnlp 파일은, 이 버젼으로 동작합니다만, 이전의 버젼의 Java Web Start 에서는 동작하지 않습니다.

codebase 속성:JNLP 파일의 href 속성으로 지정된 상대 URL 는 모두, 이 URL 를 기준으로서 처리됩니다.

href 속성:이것은, JNLP 파일 자체의 포함 장소를 나타내는 URL 입니다.

information 요소

title 요소:어플리케이션의 이름.

vendor 요소:어플리케이션 벤더의 이름.

homepage 요소:단일의 속성 href 를 포함합니다. 이것은, 어플리케이션의 홈 페이지를 나타내는 URL 입니다. 이 요소는, Java 어플리케이션 캐쉬 viewer가, 어플리케이션의 상세 정보를 포함한 Web 페이지에 사용자를 리다이렉트(redirect) 할 때에 사용됩니다.

description 요소:어플리케이션에 관한 짧은 문장. description 요소는 생략 가능합니다. kind 속성은, description 의 용도를 정의합니다. 지정 가능한 값은, 다음의 머지않아인가입니다.

  • one-line:어플리케이션에의 참조를 리스트 또는 겉(표)의 1 행에 표시하는 경우에, 이 description 가 사용됩니다.
  • short:어플리케이션에의 참조를 표시할 때에 1 단락 분의 스페이스가 잡히는 경우에, 이 description 가 사용됩니다.
  • tooltip:어플리케이션에의 참조를 툴팁내에 표시하는 경우에, 이 description 가 사용됩니다.

개개의 종류에 대해서 지정 가능한 description 요소는, 1 개(살) 뿐입니다. kind 속성을 포함하지 않는 description 요소는, 디폴트 값로서 사용됩니다. 따라서, Java Web Start 가 short 의 종류의 description 요소를 필요로 하고 있지만, 그것이 JNLP 파일내로 지정되어 있지 않았던 경우, 속성을 포함하지 않는 description 요소의 텍스트가 사용됩니다.

description 요소에 포함되는 텍스트는, plaintext가 아니면 안됩니다. HTML 태그등을 포함한 형식은 지원되고 있지 않습니다.

icon 요소:GIF 형식 또는 JPEG 형식의 이미지 파일에의 HTTP URL 를 포함합니다. 이러한 아이콘은, 다음의 경우에 어플리케이션을 표현하기 위해서 사용됩니다.

  • 어플리케이션 기동시에 Java Web Start 가 어플리케이션을 사용자에게 표시할 때
  • Java 어플리케이션 캐쉬 viewer내
  • 데스크탑 쇼트 컷내

다운로드중은 64x64 의 아이콘이 표시되어 Java 어플리케이션 캐쉬 viewer내와 데스크탑 쇼트 컷내에서는 32x32 의 아이콘이 사용됩니다. Java Web Start 는, 아이콘의 사이즈 변경을 필요에 따라서 자동적으로 실시합니다.

생략 가능한 속성 width 와 height 를 사용하면, 이미지의 사이즈를 지정할 수 있습니다.

생략 가능한 속성 kind="splash" 를 icon 요소내로 지정하면, 그 이미지가 어플리케이션 기동시의 「스플래쉬」화면용이라고 가리킬 수가 있습니다. JNLP 파일에 kind="splash" 속성을 가지는 아이콘 요소가 포함되지 않은 경우, Java Web Start 는 information 요소의 다른 항목을 사용해 스플래쉬 화면을 구축합니다.

JNLP 파일내에 icon 이미지가 1 개나 존재하지 않았던 경우, 스플래쉬 이미지는, JNLP 파일로부터 취득된 어플리케이션의 타이틀과 벤더로 구성됩니다.

JNLP 파일의 icon 요소를 추가 또는 변경한 뒤 처음으로 어플리케이션을 기동하려면 , 이전의 스플래쉬 이미지가 표시됩니다. 2 번째 이후의 어플리케이션 기동에서는, 새로운 스플래쉬 이미지가 표시됩니다.

offline-allowed 요소:생략 가능한 offline-allowed 요소는, 오프 라인에서의 어플리케이션 기동을 허가할지 어떨지를 나타냅니다.

offline-allowed 가 지정되고 있는 경우, Java 어플리케이션 캐쉬 viewer로 어플리케이션을 오프 라인으로 기동할 수 있습니다. 또 어플리케이션을 오프 라인으로 기동하는 쇼트 컷을 작성할 수 있습니다.

어플리케이션을 오프 라인으로 기동했을 경우, 갱신의 체크는 행해지지 않습니다. 또, API 호출 BasicService.isOffline() 는 true 를 돌려줍니다.

또,offline-allowed 요소는, Java Web Start 가 어플리케이션의 갱신을 체크하는 방법도 제어합니다. 이 요소가 지정되어 있지 않은 경우 (즉, 어플리케이션을 온라인으로 실행할 필요가 있는 경우), Java Web Start 는 항상, 갱신판이 있을지 어떨지를 체크한 후에 어플리케이션을 기동합니다. 갱신판이 발견되었을 경우는, 그 새로운 어플리케이션을 다운로드 및 기동합니다. 따라서, 사용자가 실행하는 어플리케이션이 항상 최신판인 것이 보증됩니다. 다만, 오프 라인의 경우, 어플리케이션을 실행할 수 없습니다.

offline-allowed 가 지정되고 있는 경우도, Java Web Start 는 갱신판이 있을지 어떨지를 체크합니다. 다만, 어플리케이션이 다운로드가 끝난 상태였을 경우, 그 체크 처리는 몇 초로 타임 아웃이 됩니다. 타임 아웃이 되었을 경우, 캐쉬 끝난 어플리케이션이 대신에 기동됩니다. 서버와의 접속이 어느 정도 고속으로 있으면, 대부분은 최신판의 어플리케이션이 실행될 것입니다만, 보증은 없습니다. 다만, 어플리케이션은 오프 라인으로 실행할 수 있습니다.

shortcut 요소:생략 가능한 shortcut 요소를 사용하면(자), 어플리케이션의 데스크탑의 통합에 대한 설정 방법법을 나타낼 수가 있습니다. shortcut 요소와 그 서브 요소는, JNLP Client 가 사용할 가능성이 있는 힌트를 제공합니다. shortcut 요소에는 생략 가능한 online 속성과 2 개의 서브 요소 desktop,menu 가 포함됩니다.

association 요소:생략 가능한 association 요소는, operating system에 특정의 확장자(extension)의 primary 핸들러로서 또 특정의 MIME 타입으로서의 등록을 실시하고 싶은 JNLP 클라이언트에의 힌트입니다. association 요소에는 반드시 확장자(extension)와 MIME 타입 속성을 할당할 수 있습니다.

related-content 요소:생략 가능한 related-content 요소는, README 파일, 헬프 페이지, 등록 페이지로의 링크등의 관련 컨텐츠의 추가 요소를, JNLP Client 에의 힌트로서 설명합니다. 어플리케이션은, 이 컨텐츠를 데스크탑 통합에 포함하도록(듯이) 요구합니다. related-content 요소에는, href 와 title 의 필수 속성을 할당할 수 있고 있습니다. 다음의 2 개의 서브 요소의 어느쪽이든을 할당할 수도 있습니다.

  • description 요소:관련 컨텐츠의 간단한 설명
  • icon 요소:JNLP Client 가 사용자에게로의 관련 컨텐츠를 식별하기 위해서 사용하는 아이콘

security 요소

각 어플리케이션은 디폴트로, 애플릿의 샌드 박스에 유사한, 제한된 런타임내에서 실행됩니다.  security 요소를 사용하는 것으로, 무제한의 액세스권을 요구할 수 있습니다.

all-permissions 요소가 지정되었을 경우, 어플리케이션은 클라이언트 머신과 로컬 네트워크에 대한 완전한 액세스권을 가집니다. 완전한 액세스권을 요구하는 어플리케이션에서는, 모든 JAR 파일에 서명할 필요가 있습니다. 어플리케이션의 첫회 기동시에, 사용자는 증명서에의 동의가 요구됩니다.

update 요소

update 요소는, Java Web Start 가 어플리케이션의 갱신을 어떻게 처리하는지를 지정하는데 사용됩니다.

update 요소에는, 다음에 나타내는 2 개의 속성을 옵션으로 포함할 수가 있습니다.

check 속성:check 속성은, JNLP 클라이언트가 갱신을 확인할 필요가 있는 시기에 대한 설정을 나타냅니다. 값에는, 「always」, 「timeout」, 「background」의 3 개 가운데, 어느쪽이든을 사용할 수 있습니다.

「always」의 값은, 어플리케이션을 기동하기 전에 항상 갱신 확인을 실시하는 것을 의미합니다.

「timeout」의 값 (디폴트)은, 어플리케이션을 기동하기 전에, 타임 아웃이 될 때까지 갱신 확인을 실시하는 것을 의미합니다. 갱신 확인이 완료하지 않고 타임 아웃 했을 경우, 어플리케이션의 기동 후에, 갱신 확인이 그대로 백그라운드에서 계속해 행해집니다.

「background」의 값은, 백그라운드에서의 갱신 확인의 실행중에 어플리케이션을 기동하는 것을 의미합니다.

policy 속성:policy 속성은, 어플리케이션의 기동전에 갱신이 가능하다라고 판단되었을 때, JNLP 클라이언트가 어플리케이션 갱신에 실시하는 처리 방법의 설정을 나타냅니다. 값에는, 「always」, 「prompt-update」, 「prompt-run」의 3 개 가운데, 어느쪽이든을 사용할 수 있습니다.

「always」의 값 (디폴트)은, 사용자에게 통지하는 일 없이, 항상 갱신을 다운로드하는 것을 의미합니다.

「prompt-update」의 값을 사용하면(자), 사용자에게 갱신 버젼을 다운로드해 실행할지 어떨지, 또는 캐쉬된 버젼을 기동할지 어떨지를 묻습니다.

「prompt-run」의 값을 사용하면(자), 사용자에게 갱신 버젼을 다운로드해 실행할지 어떨지, 또는 캔슬해 어플리케이션의 실행을 중지할지 어떨지를 묻습니다.

예를 나타냅니다.

<update check="always" policy="prompt-update">

resources 요소

resources 요소는, 어플리케이션에 포함되는 Java 클래스 파일, 네이티브 라이브러리, 시스템 프로퍼티 등, 모든 자원을 지정하기 위해서 사용합니다. 자원의 정의는, 특정의 페레-팅 시스템 (os 속성), 아키텍쳐(architecture) (arch 속성), 로케일 (locale 속성)로 한정하는 것이 가능합니다.

거기에는,os,arch,locale 의 각 속성을 사용합니다. resources 요소로 지정 가능한 서브 요소는,jar,nativelib,j2se,property,package,extension 의 6 개입니다. 이 개발자 가이드에서는,package 요소와 extension 요소에 대해서는 설명하지 않습니다. 상세한 것에 대하여는,「Java Network Launching Protocol & API Specification (JSR-56) version 6.0」을 참조해 주세요.

jar 요소에는, 어플리케이션의 클래스 패스에 포함하는 JAR 파일을 지정합니다.  예를 나타냅니다.

  <jar href="myjar.jar"/>

ClassLoader 객체를 사용하면(자),jar 파일이 JVM 에 로드 됩니다.  일반적으로,jar 파일에는, 특정의 어플리케이션의 코드를 포함한 Java 클래스가 포함됩니다만,getResource 기구로 사용할 수 있는 아이콘이나 구성 파일과 같은 다른 자원도 포함되어 있습니다.

nativelib 요소는, 네이티브 라이브러리를 포함한 JAR 파일을 지정합니다.  예를 나타냅니다.

    <nativelib href="lib/windows/corelib.jar"/>

JNLP 클라이언트는,System.loadLibrary 메소드를 사용해, JAR 파일의 루트 디렉토리 (/)에 있는 각 파일 엔트리가 실행중의 프로세스에 확실히 로드할 수 있도록(듯이) 할 필요가 있습니다.   각 엔트리에는, 플랫폼 고유의 공유 라이브러리를 올바른 명명 규약을 사용해 포함할 필요가 있습니다 (Windows 의 *. dll, Solaris 나 Linux 의 lib*.so 등). System.loadLibrary 에의 호출을 실제로 실시하는 것은, 어플리케이션의 역할입니다.

일반적으로, 네이티브 라이브러리는, 특정의 operating system과 아키텍쳐(architecture)에 대응한 resources 요소에 포함됩니다.  예를 나타냅니다.

    <resources os="SunOS" arch="sparc">
        <nativelib href="lib/solaris/corelibs.jar"/>
    </resource>

디폴트에서는,jar 와 nativelib 자원이 정기적으로 다운로드됩니다. 이 2 개(살)은 다운로드된 뒤, 어플리케이션이 기동되기 전에, 어플리케이션을 실행하고 있는 JVM 가 로컬에 사용합니다.  jar 와 nativelib 요소에서는, 자원을 lazy 와 지정할 수도 있습니다.  이 설정으로 하면(자), 어플리케이션의 기동전에 자원을 클라이언트 시스템에 다운로드할 필요가 없어집니다.

download 요소는, 자원을 정기적으로 다운로드하는지, 부정기에 다운로드할까의 제어에 사용합니다.  예를 나타냅니다.

    <jar href="sound.jar" download="lazy"/>
    <nativelib href="native-sound.jar" download="eager"/>

j2se 요소는, Java 가상 머신의 표준 파라미터와 어플리케이션이 지원되는 Java(TM) Platform Standard Edition Runtime Environment (JRE) 버젼을 지정합니다.  복수의 JRE 를 지정하는 경우, 지원 대상의 JRE 를 우선순서에 리스트에 나타냅니다. 무엇보다 바람직한 버젼을 최초로 선택합니다.  예를 나타냅니다.

       <j2se version="1.3" initial-heap-size="64m" max-heap-size="128m"/>
       <j2se version="1.4. 2+" href="http://java.sun.com/products/autodl/j2se" java-vm-args="-esa -Xnoclassgc"/>

version 속성은, 디폴트로 Java(TM) Platform Standard Edition 플랫폼의 플랫폼 버젼 (스펙 버젼)을 참조합니다. 현시점에서 정의되고 있는 플랫폼 버젼은, 1.2, 1.3, 1.4, 1.5 및 1.6 입니다 (일반적으로, 플랫폼 버젼에는, 마이크로 버젼 번호(1.4. 2 등)은 포함되지 않는다).

정확한 제품 버젼 (구현 버젼)을,href 속성을 사용해 지정할 수도 있습니다. 예를 들어, 1.3. 1_07, 1.4. 2, 또는 1.5. 0-beta2 와 같이 됩니다. 다음과 같이 지정합니다.

<j2se version="1.4. 2" href="http://java.sun.com/products/autodl/j2se"/

또는

<j2se version="1.4. 2_04" href="http://java.sun.com/products/autodl/j2se"/>

플랫폼 버젼이 지정되었을 경우 (즉,href 속성이 지정되지 않았던 경우), Java Web Start 는, 인스톨 끝난 비 FCS (마일 스톤)의 JRE 를, 요구에 일치하는 것으로 간주하지 않습니다. 다음에 예를 나타냅니다.

<j2se version="1.4+"/>

이 경우, 1.4. 1-ea 나 1.4. 2-beta 의 JRE 가 인스톨 되고 있어도, 그것들은 요구에 일치하는 JRE 라고는 보이지 않습니다. Sun Microsystems, Inc. 하지만 제공하는 JRE 에서는, 1.3. 0 이후, 비 FCS (마일 스톤)의 버젼 캐릭터 라인에는 데쉬 (-)를 포함하게 되어 있습니다.

j2se 요소의 java-vm-args 속성은, Java 의 기동시에 우선적으로 사용되는 가상 머신의 일련의 인수를 지정합니다.

<j2se version="1.4+" java-vm-args="-ea -Xincgc"/>

이 버젼에서는, 다음의 java-vm-args 가 지원됩니다. 

 -d32,							        /* use a 32-bit data model if available */
 -client,                      				        /* to select the client VM */

application-desc 요소

application 요소는, JNLP 파일의 기동 대상이, 애플릿은 아니고 어플리케이션인 것을 나타냅니다. application-desc 요소에는 main-class 속성을 지정할 수 있습니다. 생략 가능한 이 속성을 사용하면, 어플리케이션의 메인 클래스 (최초로 실행해야 할 public static void main(String argv[]) 메소드를 포함하고 있는 클래스)의 이름을 지정할 수 있습니다.

main-class 속성을 생략 할 수 있는 것은, JNLP 파일내로 지정된 최초의 JAR 파일에 main 클래스를 포함한 Manifest 파일이 포함되고 있는 경우입니다.

어플리케이션에의 인수를 지정하려면 , 1 개(살) 또는 복수의 상자로 된 argument 요소를 포함합니다. 예를 나타냅니다.

  <application-desc main-class="Main">
    <argument>arg1</argument>
    <argument>arg2</argument>
  </application-desc>

applet-desc 요소

Java Web Start 는, Java 애플릿의 기동을 지원합니다. 이 지원에 의해, 기존 코드의 Java Web Start 에의 이행이 용이하게 됩니다.

애플릿을 기동하려면 ,application-desc 요소 대신에 applet-desc 요소를 사용합니다. 예를 나타냅니다.

  <applet-desc
      documentBase="http://..."
      name="TimePilot"
      main-class="TimePilot.TimePilotApp"
      width="527"
      height="428">
    <param name="key1" value="value1"/>
    <param name="key2" value="value2"/>
  </applet-desc>

애플릿을 구성하는 JAR 파일은, 어플리케이션의 경우와 같이,resources 요소를 사용해 기술합니다. documentBase 를 명시적으로 지정할 필요가 있는 것은, JNLP 파일이 HTML 페이지에 파묻히지 않기 때문입니다. 그 외의 속성은 각각, 특정의 HTML 애플릿 태그 요소에 대응합니다. code 속성 대신에 main-class 속성이 사용됩니다.  

main-class 속성은,code 속성 대신에 사용합니다.  main-class 속성에는 Applet 클래스의 이름 (. class 의 확장자(extension)는 제외하다)를 지정합니다.  메인의 JAR 파일내의 Main-Class Manifest 엔트리로부터 Applet 클래스가 검색 가능한 경우, 이 속성을 생략 해도 괜찮습니다.

주:애플릿은, JAR 파일내에 패키지화하지 않으면 Java Web Start 로 올바르게 동작하지 않습니다.

component-desc 요소

component-desc 요소는, jnlp 파일이 어플리케이션 또는 애플릿은 아니고, 어플리케이션이나 애플릿, 다른 확장 기능의 자원으로서 사용할 수 있는 확장 기능인 것을 의미합니다.

일반적으로, 컴퍼넌트 확장 기능은, 복수의 어플리케이션간에 공유된다, 또는 개별의 시큐리티가 필요한 자원 세트를 없애기 위해서(때문에) 사용됩니다.

installer-desc 요소

installer-desc 요소는, jnlp 파일이 어플리케이션을 정의하는 인스톨러의 확장 기능인 것을 의미합니다. 이 확장 기능의 jnlp 파일은, 어플리케이션이나 애플릿, 다른 확장 기능으로 최초로 한 번 사용되면(자), 그 후는 실행되지 않습니다.

일반적으로, 인스톨러의 확장 기능은, 네이티브 라이브러리를 VM 에 단순하게 로드하는 이상으로 복잡한 설정을 필요로 하는, 플랫폼 고유의 native code를 인스톨 하기 위해서 사용됩니다.
