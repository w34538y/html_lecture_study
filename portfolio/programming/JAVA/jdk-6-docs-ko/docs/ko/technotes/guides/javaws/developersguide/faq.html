Java Web Start 란 무엇입니까 ?

Java Web Start 는, 플랫폼에 의존하지 않는 안전하고 강고한 배치용 테크놀러지를 제공합니다. 이것에 의해, 개발자는 풀 기능의 어플리케이션을 표준의 Web 서버로 사용할 수 있도록(듯이) 하는 것으로, 어플리케이션을 최종 사용자에 배치할 수 있습니다. Web 브라우저가 있으면, 최종 사용자는 어플리케이션을 기동하는 것만으로, 항상 최신의 버젼을 안심해 이용할 수 있습니다.

왜 Java Web Start 가 필요합니까 ?

어플리케이션을 Web 로부터 직접 배치하는 간단하고 강고하고 안전한 방법이기 때문입니다. 개발자는, 간단하게 어플리케이션을 Web 경유로 사용할 수 있도록(듯이) 할 수가 있습니다. 또, Java Web Start 에는 Java 실행 환경 (JRE) 관리 기능도 갖춰지고 있기 (위해)때문에, 설정이 간단해, 브라우저에 의존하지 않고, Web 어플리케이션의 솔루션을 효율적으로 배치할 수 있습니다.

사용자는, 인스톨 순서를 개별적으로 실시하지 않아도, 어플리케이션 (Web 페이지)에 간단하게 액세스 할 수 있습니다. 사용자는, 데스크탑으로부터, Web 페이지로 이용하는 것보다도 풍부로 곧바로 응답하는 사용자-인터페이스를 사용해, Java 어플리케이션에 액세스 해 사용할 수 있습니다. 또, Java Web Start 베이스의 어플리케이션은, 한 번 인스톨 하면(자), 필요한 때에 언제라도 클릭하는 것만으로 실행할 수 있습니다.

어플리케이션을 기동할 때마다, 사용자에게 의식시키지 않고 Web 로부터 어플리케이션이 갱신되기 (위해)때문에, 사용자는 수동으로 어플리케이션을 갱신할 필요가 없고, 항상 최신 버젼을 사용할 수 있습니다.

Java Web Start 를 사용해 어플리케이션을 기동하기에는 어떻게 하면 좋습니까 ?

우선, Web 페이지로부터 링크를 클릭해, 새로운 어플리케이션을 기동합니다.

어플리케이션을 빈번하게 사용하는 경우는, Java Web Start 로 데스크탑에 아이콘을 배치할 수 있도록(듯이) 해, 데스크탑 또는 [스타트] 메뉴로부터 쇼트 컷을 작성합니다. Java Web Start 로부터, 쇼트 컷을 작성하는지,[스타트]
메뉴에 항목을 작성할까를 묻는 일이 있습니다. 「네」를 클릭하면(자), 다음번부터 브라우저를 사용하지 말고 어플리케이션을 기동할 수 있습니다.

또, Java Web Start 에는, Java 컨트롤 패널로부터 기동할 수 있는 어플리케이션 캐쉬 viewer가 준비되어 있습니다.  어플리케이션 캐쉬 viewer를 사용하면(자), 다운로드한 어플리케이션을 직접 기동할 수 있습니다.

게다가 커멘드 prompt로 “javaws <jnlp_url>”라고 입력해, 어플리케이션을 기동할 수도 있습니다. 여기서 < 는 어플리케이션의 JNLP 파일에의 URL 입니다.

어플리케이션의 기동 방법에는 의미가 있습니까 ?

아니오. 어떠한 방법을 사용해도, 어플리케이션은 똑같이 기동합니다. 즉, Web 페이지, 데스크탑상의 쇼트 컷,[스타트] 메뉴, 또는 Java 어플리케이션 캐쉬 viewer의 어느 방법으로 기동해도 차이는 없습니다.

Java Web Start 는, 반드시 최신 버젼의 어플리케이션을 사용할 수 있을지 어떨지를 확인해, 사용할 수 있는 경우는 자동적으로 다운로드합니다.

사용하고 있는 어플리케이션이 디지털 서명되어 있지 않은 경우, Java Web Start 는 그 어플리케이션이 제한된 안전한 실행 환경에서 기동합니다. 서명이 없는 어플리케이션이나 신뢰할 수 없는 어플리케이션이, 로컬 시스템이나 네트워크에 대한 무제한의 액세스권으로 실행될 것은 없습니다.

Java Web Start 의 서버 요건은 무엇입니까 ?

Java Web Start 에서는, 클라이언트와 서버와의 통신에 HTTP 를 사용합니다. 표준의 Web 서버를 어플리케이션의 호스트로서 사용할 수 있습니다. 그 외의 서비스 (버젼 베이스의 다운로드, 증분 갱신, 어플리케이션의 pack200 압축등)가 필요한 경우는, Web 서버로 서브 렛이나 Java Server Pages 를 지원할 필요가 있습니다. 이러한 기능을 구현하는 서브 렛의 예가 JDK 의 샘플용의 디렉토리에 넣을 수 있고 있습니다.

Java Web Start 는 표준에 근거하고 있습니까 ?

할 수 있습니다. Java Web Start 1.5. 0 은, Java Community Process 를 통해서 개발된 Java Network Launching Protocol (JNLP) 테크놀러지의 제품 품질의 레퍼런스 구현입니다.  JNLP 는, 스펙 번호 JSR 056 입니다.  다른 플랫폼 벤더는, Java Web Start 를 우리들의 플랫폼에 이식하거나 그 스펙을 구현하거나 할 수 있습니다.

Manifest 파일의 Class-Path 를 신뢰할 수 있습니까 ?

Java Web Start 에서는, Manifest 파일의 Class-Path 엔트리를 지원하고 있습니다. Class-Path 속성은 완전하게 파일 중심인데 대해, Java Web Start 와 JNLP 는 Web 중심 (URL 베이스)입니다. 이 때문에, 2 개의 모델을 1 개로 정리하는 것은 간단하지는 않습니다.

Class-Path 엔트리에 의지하는 것이 아니라, JNLP 파일에 포함되는 복수의 JAR 파일의 일람을 표시할 수 있습니다. 다음에 예를 나타냅니다. 

<resources>
 <jar href="A.jar"/>
 <jar href="B.jar"/>
</resources>

JNLP 파일에서는,<extension... > 요소를 사용해, 1조의 JAR 파일과의 의존관계(dependencies)를 다른 JNLP 파일로 나눌 수가 있습니다. 이 때문에, Class-Path 엔트리를 사용했을 경우와 같게 재사용하기 쉽고, 보수도 간단하게 실시할 수 있습니다. 이 기능에 대해서는, JNLP 의 스펙에 설명되고 있습니다.

JNLP 는, 애플릿과 같게, Just-in-Time 다운로드 기구도 구현합니다. JNLP 파일내의 자원 마다, 어느 부분의 다운로드를 곧바로 실시하는지, 나중에 실시하는지를 지정할 수 있습니다.  곧바로 로드하는 자원은 어플리케이션이 기동되기 전에 로드 되어 나중에 로드하는 자원은 당분간 지나고 나서 로드 됩니다.  디폴트 설정에서는 어플리케이션이 기동되기 전에 로드 됩니다. 디폴트에서는, 곧바로 다운로드됩니다. 게다가 스펙에는, 어느 자원을 이용할 수 있을까를 프로그램에 의해 Java Web Start 에 문의해 이용할 수 있는 자원을 다운로드하도록(듯이) 요구하기 위한 API 도 포함되어 있습니다. 따라서, 다운로드/네트워크 대응의 어플리케이션을 작성할 수 있습니다.

자세한 것은,「JNLP 스펙」을 참조해 주세요.

Class.forName 와 ClassLoader.getSystemClassLoader 를 사용해 어플리케이션 내부에 자원을 로드하기에는 어떻게 하면 좋습니까 ?

Java Web Start 에서는, 사용자 레벨의 클래스 로더를 사용해, JNLP 파일로 지정된 모든 어플리케이션 자원을 로드합니다. 이 클래스 로더에 의해, JNLP 스펙으로 정의된 시큐리티 모델과 다운로드 모델이 구현됩니다. 이것은, AppletViewer 나 Java Plug-In 의 동작과 서로 다릅니다.

다만, 이 클래스 로더에게는,Class.forName 가 JNLP 파일에 정의되고 있는 자원을 검색하지 않는다고 하는 유감인 부작용이 있습니다. 시스템 클래스 로더 (ClassLoader.getSystemClassLoader)를 사용해 자원이나 클래스를 검색하는 경우도 같습니다.

Java Web Start 로 어플리케이션 자원을 찾아내려면 , 어플리케이션을 로드한 클래스 로더를 사용합니다. 예를 들어, 어플리케이션의 메인 thread내에서 다음의 메소드를 호출합니다.

this.getClass(). getClassLoader();

다음의 메소드도 사용할 수 있습니다.

Thread.getCurrent(). getContextClassLoader();

어플리케이션을 일의에 식별하려면 어떻게 합니까 ?

어플리케이션은 일반적으로 그 href (그 JNLP 파일에의 URL)에 의해 식별됩니다. JNLP 파일에의 href 가 없는 경우, 어플리케이션은 그 jar 메인 파일에의 href 에 의해 식별됩니다. 복수의 어플리케이션이 같은 jar 메인 파일을 공유하고 있는 경우는, 그러한 어플리케이션에 href 를 1 개만 지정할 필요가 있습니다. 그렇게 하지 않으면 그것들은 같은 어플리케이션으로 간주해집니다.

버젼 1.5. 0 이상에서는, Java 어플리케이션 캐쉬 viewer에 모든 어플리케이션 (href 가 지정되어 있지 않은 어플리케이션도 포함한다)이 표시됩니다. 이 viewer에서는, href 가 지정되고 있는 어플리케이션만을 온라인으로 기동할 수 있습니다. href 가 지정되어 있지 않은 어플리케이션은, JNLP 파일의 허가가 있으면 (JNLP 파일에 <offline-allowed> 가 지정되고 있으면), 오프 라인으로 기동할 수 있습니다.

이전의 버젼에서는, 어플리케이션 매니저에게 href 가 지정되고 있는 어플리케이션만이 표시됩니다.

쇼트 컷을 작성하는 경우도 같습니다. 1.5. 0 에서는, href 가 지정되고 있는 경우에게만 온라인으로 기동하는 쇼트 컷을 작성할 수 있어 <offline-allowed> 가 지정되고 있는 경우에게만 오프 라인의 쇼트 컷을 작성할 수 있습니다. 1.4. 2 에서는, href 가 지정되고 있는 어플리케이션의 쇼트 컷만을 작성할 수 있습니다.

JNLP 어플리케이션에 관련짓는 복수의 파일 확장자(extension)나 MIME 타입을 작성하는 경우, 복수의 관련 태그를 사용하는 것을 추천합니다. 또, 파일 확장자(extension)와 MIME 타입의 양쪽 모두에 각 관련 태그를 지정할 필요가 있습니다. 예를 나타냅니다.

시큐리티에 관한 질문

Java Web Start 의 시큐리티 보호는 어떻게 행해지고 있습니까 ?

시큐리티는, Java Web Start 를 설계하는데 있어서 고려해야 할 중요한 포인트의 1 개입니다.  Java Web Start 에서는, 모든 어플리케이션이 디폴트로 안전한 「샌드 박스」로 기동됩니다.
탑에 돌아온다

안전한 샌드 박스 (sandbox)란 무엇입니까 ?

안전한 샌드 박스에 있는 어플리케이션은, 디스크나 네트워크등의 로컬의 컴퓨팅 자원에의 액세스가 제한됩니다.
탑에 돌아온다

Java Web Start 의 안전한 샌드 박스는 애플릿의 샌드 박스에 비해 어느 정도 유연성이 높습니까 ?

애플릿의 샌드 박스와 같게, 모든 어플리케이션은 디폴트에서는 샌드 박스 환경에서 실행됩니다. 다만, Java Web Start 에는 안전한 API 가 준비되어 있어 어플리케이션은 그것을 사용해 사용자의 제어하에 있는 로컬 디스크로부터 파일을 임포트 하거나 export 하거나 할 수 있습니다. 이 API 에는, 파일의 보존이나 파일의 오픈이라고 하는 조작용의 다이알로그 박스가 있습니다만, 그러한 다이알로그 박스는 실제로는 어플리케이션 그 자체는 아니고, Java Web Start 에 의해 draw 됩니다.

이 샌드 박스의 디자인은, HTML 를 사용해 작성할 수 있는 것과 닮아 있습니다. HTML 형식의 파일 입력 필드를 사용하면(자), 로컬 디스크로부터 파일을 꺼내, 이름 (패스를 제외하다)과 컨텐츠를 Web 서버에 송신할 수 있습니다.  같이 대부분의 브라우저에서는 「이름을 붙여 보존 (Save as...)」옵션을 지원하고 있습니다.
탑에 돌아온다

어플리케이션이 시스템에 대한 무제한의 액세스권을 취득할 수 있도록(듯이) 하기에는 어떻게 하면 좋습니까 ?

무제한의 시스템 액세스를 요구하는 어플리케이션에는, 디지털 서명이 필요합니다. 그러한 어플리케이션을 최초로 기동하면(자), 시큐리티용의 다이알로그 박스가 표시됩니다. Java Web Start 에서는, 어플리케이션을 실행하기 전에, 디지털 증명서에 동의 하도록(듯이) 사용자에게 요구합니다. 사용자가 증명서에 동의 하면(자), 그 후의 호출에서는 시큐리티용의 다이알로그 박스는 표시되지 않게 됩니다.
탑에 돌아온다

JAR 파일의 암호화된 다운로드는 지원되고 있습니까 ?

서버로부터 클라이언트에 전송 되는 데이터의 암호화를 지원하는 것은 매우 중요합니다. 이것은, 향후가 과제로 해 중요한 포인트의 1 개이며, Java Web Start 로 기동되는 어플리케이션으로 HTTPS 라이브러리를 확실히 사용할 수 있도록(듯이) 할 예정입니다. HTTPS 는 옵션 패키지로 지원되는 일이 있어, 그 경우는 어플리케이션이 의존하고 있는 것을 지정할 필요가 있습니다.

서버로부터 클라이언트에 전송 되는 JAR 파일의 암호화는 필요없을지도 모릅니다. 왜냐하면, 클래스 파일이 회선상에서 암호화되고 있는 경우에서도, 클라이언트측에서 복호화해 디스크에 포함할 필요가 있기 (위해)때문에, JVM 는 그러한 클래스를 로드할 수 있기 때문입니다. 따라서, 크래커가 JAR 파일의 암호화를 빠져나가는 것은 매우 용이한 것일 가능성이 있습니다.

그 대신에, JAR 파일에 있어 중요한 (일)것은, 파일을 서명 첨부로 해, 상정되고 있는 벤더로부터 어플리케이션이 확실히 입수되도록(듯이) 하는 것입니다. 이것은 벌써 지원되고 있습니다.

Java Web Start 로 NTLM 인증을 사용할 수 있습니까 ?

NTLM 의 지원는, Windows 플랫폼에서만 Java 1.4. 2 에 추가되었습니다. NTLM 인증은, Java Web Start 1.4. 2 이상으로 사용할 수 있습니다. Java Web Start 1.4. 2 로 NTLM 를 사용하고 있을 때, JAuthenticator 다이얼로그에 잘못한 텍스트가 표시되는 일이 있습니다만, 이것은 Java Web Start 1.5. 0 으로 수정되었습니다.
