Javadoc 툴

이 문서에서는, 버젼 1.4. 2 에서 5.0 (이전의 명칭은 1.5. 0)에의 이행으로 Javadoc 툴에 대해서 행해진 변경에 대해 설명합니다.

게재되고 있는 변경점에는, 버그 번호와 그 수정이 Javadoc 툴의 프론트엔드에 대하는 것 ( 「툴」)인지, 또는 표준 도크 렛에 대하는 것 ( 「표준 도크 렛」)일지가 나타나고 있습니다. 「리그렛션」이라고 하는 용어는, 1.3.x 에서는 동작하고 있었는데, 1.4.x 로 동작하지 않게 되어, 5.0 으로 수정된 기능에 대해서 사용됩니다.

주요한 신기능

  • 총칭, 열거형, 가변 인수: 이러한 새로운 언어 기능에 대한 지원를, javadoc 툴의 프론트엔드 (도크 렛 API) (4421066 , 툴)와 표준 도크 렛 (4789689 , 표준 도크 렛, 투표수 1)에 추가했습니다.
  • 주석: 주석 및 주석형에 대한 지원를, javadoc 툴의 프론트엔드 (도크 렛 API) (4904495 , 툴)과 표준 도크 렛 (4905985 , 표준 도크 렛)에 추가했습니다.
  • BreakIterator: 5.0 에서는, -breakiterator 의 경고 메세지를 제거했습니다. 또, 문장의 단락을 판별하는 디폴트 알고리즘은, 이전의 릴리스와 같습니다. 즉, -breakiterator 옵션은, 5.0 에서는 디폴트가 아니게 되었습니다. 또, 향후 디폴트로 할 예정도 없습니다. 이것은, 「다음의 메이저 릴리스」(5.0)로 디폴트를 변경할 예정이라고 하기 이전의 방침이 변경이 된 것을 의미합니다. 따라서, 1.4.x 로 breakiterator 의 경고의 표시를 없애기 위한 원시 코드 변경을 행하지 않은 경우는, 아무것도 조치를 취할 필요가 없습니다. 5.0 에서는 경고가 표시되지 않게 되기 때문입니다. 이러한 방침 전환의 이유는, -breakiterator 를 디폴트로 하는 것의 메리트보다, 그 변경에 수반해 필요하게 되는 원시 코드 변경의 수고라고 하는 디메리트가 크다고 생각한 것입니다. 이 건으로 여러분에게 여분의 수고를 걸쳐 , 혼란을 부른 것을 사과하겠습니다. (덧붙여 -breakiterator 옵션은 1.4. 0 으로 추가되었다. 또, 영어 로케일의 경우, 이 옵션을 지정하면(자) 다큐멘테이션 코멘트내의 최초의 문장의 마지막을 판별하기 위해서 BreakIterator 클래스가 강제적으로 사용됩니다. (영어 이외의 언어에서는, BreakIterator 에 의한 판별이 이전부터 실행되고 있었습니다. 4959985 , 툴)
    게다가 장래의 릴리스 (가능하면 5.1 로부터에서는, 새로운 javadoc 태그를 추가해 (기능 요구 4921374 ), 개요를 명시적으로 마크 할 수 있도록(듯이) 할 예정입니다. 그 결과, 어떠한 검출법을 사용할 필요가 없어집니다.
  • 새로운 태그 {@literal} 및 {@code}: {@literal} 태그는, 리터럴 텍스트를 나타냅니다. 예를 들어, 다큐멘테이션 코멘트 {@literal a<B>c} 는, 생성되는 HTML 페이지로 미변경인 채 a<B>c 라고 표시됩니다. 즉,<B> 는 볼드 태그로서 해석되지 않습니다.
    {@code} 태그는, 리터럴 텍스트를 코드 폰트로서 서식 지정합니다. {@code abc} 는 <code>{@literal abc}</code> 와 등가입니다.
  • 태그 {@value arg}:이 인 라인 태그는, 다음과 같이, 프로그램 요소명을 받아들이게 되었습니다. {@value package.class#field}. 이러한 표기는, 정수 필드의 다큐멘테이션 코멘트내 뿐만이 아니라, 임의의 다큐멘테이션 코멘트내에서 사용할 수 있습니다. (4764045 , 표준 도크 렛, 투표수 2)
  • package-info.java:이 이름으로 원시 파일을 작성해, 패키지의 코멘트와 주석을 보존할 수가 있습니다. package.html (코멘트만을 지원해, 주석은 지원하지 않는다)의 대용이 됩니다. package-info.java 가 존재하는 경우, javadoc 는 package.html 를 무시해, (경우에 따라서는 주석을 붙일 수 있었던) 패키지 선언의 직전에 package doc 코멘트를 찾습니다. javadoc 툴은,package-info.java 및 package.html 의 선택과 해석을 실시해, doc 코멘트를 도크 렛에 건네줍니다. 「패키지 코멘트 파일」을 참조해 주세요.
  • @Deprecated 주석:이것은 5.0 의 새로운 주석으로, 프로그램 요소를 비추천으로 하기 위한 @deprecated 태그에 추가해 사용합니다. Java 언어 스펙에서는,@Deprecated 주석 (반드시 태그는 아니다)을 사용하는 경우, 경고를 발행하는 컴파일러가 필요합니다. @deprecated 태그에는, 비추천 프로그램 요소에 대신하는 요소를 기술하는 장소가 있습니다. 「@Deprecated 주석」을 참조해 주세요.
  • 표준 도크 렛을 재구현: 표준 도크 렛 및 MIF 도크 렛이 재구현되어 가능한 한 많은 코드를 공유하게 되었습니다. 이것에 의해, 양쪽 모두의 도크 렛의 버그를 삭감하는 것이나, MIF 도크 렛의 기능을 표준 도크 렛의 기능에 승격 지난 일이 가능하게 됩니다. 덧붙여 새로운 MIF 도크 렛은 아직 릴리스 되고 있지 않습니다 (아마, 버젼 1.4 Beta 1 이라고 하는 명칭이 될 예정). 이 2 개의 도크 렛을 병합 하는 주된 이유는, 어느 쪽의 도크 렛도 형식이 다른 (HTML 와 PDF)만으로 본질적으로는 같은 문서를 생성하는 것이기 때문입니다. (4638723 , 표준 도크 렛, 투표수 2)
    이 공유 코드는, 도크 렛 툴 킷으로서 공개하는 것을 예정하고 있습니다. Javadoc 의 홈 페이지를 참조해 주세요.

주요한 버그 수정

  • 옵션 -tag:(리그렛션) -tag 태그를 수정해, 그 이름에 데쉬 「-」을 포함할 수가 있도록(듯이) 되돌렸습니다 (1.4. 1 으로 같이). 4920381 , 표준 도크 렛, 투표수 59)
  • {@inheritDoc} 가 원인의 행업:{@inheritDoc} 를 (1) 상속할 수 없는 경우, 혹은 (2) super-superinterface 를 상속하지 않는 경우, 행업 하지 않게 되었습니다. (4812240 , 표준 도크 렛, 투표수 18)
  • 태그 @param:(리그렛션) @param 를 수정해, 파라미터명 ( 제 1 인수)이 잘못되어 있어도, 파라미터명을 출력하도록(듯이) 했습니다 (1.3. 1 으로 같이). 4802275 , 표준 도크 렛, 투표수 1)
  • 태그 {@value}:{@value} 를 커스텀 태그 중(안)에서 사용해도, NullPointerException 가 생성되지 않게 되었습니다. (4821796 , 표준 도크 렛, 투표수 1)
  • 브라우저의 스크롤 바:정수치의 개요로, 클래스 표제의 주위에 테이블의 테두리를 추가했습니다. (4852280 , 표준 도크 렛, 투표수 5)
  • package-private 문서의 상속:-private 옵션을 사용했을 때, package-private 클래스로부터 상속된 메소드가, 상속된 메소드의 개요 테이블에만 문서화 되어 상세 섹션에는 문서화 되지 않는다고 하는, 올바른 동작을 하게 되었습니다. (4780441 , 표준 도크 렛, 투표수 5)
  • 최초의 문장이 {@inheritDoc} 로 끝나 있는 경우:{@inheritDoc} 를 포함한 최초의 문장의 마지막이 올바르게 판단됩니다. (4767038 , 표준 도크 렛, 투표수 4)
  • 최초의 문장에 {@docRoot} 가 포함되어 있으면(자) 링크가 망가지는:{@docRoot} 링크를 포함한 최초의 문장은, 단일 페이지 인덱스에서는 망가지지 않습니다. (4760191 , 표준 도크 렛, 투표수 3)
  • 상속된 코멘트에의 링크:상속된 코멘트가, 단지 클래스에 링크할 뿐만 아니라, 적절한 멤버에게 직접 링크하게 되었습니다. (4368820 , 표준 도크 렛, 투표수 1)
  • 모든 버그 수정에 대해서는,아래에 게재합니다. 기존의 버그 Javadoc 5.0 에는, 행업이나 크래쉬를 일으키는 기존의 버그가 있습니다. (그 일람은, 「Bug Parade」의 데이타베이스로 볼 수가 있습니다.

    상세한 것에 대하여는,Javadoc FAQ 를 참조해 주세요.

기존의 버그

Javadoc 5.0 에는, 행업이나 크래쉬의 원인이 되는 버그는 없습니다. Javadoc FAQ 에 기재된 Bug Parade 로, 데이타베이스의 일람표를 참조해 주세요. 주요한 버그는 다음과 같습니다.
  • 이러한 기능에 대응하기 위해(때문에), 커스텀 도크 렛도 개정할 필요가 있습니다. Javadoc 툴은, 이른바 「유산 (종래의)」도크 렛에 대해서, 가능한 범위에서 다음과 같은 프로그람뷰를 제시하려고 합니다. 회피책:없음 (5081166 )
  • 따라서, 예를 들어, 범용형의 구조로부터는 형태 파라미터와 형태 인수가 제거되어 형태 변수와 와일드 카드형은 새롭게 옮겨놓을 수 있어ClassDoc.fields() 는 enum 정수를 돌려줍니다. 회피책:없음 (4778311 )
  • 부정한 「정수치」페이지를 생성 정수에 꺽쇄묶음 '<'가 포함되는 경우
    회피책：없음 (5077317 )
  • 예를 들어, 5.0 의 Javadoc 툴을 사용하는 경우에, 도크 렛은 1.4 를 베이스로 하고 있어 처리 대상의 원시 코드도 1.4 를 베이스로 하고 있다고 해도, 그 도크 렛을 컴파일 할 수가 있어 도크 렛은 이전과 같이 동작합니다 (버그 수정은 제외하다). 에러 메세지:java.util.MissingResourceException:이후에의 리스트에는, 5.0 에 있어서의 실질적으로 모든 신기능과 버그의 수정을 게재합니다. 매우 작은 변경은 생략 하고 있습니다 (도크 렛 API 문서로 망가져 있던 링크의 수정 등). 커멘드행 옵션 신기능: -keywords 옵션에 의해, 생성되는 각 HTML 파일에, 검색에 편리한 META 태그가 추가되도록(듯이) 했습니다. 예를 들어 <action> 가 부정한 <a> 태그라고 해석됩니다. (5082928 )

커스텀 도크 렛과의 비호환성

5.0 보다 전에 기술된 커스텀 도크 렛은, 5.0 의 새로운 언어 기능을 사용하는 원시 파일에 대해서 실행하면(자), 호환성의 문제가 발생합니다.
  • 새로운 언어 기능:도크 렛 API 및 표준 도크 렛은, 5.0 의 새로운 언어 기능 (범용형, 열거형, 가변 인수 및 주석)을 처리할 수 있도록(듯이) 개정되었습니다. 이러한 기능에 대응하기 위해(때문에), 커스텀 도크 렛도 개정할 필요가 있습니다.

    Javadoc 툴은, 이른바 「유산 (종래의)」도크 렛에 대해서, 가능한 범위에서 다음과 같은 프로그람뷰를 제시하려고 합니다. (1) 5.0 보다 전의 원시 코드에서도 계속해 동작한다. (2) 5.0 원시 코드로 기대되는 일에 적합한다. 따라서, 예를 들어, 총칭의 구조로부터는 형태 파라미터와 형태 인수가 제거되어 형태 변수와 와일드 카드형은 새롭게 옮겨놓을 수 있어ClassDoc.fields() 는 enum 정수를 돌려줍니다.

    그렇다고는 해도, 유산 지원의 스펙은 정밀도가 높지는 않습니다. 처리 대상의 언어를 이해할 수 있도록(듯이) 도크 렛이 갱신될 때까지 의, 잠정적인 호환 모드입니다. 보증되고 있는 것은, 도크 렛 그 자체의 소스 및 바이너리의 호환성 뿐입니다. 예를 들어, 5.0 의 Javadoc 툴을 사용하는 경우에, 도크 렛은 1.4 를 베이스로 하고 있어 처리 대상의 원시 코드도 1.4 를 베이스로 하고 있다고 해도, 그 도크 렛을 컴파일 할 수가 있어 도크 렛은 이전과 같이 동작합니다 (버그 수정은 제외하다).

신기능과 버그 수정

이후에의 리스트에는, 5.0 에 있어서의 실질적으로 모든 신기능과 버그의 수정을 게재합니다. 매우 작은 변경은 생략 하고 있습니다 (도크 렛 API 문서로 망가져 있던 링크의 수정 등).

커멘드행 옵션

  • 신기능: -keywords 옵션에 의해, 생성되는 각 HTML 파일에, 검색에 편리한 META 태그가 추가되도록(듯이) 했습니다. (4776637 , 표준 도크 렛)
  • 신기능:새롭다 -notimestamp 옵션을 사용하면(자), HTML 페이지 위에 파묻히고 있는 은폐 표시의 타임 스탬프가 억제됩니다. (4777717 )
  • 신기능:이름에 하이픈을 포함한 테스트용의 서브 패키지를 작성할 수 있게 되었습니다. -subpackages 를 사용했을 때, 그러한 서브 패키지는 스킵 되어 경고는 생성되지 않습니다. 즉, 이 옵션에서는, Java 명으로서 무효인 이름의 디렉토리에 조우해도 경고를 생성하지 않습니다 (Java 명으로서 무효인 이름의 규칙은, 무효인 원시 파일명과 같다). (4773013 , 툴)
  • 버그 수정 :-help 옵션으로 사용 방법으로 「[classnames]」가 표시된다고 하는 실수가 수정되었습니다. (4775743 , 툴)

  • 신기능: {@literal} 태그는 리터럴 텍스트를 나타냅니다. 이 태그로 둘러싸인 텍스트에는, HTML 마크 업이나, 상자의 javadoc 태그가 포함되지 않는 것으로서 해석됩니다. 예를 들어, 다음의 다큐멘테이션 코멘트{@literal a<B>c} 는, 생성되는 HTML 페이지로 미변경인 채 표시됩니다. a<B>c - 즉, <B> 는 볼드 태그로서 해석되지 않습니다.
  • 신기능: {@code} 태그는, 리터럴 텍스트를 코드 폰트로서 서식 지정합니다. 이것은 <code>{@literal}</code> 와 등가입니다.
  • 신기능: {@value} 인 라인 태그가, 다음과 같이, 프로그램 요소 및 라벨의 이름을 받아들이게 되었습니다. {@value package.class#member label}. 그 때문에, 임의의 다큐멘테이션 코멘트 중(안)에서 이 태그를 사용할 수 있습니다. (4764045 , 표준 도크 렛)
  • 신기능:Javadoc 태그가 아닌 @Deprecated 주석은 5.0 으로부터의 신기능으로, 프로그램 요소를 비추천으로 하기 위한 @deprecated 태그에 추가해 사용합니다. 「@Deprecated 주석」을 참조해 주세요.
  • 버그 수정:(리그렛션) :@param 를 수정해, 파라미터명 ( 제 1 인수)이 잘못되어 있어도, 파라미터명을 출력하도록(듯이) 했습니다 (1.3. 1 으로 같다). (4802275 , 표준 도크 렛)
