부록 2:Sun 의 지원하는 Specification-Version 및 Implemenation-Version 의 형식

Specification-Version 와 Implementation-Version 는, 다음에 나타내는 Sun 제품의 규칙에 따르고 있습니다. 또, 서드 파티의 제품은, 확장 기능이 최신인가 어떤가를 판단하기 위해서(때문에), Java Plug-in 와 같은 규칙에 따를 필요가 있습니다.

Specification-Version 캐릭터 라인은, 이하의 형식을 취합니다.

n1.n2[. n3]

n1,n2,n3 에는 정수가 들어가,n1.n2 는 「메이저 버젼 번호」, 옵션의 n3 는 「마이너 버젼 번호」 ( 「보수 버젼 번호」라고도 불린다)을 나타냅니다.

Implementation-Version 도 최초의 부분은 같은 형식입니다만, 이하의 부분이 차이가 납니다.

  • 밑줄 (_)에 이어 n4n5 를 추가해, 「패치 버젼 번호」 ( 「갱신 버젼 번호」라고도 불린다)을 나타낸다
  • 또는, 하이픈 (-)을 붙여 마일 스톤명 (ea,alpha,beta,rc ...)(을)를 추가한다. 이것에는 뒤로 정수의 번호를 붙이는 일도 있다 (ea1,beta2,rc1 ...)

패치 버젼 번호와 마일 스톤명의 양쪽 모두를,Implemenation-Version 캐릭터 라인으로 동시에 사용할 수 없습니다.

일반적인 형식은 이하대로입니다.

n1.n2[. n3][_<patch_number>|-<milestone_name>]

상기의 버젼 형식에 사용할 수 있는 것은, 정수 (n1,n2,n3 ...), 문자, 닷, 하이픈, 및 밑줄만으로, 그 이외의 문자 (*,+ ...)(은)는 사용할 수 없습니다.

Specification-Version 및 Implementation-Version 의 번호는 이론에 근거해 붙일 수 있는 것은 아닙니다만, 실제로는 대부분 서로 동기를 유지하고 있습니다.

례:

Specification-Version 의 예: 1.3, 1.4

Implementation-Version 의 예:1.3. 1, 1.4. 0_02, 1.4. 0-beta3

상기에 든 규칙은, 서드 파티의 확장 기능 JAR 를 사용해 MANIFEST.MF 파일을 조사해 거기에 포함되는 실제의 값을 확인하는 경우에도, 항상 유효합니다. 이 규칙에 따르지 않은 경우, 애플릿의 JAR 와 거기에 대응하는 MANIFEST.MF 를 변경하거나 확장 기능 JAR 를 변경할 필요가 있습니다.
