Web 페이지 작성자전용의 정보 

Q:Web 페이지 작성자는, Java Plug-in 소프트웨어를 어떻게 사용할 수 있습니까.

A:Java Standard Edition 의 기능을 모두 이용하려면 , Web 페이지 작성자는, 페이지의 HTML 를 변경해, Java Plug-in 소프트웨어 경유로 Sun 의 JRE 를 사용할 필요가 있습니다. Sun 는, Web 페이지 작성자가 이러한 변경을 실행할 수 있도록(듯이) 스펙 (「Java Plug-in 에 있어서의 OBJECT,EMBED, 및 APPLET 태그의 사용」을 참조)을 제공하고 있습니다. 또, Sun 는 Java Plug-in 를 무료로 제공하고 있습니다.

Q:Windows 의 경우, 상기의 스펙에서는,. cab 파일을 개입시켜 자동적으로 다운로드할 수 있는 JRE 릴리스에 대해 진술되고 있습니다. 이러한 JRE 릴리스의 일람은 어디에 있습니까.

A:「자동 다운로드 파일 (Windows 마셔)」를 참조해 주세요.

Q:JAR 파일을 OBJECT 나 EMBED 태그의 일부로서 지정하는 방법을 가르쳐 주세요.  

A:OBJECT 나 EMBED 태그에 대해,archive,cache_archive, 또는 cache_archive_ex 파라미터를 정의하는 것으로써, 1 개(살) 이상의 JAR 파일을 지정할 수 있습니다.

OBJECT 태그에 의한 archive 파라미터는 이하와 같이 됩니다.

<PARAM NAME="archive" VALUE="demo.jar, fred.jar">

cache_archive 및 cache_archive_ex 의 상세한 것에 대하여는,「애플릿 캐쉬」를 참조해 주세요.

EMBED 태그의 경우, 다음과 같이 됩니다.

<EMBED ... archive="demo.jar, fred.jar" ... >

Q:Java Plug-in 소프트웨어에서는,APPLET 태그의 archive 속성에 복수의 JAR 파일을 지정 가능합니까. 지정 가능한 경우, 이것이 동작하지 않는 원인은 무엇입니까.

A:archive 속성은,EMBED 와 OBJECT 의 양쪽 모두의 태그로 지원되고 있습니다. JAR 파일을 부정한 순서로 배치해 버리는 잘못은 가장 일반적입니다. 예를 들어, Java Plug-in 로 Swing 세트를 사용해,archive="Myjar.jar, swing.jar,..."(을)를 지정하는 경우, Java Plug-in 는 애플릿의 로드에 실패합니다. 원인은,Myjar.jar 가 로드 되어 Java Plug-in 가 애플릿의 초기화를 시도하는 시점에서,swing.jar 가 로드되어 있지 않기 때문입니다. archive 내의 JAR 파일은, 의존성에 따른 순서에 배치할 필요가 있습니다. Myjar.jar 는 다른 JAR 파일에 의존하기 위해(때문에), 리스트의 마지막에 배치할 필요가 있습니다. 또, JAR 파일 리스트에 공백 또는 패스를 입력해 버리는 잘못도 일반적입니다.

애플릿 개발자전용의 정보 

Q:Java Plug-in 를 지원하기 위해서, 개발자는 애플릿을 변경할 필요가 있습니까.

A:그 필요는 없습니다. Java SDK, Standard Edition 의 100% Pure Java 애플릿은, Java Plug-in 를 사용해 수정하는 일 없이 실행할 수 있습니다.

Q:Java Plug-in 소프트웨어의 애플릿의 라이프 사이클에 대해 가르쳐 주세요.

A:HTML 페이지에 애플릿이 발견되면(자), 애플릿의 초기화 및 기동을 합니다. HTML 페이지를 닫는지,[돌아오는] 버튼을 클릭하면(자), 애플릿은 즉석에서 정지해, 파기됩니다.

같은 HTML 페이지에 재차 조우하는 경우, 애플릿의 초기화 및 기동이 재차 실행됩니다.

Q:Java Plug-in 소프트웨어는, 애플릿간의 드래그＆드롭 및 네이티브 환경을 지원합니까. 지원하는 경우, 이것이 동작하지 않는 원인은 무엇입니까.

A:네. Java Plug-in 소프트웨어는 드래그＆드롭을 지원합니다. 이 기능을 이용하려면 ,policytool 를 사용해, 애플릿에 적절한 소켓 액세스권을 확실히 부여할 필요가 있습니다. 자세한 것은,SocketPermission 클래스의 문서를 참조해 주세요.

Q:애플릿의 다운로드 시간을 단축하는 방법을 가르쳐 주세요.

A:몇개의 방법이 있습니다.

  • JAR 파일 - 애플릿을 JAR 파일로서 번들 합니다.
  • JAR 인덱스 - 클래스의 로드 처리를 최적화합니다. 자세한 것은,「JAR 파일의 스펙」 을 참조해 주세요.
  • 애플릿 캐싱 - 고정 캐쉬내에 애플릿을 캐쉬합니다 (「애플릿 캐쉬」를 참조).

Q:경고 배너가 GUI 상을 가려 버리는 것을 막기에는 어떻게 하면 좋습니까.  

A:getInsets() 메소드를 사용해, 프레임의 장식 경계의 사이즈를 확인합니다. 이것에는, 경고 배너도 포함됩니다. 예를 들어, 작성하는 Frame 의 사이즈가 100x100 인 경우,insets [top=42, left=5, bottom=5, right=6] 가 되어, draw 가능한 영역은 89x53 인 것을 알 수 있습니다. 이 draw 가능 영역내에 배치할 필요가 있습니다.

특정 사이즈의 draw 가능 영역을 작성할 필요가 있는 경우, 이하를 실행합니다.

  1. Frame 를 작성 및 표시
  2. getInsets 를 사용해 인 세트의 사이즈를 확인
  3. 인 세트에 필요한 사이즈를 추가해 frame size를 산출
  4. frame.setSize() 를 사용해 프레임을 그 사이즈로 설정

Q:InetAddress.getLocalHost(). getHostName() 는 왜 localhost 를 돌려줍니까.

A: 이것은, Java 플랫폼의 의도적인 시큐리티 기능입니다. 신뢰되지 않는 애플릿에는, 실제의 호스트명은 통지되지 않습니다. 신뢰되는 애플릿 (서명 첨부 애플릿등)에는, 실제의 호스트명이 통지됩니다.

시스템 관리 책임자전용의 정보

Q:시스템 관리 툴을 사용해, 몇천의 머신의 Java Plug-in 설정을 변경할 필요가 있습니다. 어떻게 하면 좋습니까.

A:Java Plug-in 는, 사용자 고유의 설정을 <user.home>/.java/properties<version> 아래에 포함합니다. 이 때문에, 관리자는, 시스템 관리 툴을 사용해, 모든 머신상에서 파일을 갱신하는 것으로써, Java Plug-in 설정을 글로벌로 변경할 수 있습니다.

Q:Java Plug-in 소프트웨어에 의해, Microsoft 나 Netscape 의 Java Runtime 는 Sun 의 JRE 로 옮겨집니까.

A:아니오. Java Plug-in 소프트웨어는, 브라우저의 기반을 이루는 가상 머신이 장비 되고 있으면(자), 그것을 옮겨놓는 것은 하지 않습니다. (Netscape 6 으로 Internet Explorere 6 의 최신 버젼에는 Java 의 가상 머신은 장비 되고 있지 않습니다. ) Java Plug-in 소프트웨어는, Web 페이지 작성자가, 디폴트의 Java Runtime 대신에 Sun 의 JRE 를 지정 가능하게 할 뿐입니다.

Q:인트라넷 환경에서 Java Plug-in 소프트웨어의 배치를 시도하고 있습니다. 「Java Plug-in 에 있어서의 OBJECT,EMBED 및 APPLET 태그의 사용」 이라고 하는 장이나 그 외의 문서에서는, Netscape 사용자가 Java Plug-in 소프트웨어를 인스톨하기 위한 인스톨 페이지를 인트라넷내로 설정하도록(듯이)와 진술되고 있습니다. 이 페이지의 기능 및 설정 방법법을 가르쳐 주세요.

A:이 인스톨 페이지의 목적은, Netscape 사용자가 Java Plug-in 소프트웨어를 인스톨 할 때의 엔트리 포인트로서 기능하는 것입니다. 사용자가 Java Plug-in 가 필요한 HTML 페이지에 조우하면(자),EMBED 태그의 pluginspage 속성에 따라 이 인스톨 페이지에 이끌립니다. 여기서, 플랫폼에 맞은 Java Plug-in 소프트웨어의 적절한 버젼을, 사용자가 인스톨 할 수 있도록(듯이) 할 필요가 있습니다. 이 때문에, 인스톨 페이지는, Java Plug-in 바이너리의 다운로드에 링크하고 있을 필요가 있습니다.

간단한 예를 다음에 나타냅니다. 

<HTML>
<HEAD>
<TITLE>Java Plug-in Software Download Page</TITLE>
</HEAD>
<BODY>
<P><A HREF="ftp://myhost.com/public/jre14-win32.exe">
    Java Plug-in Software for Microsoft Windows</A>
</P>
<P><A HREF="ftp://myhost.com/public/plugin-14-solaris.bin">
        Java Plug-in Software for Solaris</A>
</P>
</BODY>
</HTML>
Web 서버의 구성 방법에 따라서는, 다운로드용으로 FTP 는 아니고 CGI 스크립트를 사용하는 일도 생각할 수 있습니다. 자세한 것은, Web 마스터에 확인해 주세요.

Q:Netscape Navigator 와 Internet Explorer 에서는, 다운로드 및 인스톨의 동작이 다른 것은 왜입니까.

A:Java Plug-in 소프트웨어의 지원하는 Netscape Navigator 의 릴리스에는, Internet Explorer 와 같은 Java Plug-in 소프트웨어의 자동 다운로드용의 기구가 존재하지 않습니다. Netscape Navigator 가 Java Plug-in 에 대응한 Web 페이지 (기동 페이지)에 처음으로 조우하면(자), 사용자를 다른 Web 페이지에 유도해, 사용하는 시스템에의 Java Plug-in 소프트웨어의 다운로드 및 인스톨을 합니다. 그 후, 사용자는 기동 페이지에 돌아와, Java Plug-in 소프트웨어를 사용해 애플릿의 draw를 합니다. 이 때문에, 브라우저는 기동 페이지에 조우할 때마다 Java Plug-in 를 자동적으로 호출하게 됩니다.
