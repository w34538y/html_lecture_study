JNI 함수

제 4 장

이 장은, JNI 함수의 레퍼런스입니다. 이 장에서는, JNI 의 함수를 모두 채택합니다. 또, JNI 함수 테이블의 배치 그대로 기재되어 있습니다.

「? 해야 한다」(또는 「? 할 필요가 있다」)이라고 하는 표현은, JNI 프로그래머에 대한 제약을 나타내고 있습니다. 예를 들어, 어느 JNI 함수에 대해, 그것이 null 이외의 객체를 접수취등 「없으면 안 된다」라고 설명되고 있는 경우, 프로그래머의 책임에 대해, 그 JNI 함수에 null 를 건네주지 않게 해야 한다고 하는 의미입니다. 거기에 따라, JNI 구현 시에, 그 JNI 함수에 있어서의 null 의 포인터 체크를 실시할 필요가 없어집니다.

이 장의 일부는 Netscape 의 JRI 문서를 개작한 것입니다.

참조 자료에서는, 함수를 용도에 의해 그룹화 하고 있습니다. 참조 섹션은, 다음의 기능 분야로부터 구성되어 있습니다.

인터페이스 함수 테이블

각 함수는, 「JNIEnv」인수를 개입시켜, 고정 오프셋(offset)로부터 액세스 할 수 있습니다. 「JNIEnv」형은, 모든 JNI 함수의 포인터를 포함하는 구조체를 가리키는 포인터입니다. 이것은 다음과 같이 정의됩니다. 


typedef const struct JNINativeInterface *JNIEnv;

VM 는,코드예 4-1 에 나타나고 있도록(듯이), 함수 테이블을 초기화합니다. 최초의 3 엔트리는, 장래의 COM 와의 호환성을 위해서(때문에) 예약되고 있는 것에 주의해 주세요. 게다가 함수 테이블의 초의 부근에 몇개의 추가의 null 엔트리를 예약되어 있습니다. 따라서, 예를 들어, 지금부터 나타나는, 클래스 관련의 JNI 연산은, 겉(표)의 마지막은 아니고 FindClass 의 후에 추가할 수가 있습니다.

함수 테이블은, 모든 JNI 인터페이스 포인터의 사이에 공용됩니다.

버젼 정보

GetVersion

jint GetVersion(JNIEnv *env);

네이티브 메소드 인터페이스의 버젼을 돌려줍니다.

링키지:

JNIEnv 인터페이스 함수 테이블의 인덱스 4.

파라미터:

env:JNI 인터페이스 포인터

반환값:

메이저 버젼 번호를 고위의 16 비트로, 마이너 버젼 번호를 하위의 16 비트로 돌려줍니다.

SDK/JRE 1.1 에서는,GetVersion() 은 0x00010001 를 돌려줍니다.

SDK/JRE 1.2 에서는,GetVersion() 는 0x00010002 를 돌려줍니다.

SDK/JRE 1.4 에서는,GetVersion() 는 0x00010004 를 돌려줍니다.

SDK/JRE 1.6 에서는,GetVersion() 은 0x00010006 를 돌려줍니다.

정수

SDK/JRE 1.2 이후:
#define JNI_VERSION_1_1 0x00010001
#define JNI_VERSION_1_2 0x00010002

/* Error codes */
#define JNI_EDETACHED    (-2)              /* thread detached from the VM */
#define JNI_EVERSION     (-3)              /* JNI version error
SDK/JRE 1.4 이후: 
    #define JNI_VERSION_1_4 0x00010004
SDK/JRE 1.6 이후: 
    #define JNI_VERSION_1_6 0x00010006

클래스 조작

DefineClass

jclass DefineClass(JNIEnv *env, jobject loader,
const jbyte *buf, jsize bufLen);

raw 클래스 데이터의 버퍼로부터 클래스를 로드합니다. raw 클래스 데이터를 포함한 버퍼는, DefineClass 호출이 돌아온 후는 VM 에 의해 참조되지 않습니다. 필요에 따라서 파기해도 상관하지 않습니다.

링키지:

JNIEnv 인터페이스 함수 테이블의 인덱스 5.

파라미터:

env:JNI 인터페이스 포인터

loader:정의된 클래스에 할당할 수 있는 클래스 로더

buf:. class파일 데이터를 포함한 버퍼

bufLen:버퍼장

반환값:

Java 클래스 객체를 돌려줍니다. 에러가 발생했을 경우는 NULL 를 돌려줍니다.

예외:

ClassFormatError:클래스 데이터가 유효한 클래스를 지정하지 않았던 경우

ClassCircularityError:클래스 또는 인터페이스가, 그 자체의 슈퍼 클래스 또는 슈퍼 인터페이스가 되는 경우

OutOfMemoryError:시스템이 메모리 부족의 경우

FindClass

jclass FindClass(JNIEnv *env, const char *name);

이 함수는, 로컬에 정의된 클래스를 로드합니다. 또, 지정된 이름을 가지는 클래스에 대해서 CLASSPATH 환경 변수가 지정하는 디렉토리 및 ZIP 파일을 검색합니다.

링키지:

JNIEnv 인터페이스 함수 테이블의 인덱스 6.

파라미터:

env:JNI 인터페이스 포인터

name:완전 수식 클래스명 ( 「/」으로 단락지은 뒤에 클래스명을 붙인 패키지명). 그 이름이 「[」(배열 시그니챠 문자)로 시작되어 있는 경우는, 배열 클래스를 돌려줍니다.

반환값:

완전 수식명으로부터 클래스 객체를 돌려줍니다. 클래스가 발견되지 않는 경우는,NULL 를 돌려줍니다.

예외:

ClassFormatError:클래스 데이터가 유효한 클래스를 지정하지 않았던 경우

ClassCircularityError:클래스 또는 인터페이스가, 그 자체의 슈퍼 클래스 또는 슈퍼 인터페이스가 되는 경우

NoClassDefFoundError:요구된 클래스 또는 인터페이스에 대한 정의가 발견되지 않았던 경우

OutOfMemoryError:시스템이 메모리 부족의 경우

SDK/JRE 1.2 이후:

JDK 1.1 에서는,FindClass 는 CLASSPATH 내의 로컬 클래스만을 검색했습니다. 검색된 클래스는, 클래스 로더를 가지고 있지 않았습니다.

Java 시큐리티 모델은 확장되어 시스템 클래스 이외의 클래스에 의한 네이티브 메소드의 로드 및 호출이 가능하게 되었습니다. Java 2 플랫폼에서는,FindClass 가, 현재의 네이티브 메소드와 관련지을 수 있고 있는 클래스 로더를 검출합니다. native code가 시스템 클래스에 속하는 경우, 클래스 로더는 검출되지 않습니다. 그 이외의 경우에는, 적절한 클래스 로더가 불려 가 이름을 붙일 수 있었던 클래스의 로드 및 링크를 실시합니다.

FindClass 가 호출해 인터페이스에 의해 불려 갔을 경우, 현재의 네이티브 메소드 또는 거기에 관련지을 수 있었던 클래스 로더는 존재하지 않습니다. 이 경우,ClassLoader.getBaseClassLoader 의 결과가 사용됩니다. 이것은, 가상 머신이 어플리케이션용으로 작성하는 클래스 로더입니다. java.class.path 프로퍼티에 리스트 된 클래스를 검색할 수 있습니다.

GetSuperclass

jclass GetSuperclass(JNIEnv *env, jclass clazz);

clazz 가 Object 클래스 이외의 클래스를 나타내는 경우, 이 함수는,clazz 에 의해 지정된 클래스의 슈퍼 클래스를 나타내는 객체를 돌려줍니다.

clazz 가 Object 클래스를 지정하는 경우, 또는 clazz 가 인터페이스를 나타내는 경우는, 이 함수는 NULL 를 돌려줍니다.

링키지:

JNIEnv 인터페이스 함수 테이블의 인덱스 10.

파라미터:

env:JNI 인터페이스 포인터

clazz:Java 클래스 객체

반환값:

clazz 에 의해 나타내지는 클래스의 슈퍼 클래스, 또는 NULL 를 돌려줍니다.

IsAssignableFrom

jboolean IsAssignableFrom(JNIEnv *env, jclass clazz1,
jclass clazz2);

clazz1 의 객체가 안전하게 clazz2 에 캐스트 될지 어떨지를 결정합니다.

링키지:

JNIEnv 인터페이스 함수 테이블의 인덱스 11.

파라미터:

env:JNI 인터페이스 포인터

clazz1:최초의 클래스 인수

clazz2:2 번째의 클래스 인수

반환값:

다음중 한쪽이 진정한 경우에 JNI_TRUE 를 돌려줍니다.

  • 최초와 2 번째의 클래스 인수가 같은 Java 클래스를 나타내고 있다
  • 최초의 클래스가 2 번째의 클래스의 서브 클래스이다
  • 최초의 클래스가 그 인터페이스로서 2 번째의 클래스를 가지고 있다

약글로벌 참조

약글로벌 참조는, 특별한 종류의 글로벌 참조입니다. 일반적으로의 글로벌 참조와 달리, 약글로벌 참조를 사용하면(자), 부하의 Java 객체를 가베지 컬렉트할 수가 있습니다. 약글로벌 참조는, 글로벌 또는 로컬 참조가 사용되고 있는 상황이라면 어디서에서도 사용할 수 있습니다. 가비지 컬렉터를 실행하면(자), 부하의 객체가 약참조 에 의해서만 참조되고 있는 경우, 그 객체가 해제됩니다. 해제된 객체를 참조하는 약글로벌 참조는, 기능적으로 null 와 동등합니다. 프로그래머는,IsSameObject 를 사용해 약참조와 null 를 비교하는 것으로써, 약글로벌 참조가 해제된 객체를 참조하고 있을지 어떨지 확인할 수 있습니다.

JNI 의 약글로벌 참조는, Java 2 플랫폼 API (java.lang.ref 및 그 클래스)의 일부로서 입수 가능한 Java 약참조의 간이판입니다.

해설     (2001년 6월에 추가)

가베지 컬렉션은 네이티브 메소드의 실행중에 발생하기 위해(때문에), 약글로벌 참조로 참조되고 있는 객체는 언제라도 해제할 수 있습니다. 약글로벌 참조는, 글로벌 참조가 사용되고 있는 곳(중)이라면 어디에서라도 사용할 수 있습니다만, 그처럼 사용한다고 예고없이 null 와 기능적으로 동등하게 되는 경우가 있으므로, 일반적으로는 부적절합니다.

「IsSameObject」는 약글로벌 참조가 해제된 객체를 참조하고 있을지 어떨지를 판별하는데 사용할 수 있습니다만, 객체가 그 직후에 해제되는 것을 방지하는 것은 아닙니다. 그 때문에, 프로그래머는 이 검사에 근거해, 그 후의 JNI 호출로 약글로벌 참조가 ( null 참조 이외로) 사용되고 있는지 어떤지를 판별할 수 없습니다.

이 상속 제한을 해소하기 위해(때문에), JNI 함수 「NewLocalRef」또는 「NewGlobalRef」를 사용해 동일한 객체에의 표준 (강하다) 로컬 참조 또는 글로벌 참조를 취득해, 이 강한 참조를 사용해 해당하는 객체에 액세스 하는 것을 추천합니다. 객체가 해제되고 있는 경우, 이 함수는 null 를 돌려줍니다. 그 외의 경우, 강한 참조를 돌려줍니다 (강한 참조는 객체가 해제되는 것을 방지합니다). 객체에의 직접 액세스가 불필요하게 되었을 때는, 객체를 해제할 수 있도록(듯이), 새로운 참조를 명시적으로 삭제할 필요가 있습니다.

약글로벌 참조는, 다른 종류의 약한 참조 (SoftReference 클래스 또는 WeakReference 클래스의 Java 객체)보다 약한 참조입니다. 특정의 객체에의 약한 글로벌 참조는, 그 객체를 참조하는 SoftReference 객체 또는 WeakReference 객체가 참조를 해제할 때까지, 기능적으로 null 와 동등하게는 되지 않습니다.

약글로벌 참조는, finalize를 필요로 하는 객체에의 Java 의 내부 참조보다 약한 참조입니다. 약글로벌 참조는, 참조처의 객체의 파이나라이자가 존재하는 경우, 그것이 완료할 때까지,null 와 기능적으로 동등하게는 되지 않습니다.

약글로벌 참조와 PhantomReference 와의 상호 동작은 미정도리입니다. 특히, Java VM 의 구현은, PhantomReference 의 후에 약글로벌 참조를 처리하는 경우가 있어, PhantomReference 객체에서도 참조되고 있는 객체를 보관 유지하기 위해서 약글로벌 참조를 사용하는 것이 가능한 경우가 있습니다. 약글로벌 참조를 이러한 미정도리의 방법으로 사용하는 것은 피해야 합니다.

GetObjectRefType

jobjectRefType GetObjectRefType(JNIEnv* env, jobject obj);

obj 인수에 의해 참조되는 객체의 형태를 돌려줍니다. 인수 obj 는, 로컬 참조, 글로벌 참조, 약글로벌 참조의 머지않아인가입니다.

링키지:

JNIEnv 인터페이스 함수 테이블의 인덱스 232.

파라미터:

env:JNI 인터페이스 포인터

obj:로컬 참조, 글로벌 참조, 또는 약글로벌 참조.

vm:인터페이스 취득원의 가상 머신 인스턴스.

env:현재의 thread의 JNI 인터페이스 포인터의 배치 위치를 참조하는 포인터.

version:요구된 JNI 버젼.

반환값:

함수 GetObjectRefType 는, jobjectRefType 로서 정의된 다음의 열거치의 1 개를 돌려줍니다.

JNIInvalidRefType = 0,
JNILocalRefType = 1,
JNIGlobalRefType = 2,
JNIWeakGlobalRefType = 3

인수 obj 가 약글로벌 참조형의 경우, 반환값은 JNIWeakGlobalRefType 가 됩니다.

인수 obj 가 글로벌 참조형의 경우, 반환값은 JNIGlobalRefType 가 됩니다.

인수 obj 가 로컬 참조형의 경우, 반환값은 JNILocalRefType 가 됩니다.

인수 obj 가 유효한 참조가 아닌 경우, 이 함수의 반환값은 JNIInvalidRefType 가 됩니다.

무효인 참조란, 유효한 핸들이 아닌 참조입니다. 즉,obj 포인터 주소는, 몇개의 Ref 작성 함수로부터 할당할 수 있었던 메모리내의 위치나 JNI 함수로부터 반환된 메모리내의 위치를 지시하고 있지 않는 참조입니다.

따라서,NULL 는 무효인 참조가 되어,GetObjectRefType(env, NULL) 은 JNIInvalidRefType 를 돌려줍니다.

한편, null 를 지시하는 참조인 null 참조는, 그 null 참조가 최초로 작성되었을 때의 참조의 형태를 돌려줍니다.

GetObjectRefType 는, 삭제된 참조에서는 사용할 수 없습니다.

참조는 일반적으로, VM 로 참조 할당 서비스에 의한 재사용의 가능성이 있는 메모리 데이터 구조의 포인터로서 구현됩니다. 따라서, 참조를 삭제하면(자) GetObjectRefType 의 반환값을 지정할 수 없습니다.

이것들 3 종류의 연산 패밀리는, 네이티브 메소드로부터 Java 인스턴스 메소드를 호출할 때에 사용됩니다. 이것들 3 종류의 패밀리는, 호출한 메소드에 파라미터를 건네주는 기구가 다를 뿐입니다.

이러한 연산 패밀리는, 지정된 메소드 ID 에 따라, Java 객체상의 인스턴스 (비 static) 메소드를 호출합니다. methodID 인수는,GetMethodID() 를 호출하는 것에 의해 취득할 필요가 있습니다.

이러한 함수를 사용해 private 메소드나 생성자 을 호출하는 경우는, 메소드 ID 를 obj 의 열매 클래스의 슈퍼 클래스의 1 개로부터는 아니고, 열매 클래스 자체로부터 취득할 필요가 있습니다.

Call<type>MethodA 루틴

프로그래머는, 메소드에 건네주는 인수를 모두,methodID 인수의 직후의 va_list 형의 args 인수에 넣습니다. Call<type>MethodV 루틴은, 인수를 받아, 프로그래머가 호출하고 싶은 Java 메소드에 건네줍니다.

주:

SDK/JRE 1.1 에서는, 프로그래머는 Get/ReleaseArrayElements 함수를 사용해, 프리미티브(primitive) 배열 요소에의 포인터를 취득할 수 있었습니다. VM 가 피닝을 지원하고 있는 경우, 원의 데이터에의 포인터가 돌려주어졌습니다. 지원하고 있지 않는 경우에는, 카피가 작성되었습니다.

SDK/JRE 1.3 에 도입된 새로운 함수를 사용하면(자), VM 가 피닝을 지원하고 있지 않는 경우에서도, native code는 배열 요소에의 직접 포인터를 취득할 수가 있습니다.

GetPrimitiveArrayCritical
ReleasePrimitiveArrayCritical

void * GetPrimitiveArrayCritical(JNIEnv *env, jarray array, jboolean *isCopy);
void ReleasePrimitiveArrayCritical(JNIEnv *env, jarray array, void *carray, jint mode);

위의 2 개의 함수의 시멘틱스는, 기존의 Get/Release<PrimitiveType>ArrayElements 함수와 매우 자주(잘) 닮았습니다. 가능한 경우는, VM 는 프리미티브(primitive) 배열에의 포인터를 돌려줍니다. 그렇지 않은 경우는, 카피가 작성됩니다. 다만, 이러한 함수의 사용 방법에 관해서 중요한 제한이 있습니다.

GetPrimitiveArrayCritical 를 호출한 뒤,ReleasePrimitiveArrayCritical 를 호출하기 전에, native code를 특정의 기간 실행하지 않게 합니다. 이 1 조의 함수 내부의 코드는 「위기 리젼」으로 실행되고 있는 것으로서 취급할 필요가 있습니다. 위기 리젼 내부에 있어 native code는, 다른 JNI 함수를 호출해 안되어, 현재의 thread에 다른 Java thread를 블록 해 대기시키는 것을 가능하게 하는 어느 시스템 콜도 호출해서는 안됩니다. 예를 들어, 현재의 thread는, 다른 Java thread가 기술한 스트림상의 read 를 호출해서는 안됩니다.

이러한 제한은, VM 가 피닝을 지원하지 않는 경우에서도, native code가 배열의 카피되어 있지 않은 버젼을 취득할 가능성을 높입니다. 예를 들어, native code가 GetPrimitiveArrayCritical 에 의해 취득된 배열에의 포인터를 보관 유지하고 있는 경우, VM 는 일시적으로 가베지 컬렉션을 무효로 하는 일이 있습니다.

GetPrimtiveArrayCritical 및 ReleasePrimitiveArrayCritical 의 복수의 편성은, 상자로 할 수 있습니다. 예를 나타냅니다. 

  jint len = (*env)->GetArrayLength(env, arr1); 
  jbyte *a1 = (*env)->GetPrimitiveArrayCritical(env, arr1, 0);
  jbyte *a2 = (*env)->GetPrimitiveArrayCritical(env, arr2, 0);
  /* We need to check in case the VM tried to make a copy.  */
  if (a1 == NULL || a2 == NULL) {
    ... /* out of memory exception thrown */
  }
  memcpy(a1, a2, len);
  (*env)->ReleasePrimitiveArrayCritical(env, arr2, a2, 0);
  (*env)->ReleasePrimitiveArrayCritical(env, arr1, a1, 0);

VM 가 내부적으로 다른 형식에서 배열을 나타내는 경우,GetPrimitiveArrayCritical 는 아직 배열의 카피를 작성할 가능성이 있습니다. 그 때문에, 일어날 수 있는 메모리 부족의 상황에 대응해 null 에 대한 반환값을 체크할 필요가 있습니다.

NIO 의 지원

NIO 관련의 엔트리 포인트에 의해, native code가 java.nio「직접 버퍼」에 액세스 할 수 있습니다. 직접 버퍼의 컨텐츠는, 일반적으로의 가베지 컬렉트된 heap의 외측의, 네이티브메모리내에 포함할 수가 있습니다. 직접 버퍼에 대한 자세한 것은, 「New I/O API 」 및 java.nio.ByteBuffer 클래스의 스펙을 참조해 주세요.

SDK/JRE 1.4 에 도입된 3 개(살)의 새로운 함수에 의해, 직접 버퍼를 JNI 코드로 작성, 검사, 및 조작할 수가 있습니다.
  • NewDirectByteBuffer
  • GetDirectBufferAddress
  • GetDirectBufferCapacity

Java 가상 머신의 각 구현에서는 이러한 함수를 지원할 필요가 있습니다만, 모든 구현이 직접 버퍼에의 JNI 의 액세스를 지원할 필요는 없습니다. JVM 가 이러한 액세스를 지원하지 않는 경우는,NewDirectByteBuffer 및 GetDirectBufferAddress 함수는 항상 null 를 돌려주어,GetDirectBufferCapacity 함수는 항상 -1 을 돌려줍니다. JVM 가 이러한 액세스를 지원하는 경우, 각 함수는 적절한 값을 돌려주도록(듯이) 구현될 필요가 있습니다.

NewDirectByteBuffer

jobject NewDirectByteBuffer(JNIEnv* env, void* address, jlong capacity);

메모리아드레스 address 로부터 시작되는 capacity 바이트까지의 메모리브록크를 참조해, 직접 버퍼 java.nio.ByteBuffer 를 할당해 돌려줍니다.

이 함수를 호출해, Java 레벨의 코드에 최종적인 바이트 버퍼의 객체를 돌려주는 native code는, 그 버퍼가, read와 적절한 경우에는 서두의 액세스가 가능한, 유효한 메모리 영역을 참조할 필요가 있습니다. Java 코드로부터 무효인 메모리 위치에 액세스 하려고 하면(자), 시각 효과가 없는 임의의 값을 돌려주는지, 지정되어 있지 않은 예외가 발생합니다.

env:JNIEnv 의 인터페이스 포인터

buf:직접 버퍼 java.nio.Buffer 객체 (null 는 불가)

반환값:

버퍼에 관련지을 수 있었던 메모리 영역의 사이즈를 바이트수로 돌려줍니다. 지정된 객체가 직접 버퍼의 java.nio.Buffer 객체가 아닌 경우, 또는, 직접 버퍼에의 JNI 의 액세스가 이 가상 머신으로 지원되어 있지 않은 경우는,-1 을 돌려줍니다.

도입된 버젼:

SDK/JRE 1.4

리플렉션의 지원

프로그래머는, 메소드 또는 필드의 이름 및 형태를 파악하고 있는 경우, JNI 를 사용해 Java 메소드의 호출해 또는 Java 필드에의 액세스를 실시할 수가 있습니다. Java Core Reflection API 를 사용하면(자), 프로그래머는 실행시에 Java 클래스의 내부를 조사할 수가 있습니다. JNI 는, JNI 로 사용되는 필드와 메소드 ID 및 Java Core Reflection API 로 사용되는 메소드 객체의 사이의 변환 함수세트를 제공합니다.

FromReflectedMethod

jmethodID FromReflectedMethod(JNIEnv *env, jobject method);

java.lang.reflect.Method 또는 java.lang.reflect.Constructor 객체를 메소드 ID 로 변환합니다.
