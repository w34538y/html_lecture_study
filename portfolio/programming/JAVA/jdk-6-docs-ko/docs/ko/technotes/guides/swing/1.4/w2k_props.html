Windows 데스크탑 관련의 프로퍼티의 지원

이 기능은, 다음의 2 개의 분야에 관련합니다.

  • 사용자의 설정을 한층 더 존중한다
    • 배색에 관한 프로퍼티
    • 외형에 관한 프로퍼티
    • 폰트 프로퍼티
    • 시각 효과에 관한 프로퍼티
    • 음성 피드백에 관한 프로퍼티
    • 입력에 관한 프로퍼티
  • 동적 변경에 대한 응답

사용자의 설정을 한층 더 존중한다

Windows 플랫폼에서는, 사용자는, 데스크탑의 pop-up menu에 있는 「프롭퍼티」옵션 경유로, GUI 를 커스터마이즈 할 수 있습니다. Swing 의 Look & Feel 가 사용자의 시각 및 동작의 설정을 올바르게 구현하고 있는지를 확인하기 위해서, 시스템 레지스트리로부터 UI 프로퍼티의 값을 취득할 필요가 있습니다. 이러한 프로퍼티의 값은,java.awt.Toolkit 의 getDesktopProperty() 메소드로 검색 가능합니다.

이 메소드로 액세스 가능한 프로퍼티의 대부분은 Windows 플랫폼에 특유인 프로퍼티이며, 「win.」라고 하는 접두사가 붙어 있습니다. 지원되는 Windows 프로퍼티의 리스트는, 「win.propNames」프로퍼티을 프로그램으로 문의해 취득할 수 있습니다. 

     String propnames[] = (String[]) Toolkit.getDefaultToolkit(). getDesktopProperty("win.propNames");
     System.out.println("Supported windows property names:");
         for(int i = 0; i < propnames.length; i++) {
             System.out.println(propnames[i]);
         }

모든 플랫폼에서 지원되는 프로퍼티은, 접두사의 「awt.」(으)로 시작됩니다.

프로그램으로 이러한 프로퍼티에 직접 액세스 할 필요는 없고, Windows 의 Look & Feel 가 자동적으로 읽어내져 이러한 프로퍼티이 컴퍼넌트에 적절한 시각 효과와 동작을 제공하도록(듯이) 해석됩니다.

폰트의 호환성

현재, Windows 의 Look & Feel 내에서 실행되고 있는 Swing 프로그램은, Java 정의의 시스템 폰트는 아니고, 사용자가 설정한 시스템 폰트로 draw 합니다. 이 때문에, 종래의 동작에 근거하고 있는 프로그램과의 사이에 호환성의 문제가 발생할 가능성이 있으므로 (외형상의 문제나 특정의 폰트Glyph에 의존하는 로컬라이제이션의 설정등이 원인), 필요에 따라서 이 기능을 오프로 하는, 다음의 실행시의 프로퍼티이 준비되어 있습니다. 

     java -Dswing.useSystemFontSettings=false MyJavaProgram

사운드의 동적 변경

사운드 프로퍼티에 반환되는 객체는, 그 프로퍼티용으로 현재의 오디오 클립을 재생하는 단순한 Runnable 입니다. 즉, 사운드의 프로퍼티치는 현재의 사운드 설정에의 라이브 링크이기 (위해)때문에, 사운드에 관한 프로퍼티에의 동적 변경을 감시할 필요는 없습니다.

동적 변경에 대한 응답

JDK 1.3 에서는, 데스크탑에 관한 프로퍼티치의 1 개가 변화한다고 통지를 받도록(듯이) 등록해 두는 기능이, AWT 에 의해 추가되었습니다. 이 기능은,java.awt.Toolkit 의 addPropertyChangeListener() 메소드 경유로 지원됩니다. Swing 의 Look & Feel 에서는, 이 기능을 사용해 시각에 관한 프로퍼티의 동적인 변경 이벤트를 감시해, 컴퍼넌트가 최신의 시각에 관한 프로퍼티치를 가질 수 있도록(듯이), UI 를 언인스톨하거나 재인스톨 하거나 해, GUI 를 갱신합니다. 이 동적인 동작은, Windows 의 Look & Feel 를 실행중의 모든 Swing 프로그램으로 자동적으로 행해집니다.

이 기구는, GUI 의 계층을 Frame.getFrames() 로 시작해 거기로부터 모든 윈도우/컨테이너/컴퍼넌트를 횡단(traverse) 하는 것으로, GUI 컴퍼넌트를 갱신합니다. 횡단(traverse)을 할 때 클라이언트에 표시 불능인 컴퍼넌트가 있는 경우, 이러한 컴퍼넌트는 자동적으로는 갱신되지 않습니다. 이러한 컴퍼넌트의 Look & Feel 가 최신의 것임을 보증하기 위해서, 클라이언트 프로그램으로 컴퍼넌트의 UI 를 갱신할 필요가 있습니다 (컴퍼넌트가 표시 가능한 경우의 정의에 대해서는,java.awt.Component 의 isDisplayable() 메소드를 참조). 예를 들어 이러한 일은, SwingSet2 의 데모로 발생합니다. 데모 윈도우가 작성되어도, 데모의 탭 첨부 페인으로부터 선택될 때까지, 윈도우가 GUI 의 계층에 추가되지 않기 때문입니다. 따라서, 새롭게 선택한 데모 윈도우가 계층에 추가되기 전에, 최신의 프로퍼티 설정치가 되어 있는 것을 보증하기 위해서, UI 를 갱신합니다. 

    fragment from SwingSet2.java:  // Ensure panel's UI is current before making visible
    JComponent currentDemoPanel = demo.getDemoPanel();
    SwingUtilities.updateComponentTreeUI(currentDemoPanel);
    // Replace current demo with newly selected demo
    demoPanel.removeAll();
    demoPanel.add(currentDemoPanel, BorderLayout.CENTER);
