여기에서는 다음의 토픽에 대해 설명합니다.

  • 다층 어플리케이션
  • 사용자 인터페이스층
  • 서비스 (서버) 층
  • 데이터 포함 (데이타베이스) 층

다층 어플리케이션

종래의 엔터프라이즈 어플리케이션의 대부분은, 상호의 프로시저나 데이터에 제한부의 액세스를 실시하는 자기 완결형의 모노리식인 프로그램이었습니다. 그것들은, 많은 경우, 단순한 변경에서도 프로그램 전체의 재컴파일과 재테스트가 필요하기 때문에, 구축이 곤란해 보수에 고액의 비용이 들었습니다.

한편, CORBA 와 같은 분산 객체를 사용해 구축한 어플리케이션은, 다층 아키텍쳐(architecture)에 순응하고 있기 (위해)때문에, 복수의 중요 사항을 제대로 분할하는 것을 가능하게 합니다. 3 층의 어플리케이션에는, 사용자 인터페이스 코드층, 계산 코드 (비지니스 논리) 층, 데이타베이스 액세스층이 포함됩니다. 각층층간의 교환은, 모두, 각 CORBA 객체의 발행하는 인터페이스를 통해서 행해집니다. 아래의 그림은, 모노리식인 어플리케이션으로부터 다층 또한 모듈화한 어플리케이션에의 추이를 나타냅니다.

사용자 인터페이스층

사용자 인터페이스층은, 사용자와 인터랙티브인 교환을 실시하는 계층입니다. 이 계층은, 조직 전체를 통한 효율적인 사용자 인터페이스의 설계와 편리한 사용에 초점을 맞히고 있습니다. 사용자 인터페이스층은, 사용자의 데스크탑, 조직의 인트라넷, World Wide Web (인터넷) 상에 존재합니다. 동일한 서버에 액세스 하는데, 다른 몇개의 사용자 인터페이스의 구현이 사용되는 일도 있습니다. 사용자 인터페이스층은, 일반적으로 비지니스 논리층의 메소드를 호출하는 것으로, 비지니스 논리 서버의 클라이언트로서 기능합니다.

서비스 (서버) 층

서비스층 또는 비지니스 논리층은, 클라이언트 코드와 교환을 하는 서버 베이스의 코드입니다. 비지니스 논리층은, 비지니스 객체로 구성되어 있습니다. 비지니스 객체란, 재고 관리, 예산, 판매 주문, 청구서 작성등의 논리 비지니스 기능을 담당하는 CORBA 객체를 가리킵니다. 이러한 객체는, 데이터 포함층의 객체의 메소드를 호출합니다.

데이터 포함 (데이타베이스) 층

데이터 포함층은, 데이타베이스 루틴을 캡슐화한 객체로 구성되어 DBMS 제품과 직접 대화를 실시합니다. 예를 들어,get_Sales_Sum 라고 하는 메소드를 작성해, 적절한 SQL 의 SELECT 문을 사용해 RDB로부터 데이터를 취득하도록(듯이) 이 메소드를 구현할 수가 있습니다.
